3. 出力量の内訳（推定）

fa_clip_test_04 1ラウンド（48試合）≈ 8009 行 → 試合あたり約 167 行

主因（Match.simulate 全試合共通）:
- [ACTIVE] / [STARTERS] / [SUB] / 【試合動向】
- [MINUTES] / [PBP] Debug Summary / [COMMENTARY] Preview / [PLAY] Preview

副因（season.py）:
- 自チーム試合のみ postgame / preview（2試合分）
- 経営・杯・ラウンド集計

結論: quiet 化は season.py だけでは不十分。Match.simulate() への抑制伝播が必須。

A. simulate_next_round 出力 quiet化 / 表示抑制
--------------------------------------------------------------------------------

【利点】
- 直近 smoke で判明した「post summary 埋没」を直接解決
- save 構造変更なしで改善可能
- choice "1" の preview / weekly / post summary 投資を活かせる
- choice "2" / "9" / Tk 接続前の安全な土台になる

【リスク】
- Match.simulate() に print が散在（10箇所以上）。引数設計を誤るとデバッグ用途が壊れる
- simulate_next_round / simulate_multiple_rounds / simulate_to_end / Tk 全経路への伝播が必要
- 「何を残すか」の UX 判断（自チーム postgame vs post summary のみ等）

【変更範囲】
- season.py: simulate_next_round(quiet=...) 等
- match.py: simulate(quiet=...) + _print_* 系のガード
- main.py: choice "1" で quiet=True 指定
- 新規テスト（出力行数・完走確認）
- 想定: 中〜大（ファイル数は少ないが Match 内部に及ぶ）

【save 影響】
- なし（表示のみ）

【Godot 移行前価値】
- 高。CLI/Tk 双方の simulate 体験改善。Godot は readonly export 経由で詳細を見る設計のため、
  Python CLI の debug dump 抑制は Godot 本体には必須ではないが、開発・検証効率は上がる

【今すぐ度】 ★★★★★（最高）

【後工程へ回す理由】
- なし（むしろ先にやるべき）

【最小実装の方向性（調査結果）】
- quiet=True のような引数は現状存在しない → 新設が必要
- season.simulate_next_round(quiet=True) → Match(...).simulate(quiet=True) の伝播が最小筋
- choice "1" quiet 時の残し候補:
  1. preview / weekly（main.py、既存）
  2. ラウンド見出し + ラウンド集計2行（任意）
  3. post summary（main.py、既存）
- 抑制候補:
  - 非自チーム試合の Match 全 debug 出力
  - 自チーム試合の PBP/COMMENTARY/PLAY/MINUTES debug（post summary で代替）
  - 自チーム postgame 詳細は optional（post summary と重複するため quiet では省略可）
- _finalize_season 統計は別フラグ（finalize_quiet 等）の検討余地あり
