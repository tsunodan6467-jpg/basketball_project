5. CardTasks の候補評価
--------------------------------------------------------------------------------
- **役割**: 「今やること」ToDo。**sandbox では `Phase4WarningCard`** で右列主役。
- **本線**: 暗地 `StyleBoxFlat_card` + 明色ラベル。**`Phase4WarningCard` に寄せると sandbox
  と意味付けが一致**しやすい。
- **リスク**: **`CardWarnings` も警告系**。両方を `Phase4WarningCard` にすると、**注意喚起
  カードが縦に 2 枚**（間に CardNext / CardSummary が挟まるが）**視覚的に「全部危険」感**
  が強まる可能性。**`Phase4SummaryCard` にすると** sandbox とはズレるが、**ニュース白
  カードとのコントラスト**では「ToDo＝白カード」の方が主張しすぎる可能性。
- **結論**: **Theme だけで実装は可能**だが、**WarningCard か SummaryCard かは設計判断が必要**。
  **CardWarnings との役割分担**を先に決めないと、**被り**が出やすい。


--------------------------------------------------------------------------------
6. CardNext の候補評価
--------------------------------------------------------------------------------
- **役割**: **次戦 / 次イベント**。ホームで**重要度が高い**。
- **データ**: `next_game` は **1 文字列**。**複数行になりにくい**（`CardNews` と同型にしやすい）。
- **`.gd`**: **Card 自体の visible は触っていない**（常に表示）。
- **Theme**: **`Phase4SummaryCard` が自然**（情報カード）。**CardNews と同じ手順**
  （panel override 除去 + 子 Label 濃色化）で済む見込みが高い。
- **リスク**: **低**。Header ClubBand と「試合の近さ」は文脈で近いが、**本線は帯に次戦全文は
  出していない**ため、**重複は限定的**。
- **結論**: **次の 1 枚として最もバランスが良い候補**。


--------------------------------------------------------------------------------
7. CardSummary の候補評価
--------------------------------------------------------------------------------
- **役割**: `club_summary`。**export 由来の説明複数行**になりうる（`_join_lines` 全行）。
- **Theme**: **`Phase4SummaryCard` は型として合う**。
- **リスク**: **白背景 + 長文**で**縦に伸び**、**Scroll 内で密度が目立つ**。**可読性は
  autowrap と行数次第**で悪化しうる。
- **結論**: **`.tscn` のみでは技術的に可能**だが、**UX リスクが CardNext より高い**。
  **「次」より後ろ**が無難。


--------------------------------------------------------------------------------
8. CardWarnings の候補評価
--------------------------------------------------------------------------------
- **現状**: **`StyleBoxFlat_warn`** 専用。**シーン内で `StyleBoxFlat_warn` を参照しているのは
  CardWarnings のみ**（**他カードは参照しない**）。**SubResource を削除しない運用**は
  CardNews と同型。
- **Theme**: **`Phase4WarningCard`** が**意味的に最一致**（`phase4_readonly_core.tres` に
  定義あり）。**パネルは Theme に寄せ、ラベルは `Phase4NoteText` 系に寄せるか、現状の
  暖色 override を維持するか**は目視判断。
- **visible**: **空 warnings で非表示**の挙動は **`.gd` 既存のまま**。
- **リスク**: **見た目が強くなりすぎないか**は Theme の `SubResource("4")` と現行
  `StyleBoxFlat_warn` の**比較目視**が必要。**中程度**。
- **結論**: **`Phase4WarningCard` 候補の第 1 位（意味）**。**ただし「次の 1 枚」としての
  安全性**は **CardNext よりやや議論が増える**（色の強さ）。

9. CardTeamExtras の候補評価
--------------------------------------------------------------------------------
- **役割**: owner_trust / salary_cap / recent_form の**任意行**。
- **`.gd`**: **`_extras_card.visible`** が子行の可視と連動。**Theme を付けてもロジック変更不要**。
- **子ノード**: **複数行・複数ラベル**（Key + Value）。**CardNews より調整点が多い**。
- **リスク**: **中**。**Metrics（money）** や **club_summary** と**情報が隣接**し、**白カード化で
  「数字が並ぶ」**印象が強まる可能性。
- **結論**: **可能だが「次の 1 枚」としては優先度は CardNext より下**。

10. CardRank / CardMoney の候補評価
--------------------------------------------------------------------------------
- **現状**: **CardDivision のみ** Theme 済みで、**同一 HBox 内に暗地 2 枚が残る**。
- **Theme**: いずれも **`Phase4SummaryCard` が自然**（指標サマリー）。**CardDivision と同じ
  ラベル色パターン**（HDiv 系 / 本文系）へ寄せられる。
- **リスク**: **CardRank だけ**、または **CardMoney だけ**を白くすると **MetricsRow 内で
  チグハグ**。**「1 コミットで Rank+Money の 2 枚」**なら整合は取りやすいが、**「1 カード
  限定」というユーザ方針からは 2 ノード扱い**になる。
- **Header ClubBand**: **将来**帯に資金・成績を寄せる設計がある場合、**Metrics と帯の
  重複**が論点（README/docs の sandbox 調査と整合）。
- **結論**: **単独 1 枚より「MetricsRow 残り 2 枚セット」で計画する方が安全**。
  **いま単独で 1 枚だけ触るのは非推奨寄り**。

================================================================================
【結論】Cursor としての推奨（曖昧にしない）
================================================================================

■ 次に Theme 限定適用するならどのカードが最も安全か
  → **`CardNext`**（**`Phase4SummaryCard` + 子 Label 濃色化**。**データは 1 文字列**で
    **CardNews と同型**）。

■ Phase4SummaryCard にするならどれがよいか
  → **第 1: `CardNext`**。**第 2: `CardRank` と `CardMoney` をセット**（単独片方は非推奨）。
    **CardSummary は長文リスクのため後回し**。

■ Phase4WarningCard にするならどれがよいか
  → **`CardWarnings` が最自然**。**`CardTasks` は sandbox 整合で WarningCard も妥当**だが
    **`CardWarnings` と役割が近い**ため、**見た目設計をセットで決める**こと。

■ CardTasks と CardWarnings のどちらを先に触るべきか
  → **見た目の「安全・中立」優先なら先に `CardWarnings` は必須ではない**。
    **意味の一貫性優先なら `CardWarnings` を WarningCard 化**。
    **いずれにせよ、ToDo の `CardTasks` より先に決めるべきは「警告 vs ToDo の色の役割」**。
    **本レポートの実装順のデフォルト**: **先に `CardNext`** → その後 **`CardWarnings`**
    → その後 **`CardTasks`**（**Tasks を WarningCard にする場合**）。

■ CardRank / CardMoney は単独で触るべきか、MetricsRow 全体方針後にすべきか
  → **単独 1 枚は非推奨**。**方針として「Division と揃えて残り 2 枚を同じコミット」**がよい。

■ まだ触らない方がよいカードはどれか
  → **`CardRank` または `CardMoney` の片方だけ**。**`CardNavMenu`**。**根拠のない順序だけの
    大入替**。

■ いったん CardNews 1 行化 / DTO 整理へ進むべきか
  → **Theme 連鎖だけ続ける必要はない**。**案 G は合理的**。ただし **低リスクで効果が大きい
    `CardNext` の 1 コミット**を挟むと、**ホームの縦リズムが一気に揃い**、**その後 DTO に
   集中しやすい**。
