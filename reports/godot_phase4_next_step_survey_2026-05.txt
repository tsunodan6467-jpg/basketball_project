3. 候補A：Python自動起動設計
---------------------------
■ 利点
  - 開発・検証時の **手動 CLI 実行が不要**になり、表示の更新サイクルが速くなる。
  - 7 種類の JSON を揃えたい場面で、運用負荷が下がる。

■ リスク（docs/GODOT_GUI_INFORMATION_ARCHITECTURE_2026-05.md §14.3 と整合）
  - Windows で `python` / `py` / venv の **実行ファイル解決**が環境依存。
  - cwd 不一致による **import 失敗**。
  - `res://` と OS 実パス、`user://` への書き込み可否（エクスポート配布時は別問題）。
  - 日本語パス・スペースを含むパスの **子プロセス引数エスケープ**。
  - 起動失敗時も **古い from_python.json が残ると誤表示**（タイムスタンプ・失敗時削除方針が要設計）。
  - **Steam 配布**では「Python 同梱」「exe 化」「そもそも Godot 単体で完結」の **方針未決**のまま実装すると後で捨てやすい。

■ 触りそうなファイル（実装時の想定。今回は未変更）
  - Godot: 新規 **Autoload** または `home_dashboard.gd` の `_ready` / 専用「再生成」ボタン
    （いずれもシーン・スクリプト変更が伴うため Phase 4 では慎重に）
  - `project.godot`（Autoload 登録時）
  - 将来: 設定ファイル（python.exe パス、save パス、出力先 user:// 等）

■ 起動粒度の比較（設計論点のみ）
  - **各画面入室時に個別起動**: 遅延・失敗の都度ユーザーに影響。画面ごとに CLI 引数が明確。
  - **ホームで一括生成**: 1 回の子プロセスで複数 export を呼ぶラッパーが必要になるが、
    待ち時間集中・失敗時の部分成功の扱いが課題。

■ フォールバック
  - 現状どおり **mock へ落ちる**動きは既にある。自動起動失敗時も **既存 JSON を読むか mock に戻すか**
    を仕様で固定する必要あり。

■ 今すぐ実装すべきか
  **いいえ（設計メモ・配布方針の整理に留めるのが妥当）**。
  理由: 同一 docs §14.5 が「Phase 4 初期では自動起動は実装しない」「配布方針を固めてから判断」と明記。
  先に **save パス・出力先・失敗時・古い JSON** の方針を文書化し、Steam/exe 案とセットで判断するのが安全。

6. Cursor としてのおすすめ順（次に手を付ける優先度の提案）
--------------------------------------------------------
  (1) **Python 自動起動: 実装はせず、設計メモのみ**（実行ファイル解決・出力先 user:// 化・
      失敗時挙動・古い JSON・Steam/exe 方針を 1 枚にまとめる）。
      — 理由: リポジトリ正本ドキュメントが Phase 4 初期で自動起動を否定しており、
         先に実装すると配布戦略と衝突しやすい。

  (2) **第8 readonly 画面のうち、データが明確でホームと被り調整しやすいもの**
      — 第一候補: **オーナーミッション / 信頼 / 期待値**（Team フィールドが既にあり、
         ホームは短い task 行に留められる設計にできる）。
      — 第二候補: **戦術サマリー（読取のみ・編集ボタン無し）**（価値は高いが
         normalize 系の副作用に注意した export 設計が必要）。

  (3) **本格ナビ: 大改修は後回し**。必要なら **HeaderNavRow の文言短縮など 1 コミット規模**
      のみ（ナビ文書の短期第一候補と一致）。

  (4) **契約 / FA 一覧の Godot 画面**は、閲覧専用の文言・DTO 境界が固まるまで **後ろ**。
