6. 候補C：ホームにボタン → **同期**で Python 一括 export を実行
--------------------------------------------------------------------------------
【想定 API（Godot 4.6.2 系の一般的パターン・調査メモ）】
  - **`OS.execute`**（`blocking=true`）: 完了まで **メインスレッドがブロック**され、**UI フリーズ**しやすい。長い export で顕著。
  - **`OS.execute`**（`blocking=false`）: 即戻るが **終了コード・stdout の取得**が別途必要になりがち。
  - **`OS.create_process`**: 子プロセス起動。**完了待ち・出力取得は別設計**。
  - いずれも **引数エスケープ**（日本語パス・スペース・`.sav` パス）を自前で堅くする必要あり。

【利点】
  - 実装イメージが単純（「1 コマンド叩く」）。

【欠点・リスク】
  - **UI 停止**、**失敗時のユーザー説明**（stderr をどう見せるか）、**cwd がリポジトリルートでないと `python -m basketball_sim` が失敗**しうる。
  - **`py` vs `python` vs venv の `python.exe`** がマシンごとに違う。
  - **Steam 本番**で Python 同梱を前提にしない限り、**配布版では機能死**またはセキュリティ・サポート負荷。

【今やるべきか】
  - **いきなり本採用は非推奨**。試作するなら **開発ビルド限定**かつ **短いタイムアウト設計**が前提。同期ブロックは UX 上よくない。

【欠点・リスク】

  - **生成中にユーザーが画面遷移**すると、どの画面がいつ再読込されるかの仕様が必要（§11）。
  - **stdout/stderr の受け渡し**、**プロセスゾンビ**、**二重起動防止**（連打）のガード。

【今やるべきか】
  - **C がプロトタイプで問題なければ D へ段階移行**が現実的。最初から D はコスト高。

10. save path / output-dir / Python 実行ファイルの論点
--------------------------------------------------------------------------------
【save path】
  - `godot_readonly_bundle` は **`Path` での存在チェックを CLI 前段で強制していない**（runner 内で load 失敗時に例外）。Godot から起動する場合も **ファイル存在・拡張子**の事前確認が必要。
  - 開発例 `%USERPROFILE%\.basketball_sim\saves\debug_....sav` は **個人マシン依存**。**固定ハードコードは危険**（他ユーザー・CI・パス文字セット）。
  - 将来: **スロット選択**、**ファイルダイアログ**（OS Native）、または **本番セーブが user:// に来た後の正本パス**との統一が必要。現 Phase 4 では **「どのセーブを Godot が指すか」が未確定**。

【output-dir】
  - 現運用: リポジトリの **`godot/data`**（OS パス）。Godot は **`res://data/...`** で読む。**エディタ実行時**はプロジェクトフォルダ上のファイルと一致させればよい。
  - **配布ビルド（エクスポート）**では **`res://` はパック内が多く読み取り専用**になりがち → **実行時書き込み先として `res://data` は不適**になりうる。**`user://`** に JSON を出し、読込候補を `user://` を先に見るよう変更する、という **別タスク**になり得る（9 画面すべての候補パス配列の見直しが波及）。
  - Godot から `OS.get_executable_path()` や `ProjectSettings.globalize_path` で **絶対パスを組み立て**、`--output-dir` に渡すのが一般的。**相対パスは cwd 依存**で壊れやすい。

【Python 実行ファイル】
  - Windows: **`py` ランチャー**、**`python`**、**venv\Scripts\python.exe`** のどれを呼ぶかを **設定ファイル or プロジェクト設定**で可変にするのが無難。
  - **引数**: `-m basketball_sim.export.godot_readonly_bundle` は **リポジトリルートを cwd** にして実行するのが import 上安全（IA §14.3 の調査メモと同趣旨）。


11. 失敗時フォールバック案
--------------------------------------------------------------------------------
| ケース | 挙動の整理案 |
|--------|----------------|
| Python 起動失敗（PATH・venv） | ラベルで原因コードと「README のコマンドを手動実行」を表示。mock へ自動フォールバックは **データ鮮度が不透明**なので **明示メッセージ**がよい。 |
| save なし / 壊れた save | bundle 内で例外 → **0 件成功で終了**または continue モードで部分失敗。Godot は **古い from_python を残すか削除するか**を方針決め（残すと誤表示）。 |
| bundle 途中失敗（既定 continue_on_error=False） | **途中まで新ファイル・途中まで旧ファイル**の混在。**アトミック更新**（tmp に書いて rename）は Python 側拡張が必要 → **別タスク**。 |
| JSON 生成成功だが parse 失敗 | 現状各画面は **次候補へフォールバック**またはエラー表示。export 成功直後に **再読込**するときは **同じロジック**を踏むべき。 |
| 古い JSON が残る | 「最終更新」表示（mtime または generated_at）で **鮮度を可視化**（§12）。 |
| 一部だけ更新（continue_on_error 利用時） | **「9 件中 N 件失敗」**を UI に出さないとサイレント不整合。 |
| 生成中に別画面を開く | 各画面 `_ready` で都度読むなら **読み取り半端**になりうる。**ロック中は遷移禁止**や **完了後に signal で一斉リロード**の設計が必要。 |


12. generated_at / mtime 表示の扱い
--------------------------------------------------------------------------------
【generated_at を全 DTO に】
  - **一貫性は高い**が **Python export 全9系＋テスト**に波及。Phase 4 の「ボタンだけ先」より **後段または並行設計**が現実的（IA でも未着手明記）。

【Godot 側でファイル mtime のみ】
  - **`res://` の FileAccess では取得しづらい**場合があり、**OS 絶対パス**が必要になりがち。`user://` 移行後は `FileAccess.get_modified_time`（Godot 4 で利用可能か要 API 確認）で足りるケースあり。
  - **ホームだけ**表示するなら **他8画面への波及は小さい**が、「どのファイルの mtime か」（9 個の最大？ホームのみ？）を仕様化する必要あり。

【今すぐ全画面】
  - **波及大**。まずは **export ログ・コンソール**で十分な開発段階もあり。
