3. 候補A：ロスター表本体の Theme 対応
--------------------------------------------------------------------------------
現状のコード根拠:
  - roster_view.gd: _clear_rows → _add_table_header_row / _add_player_row。
  - 表ヘッダー: HBoxContainer + Label.new() ×9。font_color (0.82,0.88,0.96)、font_size 12、
    _col_width で列幅固定。直後に HSeparator。
  - 選手行: 同様に Label.new() ×9。font_color (0.92,0.94,0.98)。年俸列は右寄せ、
    名前・状態は tooltip。
  - roster_view.tscn: 全体 Background は暗色 ColorRect。Scroll は RowList の子に動的行が追加。
    ルート theme = phase4_readonly_core（ヘッダーは Phase4HeaderCard 済）。

利点:
  - ロスターは中核画面。表が読みやすいとデモ・プレイ感の向上が大きい。
  - 契約人事と同様「限定適用」の延長で説明しやすい。

欠点:
  - 表は動的生成のため .tscn だけでは完結しない。色はコード override 固定。
  - 白カード（Phase4SummaryCard）内に表を載せると、現行の明色文字はコントラスト不足になりやすい。
  - 行数が多いと Scroll 内の再レイアウト・視認性の手動確認が重い。

触りそうなファイル:
  godot/scripts/roster_view.gd（必須）
  godot/scenes/roster_view.tscn（Scroll 周りの Panel 追加、背景、variation 付与など任意）
  godot/themes/phase4_readonly_core.tres（新 variation 追加する場合）

.tscn だけで済むか:
  いいえ。現状の表はすべて GDScript で Label を生成しているため、配色・variation の切替は .gd が主戦場。

.gd 変更範囲:
  小: 動的 Label に theme_type_variation を付与し、override 色をやめる／Theme に寄せる。
  中: 行コンテナに Panel や Separator 方針を変え、ゼブラ行・hover 等。
  大: ItemList / RichTextLabel 化や列ソートなどは別タスク。

リスク:
  - 白パネル化した瞬間に「ヘッダーは白・表エリアは暗」の二層が崩れ、中間グレー帯になりがち。
  - 列幅・clip・tooltip 挙動の回帰。
  - .tscn を広く触ると UID 差分が出やすい（運用で吸収は可能だが負荷あり）。

OnDark に寄せるだけで足りるか:
  - 画面背景が暗いままなら、Theme の Phase4OnDarkTableHead / Phase4OnDarkTableCell に
    variation を寄せるだけで、現行の明色 override と整合しやすい（preview で検証済み系統）。
  - 「白ベースで表エリアもカード化」するなら、文字色を Phase4TableCell 系へ寄せる .gd 変更が必要。

Scroll / RowList の Theme 化の必要性:
  - ScrollContainer 自体はスクロールバー見た目程度で、表の主因は子 Label。
  - RowList の separation は既に tscn で override。必須ではないが、
    「表をカードで囲む」なら Scroll の親に PanelContainer を足す .tscn 変更が現実的。

今やるべきか:
  - Theme 一貫性を「データ画面」から進めるなら最有力。ただし .gd 必須でコミット単位は分けたい。

観点 1〜12: 高 / 中 / 低 / 中 / 否 / 要 / 無 / 中〜高 / 中 / 中（.gd中心なら可） / 条件付きでYes / ホーム先行やナビ大改修と同時だと負荷大


4. 候補B：契約・人事サマリーの動的 Label 対応
--------------------------------------------------------------------------------
現状のコード根拠:
  - _fill_risk_rows: 空時は Label.new + _style_body_label(12, 暗め灰)。
    データ時は複数行テキストの Label.new + _style_body_label(12, 明色 0.86,0.9,0.95)。
  - _fill_player_rows: 同様。最大 8 行。複数行 1 Label。
  - _style_body_label: autowrap、SIZE_EXPAND_FILL、font_size / font_color override のみ。
  - contract_personnel_summary_view.tscn: RiskCard / PlayersCard は StyleBoxFlat_summary（暗）。
    タイトル RiskTitle / PlayersTitle は既に明色 font override（暗パネル向け）。

利点:
  - 同一画面内で「白カード部分」と「暗パネル＋動的行」の二重構造が解消し、画面単位の説明がしやすい。
  - 変更ファイル数は原則 contract_personnel_summary_view.gd + .tscn + 必要なら theme のみで収まりやすい。

欠点:
  - 商品インパクトはロスター表やホームより限定的。
  - 画面内仕上げに寄り、プロダクト全体の「顔」には直結しにくい。

触りそうなファイル:
  godot/scripts/contract_personnel_summary_view.gd
  godot/scenes/contract_personnel_summary_view.tscn（RiskCard/PlayersCard を Phase4SummaryCard に、
    タイトル Label の override を Theme 側に寄せる等）
  godot/themes/phase4_readonly_core.tres（必要なら）

.tscn だけで済むか:
  カード枠を白系にする部分は .tscn で可能。ただし動的行の文字色は .gd の override が上書きするため、
  白化するなら .gd 側の Color か variation 付与のどちらかが必要。

.gd 変更範囲:
  - 小〜中: _style_body_label を「パネル種別に応じた色／variation」に分岐、
    または override をやめて theme_type_variation のみにする。
  - Risk / Player の行構造を HBox 化するのは別途（スコープ増）。

リスク:
  - RiskTitle 等の静的 Label と動的行の色の再バランス。
  - 長文 autowrap 時のカード高さ・Scroll 内の見切れ。

今やるべきか:
  - 「契約人事画面を一枚で統一感」優先なら良い。全体ロードマップでは A または C と並走判断。

観点 1〜12: 中 / 中 / 低 / 高い（範囲狭い） / 部分的 / 要（実質） / 無 / 中 / 中 / 高 / 条件付きYes / ロスター・ホーム優先なら後回し可


5. 候補C：ホーム本番 GUI 化試験
--------------------------------------------------------------------------------
現状のシーン構造（home_dashboard.tscn 要約）:
  - ルートに Theme リソースは未割当（SubResource StyleBoxFlat_* でカード・ヘッダーを自己完結）。
  - HeaderCard: StyleBoxFlat_header。HeaderTopRow に chip + placeholder。
  - HeaderNavRow: 5 ボタン（ロスター・クラブ史・順位表・日程・施設）のみ。
  - Scroll/Inner: CardNavMenu（NavColumns: チーム / リーグ / クラブ / 経営3ボタン）、
    主要指標 MetricsRow（所属D・順位・資金）、CardTeamExtras、CardWarnings、
    CardNext、CardSummary、CardTasks、CardNews。
  - FooterNote: 手動 JSON と「自動起動は未実装」の文言。
  - home_dashboard.gd: JSON 読込と各 Label への文字列反映、visibility 制御、
    change_scene_to_file のみ。動的ノード生成なし。

1画面試験として安全な切り方:
  - まず「見た目のみ」: HeaderCard または 1 枚のカード（例: CardDivision のみ）に
    phase4_readonly_core の variation を当て、他は現状の StyleBox のまま残す。
  - HeaderNavRow のボタン数・シグナル接続は触らない（ドキュメント上の制約とも整合）。
  - NavColumns の列追加・経営ボタン再配置は「本番 GUI 化」とセットになりやすく、
    小さく切るなら「色・フォント・パネル角」のみに留めるのが安全。

利点:
  - 商品価値・Steam 向け第一印象へのインパクト最大。
  - 90年代 GM ホームの方向性をここで試せる。

欠点:
  - 触るノード数が多く、部分適用でも差分が読みにくい。
  - カードメニューと上部ナビの二重導線の見た目バランスを崩しやすい。

触りそうなファイル:
  godot/scenes/home_dashboard.tscn（主）
  godot/themes/phase4_readonly_core.tres（Button/Label の追加 variation）
  godot/scripts/home_dashboard.gd（原則不要だが、将来の動的要素で要る可能性）

.tscn だけで済むか:
  スタイルの置換なら主に .tscn + .tres で可能。.gd は現状ほぼ不要。

.gd 変更:
  試験段階では不要が、データ更新 UI 等を足すと別論（候補E と合流）。

リスク:
  - 10 画面入口・経営3ボタン・主要指標が同一 Scroll 内にあるため、
    縦スクロール量・最小高さの再調整で他カードが見えなくなる回帰。
  - 大きな .tscn 編集は UID や子順序の差分が出やすい。

観点 1〜12: 最高 / 高 / 中（見た目のみならロジック破壊は低いが見落とし面は広い） / 低〜中 /
  主にYes / 原則不要 / 無 / 高 / 中〜高 / 低〜中 / ブランド優先ならYes / Theme縦割り未完了との並走は混乱しやすい
