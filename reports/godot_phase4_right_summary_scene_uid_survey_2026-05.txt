2. 発生した UID / load_steps 問題の概要
----------------------------------------
ユーザー報告によると、新規 scene `home_production_wire_preview_right_summary.tscn` を
Godot 4.6.2 で開き F6（現在のシーン実行）したあと、当該ファイルの先頭行だけが自動で
書き換わった。

- 変更前（リポジトリ上の意図した内容）:
  [gd_scene load_steps=6 format=3 uid="uid://c9homeprodwirers01"]

- 変更後（エディタ再シリアライズ後）:
  [gd_scene format=3 uid="uid://36856deeyhng"]

つまり (a) 先頭の `load_steps=6` が消え、(b) `uid` が手書き値から別の値へ差し替えられた。
同時に Godot 出力に次が出たとのこと:
  ERROR: Unrecognized UID: "uid://36856deeyhng"

本調査時点でワークツリーは `git checkout HEAD -- ...` により復旧済みであり、
当該 .tscn は再びコミット時点の先頭行に戻っている。

7. uid://c9homeprodwirers01 の扱い
---------------------------------
- リポジトリ内 grep では、`c9homeprodwirers01` は当該 .tscn の 1 行目のみに出現。
  project.godot や他 scene からの参照は見つからない（本線未接続の sandbox のため）。
- Godot 4.x は各リソースに UID を付け、エディタは通常 `.godot` 配下にパス↔UID の対応を保持する。
  テキストエディタや生成ツールで「未登録の uid 文字列」を .tscn に書いても、
  初回オープン時に「既存キャッシュと矛盾」「未登録」と判断され、新規 UID への差し替えが走ることがある。
- `c9homeprodwirers01` は人間が読みやすい固定文字列として意図的に付与されたが、
  その時点でエディタの UID キャッシュに「この res:// パス = この uid」のエントリが無かった、
  または別ルールで上書きされた、と推測できる。


8. uid://36856deeyhng が出た状況の整理
--------------------------------------
- これは典型的に Godot が内部生成する UID 形式に近い（英数字のランダム列）。
- ユーザー観測では「ファイル先頭がこの uid に置換された」直後に
  Unrecognized UID が出ている。
- 想定される分岐:
  (1) ファイル上の uid は更新されたが、同一セッション内の UID キャッシュ更新・ディスク書き込みの
      順序がずれ、一瞬だけローダが新 uid を解決できなかった（タイミング問題）。
  (2) 4.6 系で報告されている UID 関連の不具合・再インポート順序の問題に該当し、
      新規生成 uid が一時的に孤立参照になった（GitHub godotengine の Unrecognized UID 系 issue と類似）。
  (3) 別の場所（メモリ上のシーンキャッシュ等）が古い uid を参照したままになり、
      ログ上は新 uid 名が出ているが実際の原因はキャッシュ不整合。

いずれにせよ、「エディタが right_summary 用に新 UID を書き込んだ」こと自体は、
手書き uid がエディタ管理の UID と一致しなかった結果として説明しやすい。


9. load_steps=6 が消えた理由候補
-------------------------------
- 候補A（エディタの正規化）: Godot 4.x のシリアライザは、内部で算出可能な load_steps を
  省略する保存形式を取る場合がある。本リポジトリの `club_history_view.tscn` が
  load_steps 無しの先頭行であることは、少なくとも「省略保存」が通常運用の一部である証左。
- 候補B（4.6 の保存差分）: 4.5→4.6 以降、scene のエクスポート・保存の決定性に関する issue が
  報告されている（例: 再保存で表記が変わる）。load_steps の省略はその一環の可能性。
- 候補C（UID 差し替えとの独立性）: load_steps の省略は「UID エラー」とは独立に起きうる。
  同一の「再保存」操作の副作用として両方変わった、という理解でよい。

結論寄り: 「load_steps が消えた = ファイルが壊れた」とは限らず、
エディタが冗長フィールドを落とした可能性が高い。ただしチーム方針として
「先頭行の load_steps を常に明示したい」場合は、再保存後に git diff で監視する必要がある。


10. Godot 4.6.2 で想定される再シリアライズ挙動
---------------------------------------------
- プロジェクトを開くと、未インポート・未登録のリソースに対して UID の割当・更新が走ることがある。
- scene を初めてエディタで保存すると、ノード順・プロパティ表記・先頭行（uid / load_steps）が
  エディタの好む形に正規化されることが多い。
- 「Unrecognized UID」は、project 設定や他リソースが UID 参照しているのに
  キャッシュに無い、といった状況でも報告されている（公式 issue 参照の一般論）。

本件では right_summary が本線から参照されていないため影響範囲は限定的だが、
sandbox でも「F6 後に意図せず dirty になる」ことは運用上のノイズになる。


11. 案A: Godot が生成した UID を正式採用する
-------------------------------------------
内容:
- エディタで scene を開き、意図した見た目のまま「保存」を一度行い、
  先頭行に付いた uid（例: uid://36856deeyhng または別の新規値）をそのままコミットする。
- 同一マシンで再度開いて F6 し、先頭行が安定するか・エラーが消えるか確認。

利点:
- エディタの UID キャッシュとファイルが一致しやすい。
- 手書き uid とエディタの期待のズレを解消できる。

リスク:
- 初回保存直後に再度 Unrecognized が出る場合は、4.6 側の既知問題の可能性があり、
  プロジェクト再読み込みや .godot 削除（慎重に）まで議論が及ぶことがある。
- チームが「接頭辞 uid で grep しやすい」方針なら、可読 uid は失われる。


12. 案B: scene の uid を削除する
--------------------------------
内容:
- 先頭行から `uid="..."` を削る（手編集）。次回エディタ保存で付与されるのを待つ。

利点:
- 一度「空白」にしてエディタに任せることで、人間の固定 uid と衝突しない。

リスク:
- Godot のバージョン・設定によっては、保存前に警告や再付与挙動が異なる可能性。
- 結局「エディタが付けた uid をコミットする」（案Aと合流）ことが多い。


13. 案C: 手書き UID をやめ、エディタ生成の UID を使う
---------------------------------------------------
内容:
- 新規 scene は「Duplicate」や「New Scene」から作り、uid は一切手で書かない。

利点:
- 最もエディタと整合しやすい。本件の再発を防ぎやすい。

リスク:
- 既にコミット済みの right_summary を作り直す手間は残る（案Dと組み合わせ）。


14. 案D: 新規 scene を作り直す
------------------------------
内容:
- 既存 wire preview をエディタで複製し、右サマリー列だけ手で調整して保存。
  または空 scene から組み立て直す。

利点:
- 全ノードパス・内部 id がエディタ生成になり、再シリアライズ差分が最小化されやすい。

リスク:
- レイアウトの再現コスト。git diff が大きくなりうる。


15. 案E: 右サマリー比較 scene を一旦保留する
------------------------------------------
内容:
- 本件の UID ノイズを避け、2 カラム成功形のみ運用する。

利点:
- 本線・CI・ドキュメントに触れずに済む。判断コストが最小。

リスク:
- 右列レイアウトの比較が後ろ倒しになる。


16. 案F: 現行 2 カラム版を本命にして進む
---------------------------------------
内容:
- `home_production_wire_preview.tscn` を正とし、右サマリーは設計判断のみ文書で扱う。

利点:

17. 安全性比較（案A〜F）
-----------------------
- 本線 `home_dashboard` への影響: いずれも現状「未接続」ならゼロに近い。
  将来接続する段階では、PackedScene 参照が path か uid かで振る舞いが変わるため、
  案A/C/D で「エディタ公式の uid」を採用しておく方が安全。
- git の見通し: 手書き uid のまま再保存されると毎回先頭行が dirty になりうる → 運用リスク大。
- チーム認知負荷: 案E/F が最軽量。右列比較が必要なら案A か C+D。


18. 本線 home_dashboard への影響
-------------------------------
- 調査時点で `home_production_wire_preview_right_summary.tscn` を参照する記述は
  project.godot / 他 tscn に見当たらない。
- よって UID の揺れは本線の実行パスに直接は波及しない。
- 将来 `preload` / `PackedScene` で uid 参照に切り替える場合は、
  その時点の uid がキャッシュと一致していることが重要。


19. 今すぐ実装してよい最小修正案（※今回は実装しない前提の「方針メモ」）
--------------------------------------------------------------------
- 最小は「エディタで一度開いて保存し、出力された先頭行（uid および load_steps の有無）を
  そのままコミットする」こと（案A）。scene 本体のノードは触らなくてよい場合が多い。
- 併せて、同一セッションで F6 → 即 git diff を確認し、Unrecognized が再発するか記録する。
- 再発する場合は Godot issue の有無・.godot 削除可否を別タスクで扱うのが安全。


20. まだ実装しない方がよいこと
-----------------------------
- 手書き uid を別の手書き uid に「推測で」書き換えること（再び未登録になる恐れ）。
- project.godot の main_scene や uid 参照をいじって症状を隠すこと（本件のスコープ外かつ高リスク）。
- 問題の切り分け前に `.godot` を無計画に削除すること（他メンバーのキャッシュと議論が必要）。


21. Cursor としてのおすすめ順
----------------------------
1. 案A（エディタ保存で付いた uid を正式採用）を最優先で試す。コスト最小・本筋。
2. それでも Unrecognized が続くなら、Godot 4.6 の UID 関連 issue と照合し、
   プロジェクト再インポートやバージョンピン留めを検討。
3. 並行して案C（今後は手書き uid 禁止）をルール化。
4. 右列比較の優先度が下がるなら案E（保留）または案F（2 カラム本命）。
