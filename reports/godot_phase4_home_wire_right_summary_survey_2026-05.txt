2. 現在の到達点
--------------------------------------------------------------------------------

3. 現在の sandbox レイアウト（実ノード構造の要約）
--------------------------------------------------------------------------------
- `RootMargin` / `MainRow`（**HBoxContainer**）
  - **LeftRail**（左レール・カテゴリナビ風）
  - **RightStack**（**VBoxContainer**・`size_flags_horizontal = 3` で横いっぱい）
    - **ClubBand**（上部帯・資金・信頼・成績要約など）
    - **MidScroll**（縦スクロール・`size_flags_vertical = 3`）
      - **MidInner** / **CardsRow**（**HBoxContainer**）
        - **ColLeft**（`size_flags_horizontal = 3`）… 次戦／順位・成績／クラブ状態
        - **ColRight**（`size_flags_horizontal = 3`）… タスク／ニュース／ショートカット
    - **BottomStrip**（注記・ナビ誘導）
- 1280×720 前提では、**CardsRow は左右均等ストレッチに近い 2 カラム**。ここに**第 3 カラム**を足すか、`MainRow` に**第 3 縦列**を足すかが主な分岐。


4. 右サマリー列を検討する理由
--------------------------------------------------------------------------------
- 参考系のスポーツ GM 画面では、**常時数値列**や**サイドの状態チッカー**があることが多い。
- 本線移植時に「ホームの情報をどこに置くか」を決めるには、**2 カラムのみで足りるか**の比較材料が有用。
- 一方、既存 sandbox は**低〜中密度で成立**しており、**無理に列を増やすと中央が潰れる**リスクが高い（`center_density_survey`・`sandbox_policy` でも右列サマリーは「比較用」「別シーン fork」が無難とされていた）。

12. 右サマリー列に置く候補情報
--------------------------------------------------------------------------------
- **キャップ／サラリー余力**（数値または短ラベル）— ただし **CardClubState** と統合するか要判断。
- **チーム士気**短表記 — ClubBand の信頼と**並べない**（上下関係をラベルで明示）。
- **warnings** 1 行要約 — **CardTasks** と動詞被りに注意。
- **直近フォーム** 1 語 — **CardStandings** と被るなら片方に寄せる。
- **ロスター枠**（13/15 等）— 編成画面へのフック。
- **次戦までの日数**— **CardNext** と被らないように。


13. 右サマリー列で避けるべき情報
--------------------------------------------------------------------------------
- **ClubBand に既にある資金・オーナー信頼の全文再掲**（三重化）。
- **CardStandings と同一の立ち位置数字**。
- **タスクと同じ文言の注意**（「確認せよ」の重複）。
- **長いニュース本文**（ニュースは右列カードまたは専用画面へ）。


14. 現在の 2 カラム構成を維持する価値
--------------------------------------------------------------------------------
- **1280×720 で既に検証済み**の安定形。
- **タスク主役・ニュース軽量・クラブ状態 1 行**という役割分担が読み取りやすい。
- 本線は Header＋Scroll で情報が分散しており、sandbox を**無理に 3 列化**しなくても、本線側で**指標行の再配置**で吸収できる余地がある。


15. 別 scene 比較にする場合の注意
--------------------------------------------------------------------------------
- **UID** を新シーン用に安定割当（既存と衝突しない形）。
- **Theme／SubResource** のコピー量が増える → `sandbox_policy` のとおり差分レビューを徹底。
- README の「開くシーン」説明を**将来**追記するタイミングを決める（本コミット範囲外）。


16. 既存 sandbox を上書きする場合の注意
--------------------------------------------------------------------------------
- 成功形の **git 差分が大きく**なり、**戻しコスト**が増える。
- **F6 比較**がしづらくなるため、少なくとも**タグまたはブランチ**で現状を残す運用が望ましい（方針のみ・本レポートでは git 操作なし）。


17. 本線 home_dashboard へ移植する場合の注意
--------------------------------------------------------------------------------
- 本線は **Header の money / owner_trust**、**Scroll 内の club_summary / tasks / news / warnings** 等に既に分割。**右サマリーを足す**なら **Header との二重表示**を避けるルールが必要。
- sandbox で右サマリーを試しても、**本線は別レイアウト**になりうるため、「sandbox で決めた列＝本線そのまま」ではない前提でよい。


18. 将来 home DTO に必要そうな項目（右サマリー列を想定する場合）
--------------------------------------------------------------------------------
ユーザー列挙に沿った整理（すべて任意・短文化が前提）:
  - `money`, `owner_trust` … 既存。右列に載せるなら**短い別表現**は export で分岐しうる。
  - `morale`, `salary_room`（または `salary_cap` の派生短文）… CardClubState と**どちらに残すか**を決める。
  - `warnings` … 1 行要約 vs 本線の `CardWarnings` 全文。
  - `recent_form` … Standings カードと分担。
  - `roster_slots`, `expiring_contracts`, `injury_risk`, `fatigue_risk` … 編成・コンディション列の候補。
