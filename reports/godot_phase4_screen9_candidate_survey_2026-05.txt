4. 候補A：戦術 / ローテーションサマリー詳細
--------------------------------------------------------------------------------
【データ】
- 正本: basketball_sim/systems/team_tactics.py（TACTICS_SCHEMA_VERSION, ROTATION_DEFAULTS,
  team_strategy / playbook / rotation / usage_policy / preset_meta 等）
- Team: models/team.py に team_tactics: Dict、strategy 等。
- 先発キー: STARTER_POSITIONS = ("PG","SG","SF","PF","C")。bench_order / target_minutes / sub_policy 等。

【ensure_team_tactics_on_team と readonly】
- ensure_team_tactics_on_team(team) は「Team.team_tactics を正規化した dict で setattr 置換」する（インプレース）。
- get_safe_team_tactics も doc は「読み取り専用用途」だが実装は ensure 呼び出し後に dict コピーを返すため、
  ロード済み Team オブジェクトはメモリ上は書き換わる（save しない export でも方針上は嫌われやすい）。
- roster_readonly.py 冒頭コメント: format_gm_roster_text / compute_auto_role_tags_for_team 等、
  ensure_team_tactics_on_team を呼び得る経路は使わない、と明記済み。
- test_finance_summary_readonly_export.py / test_owner_mission_readonly_export.py に
  export ソースに "ensure_team_tactics" が含まれないことの検査あり → 新 export でも同様の規律が自然。

【安全な readonly DTO の方向性】
- getattr(team, "team_tactics", {}) を copy.deepcopy 等でコピーした上で normalize_team_tactics(raw_copy, valid_player_ids)
  の戻り値のみを JSON 化し、Team には setattr しない（normalize_team_tactics 自体は新 dict を返す設計）。
- valid_player_ids は roster の player_id 列挙から作る（読取のみ）。
- 欠損時は正規化結果でデフォルト埋めされるため「空で落ちる」より「既定表示」に寄る。

【操作UIに見える問題】
- 画面タイトル・注意文を「戦術・ローテーション（閲覧）」「変更は本ゲームの戦術メニューから」等に固定し、
  Godot 側はボタンを増やさない（既存8画面と同型）とリスク低減。

5. 候補B：選手詳細 / 注目選手プロフィール詳細
--------------------------------------------------------------------------------
【Player モデル】
- basketball_sim/models/player.py: salary, contract_years_left, contract_total_years,
  ovr 系属性、age、position、疲労・怪我系、fa_priority_*、acquisition_type 等、プロフィール向きフィールドが豊富。

【ロスターとの差別化】
- roster_readonly は order/name/position/age/ovr/salary/contract/nationality_slot/status の表形式に特化。
- 第9で「1選手の深掘り」はロスター行をクリックできない現状の Godot では、選択UIが要ると難度が跳ねる。
- 「OVR 上位3人」「最年長/最年少」「契約残1年」など **ルール固定の複数カード**なら選択UI不要で Phase4 パターンに合う。

【Season なし】
- 選手オブジェクト自体は team.players に載っていれば成立。シーズン非依存でよい。

6. 候補C：ニュース / 情報一覧詳細
--------------------------------------------------------------------------------
【既存ソース】
- home_dashboard_readonly._news_lines: 順位・勝敗・汎用文言中心。リーグ全体のニュースフィードではない。
- home_dashboard_readonly._task_lines: 施設PT、負傷数、fa_shortlist、owner_mission、定型「戦術確認」行など。
- club_history_readonly: milestones / transactions / finance 断片などを **get_club_history_report_text を使わず**
  自力イテレーションでDTO化（_ensure_history_fields 回避が明記）。

【課題】
- 「ニュース」という1語に集約できる単一正本が無く、DTO は複数 attr の合成設計になる。
- ホーム・クラブ史と被るため、第9の価値は「カテゴリ分け」「期間フィルタ」「重要度」など編集方針の確定が必要。

7. 候補D：契約 / 人事サマリー詳細
--------------------------------------------------------------------------------
【データ】
- Player: 契約年数・年俸・desired_salary、FA志向、獲得経路など。
- finance_summary_readonly: サラリー上限・年俸合計・余力・財務履歴（チーム金計）。
- roster_readonly: 各行 salary / contract 文字列。

【重複と差別化】
- 純粋に表を再掲するだけならロスター/財務と重複し第9として弱い。
- 「契約満了ウィンドウ」「翌年 FA 予備軍」「日本規制枠ストレス」など **集約指標** に振ると価値が出る。
- contract_logic.get_expiring_players 等の利用可否は、**import 先が副作用なし**であることを個別に要確認（本調査では未深掘り）。

【交渉UI誤解】
- 画面名を「契約状況（閲覧）」とし、交渉・更新ボタンを出さない Godot 方針で抑制可能。

11. Cursorとしてのおすすめ順（第9 readonly 向き）
--------------------------------------------------------------------------------
1位: A（戦術/ローテーション閲覧）— データは Team に直結しシーズン非依存。export では ensure を呼ばない方針で
     finance/owner と同じ安全文化に乗せやすい。商品価値が高い。
2位: D（契約/人事サマリー）— 財務・ロスターと被るが「集約ダッシュ」に振れば第9として成立しやすい。
3位: B（注目選手固定スナップショット）— 選択UIを避ける前提なら実装難度は低い。差別化コピーが要る。
4位: F（ドラフト/スカウト）— 価値は高いが空データ・Season 依存の説明コストが上がる。
5位: C（ニュース一覧）— 設計・データ境界の整理が先。第9より「DTO仕様の会議」が重い。
6位: G（杯/国際）— 日程・クラブ史・Season 前提と三方向に引っ張られる。
7位: H（タスク専用）— 現状の home tasks ではホームと二重になりやすく第9のインパクトが薄い。
8位: E（FA市場閲覧）— 価値は高いが誤解リスクと export 新設の検証コストが最も重い → 第9より後でも可。

12. 第9として避けた方がよい候補と理由
--------------------------------------------------------------------------------
- E（FA / 人事候補）を「最優先で第9」にするのは非推奨寄り: 閲覧専用と誤認されにくく、
  既存の「FA市場は人事の閲覧タブ」という Tk 由来の説明責務とも整合が要る。export・Godot の両方でガードが重い。
- H（タスク専用）を第9にするのは非推奨寄り: 中身が home_dashboard の tasks 拡張に留まるなら、
  専用画面を増やすよりホームDTOの拡張で足りる可能性が高い。

13. 次にChatGPT/ユーザーが判断すべきこと
--------------------------------------------------------------------------------
- 第9のプロダクト宣言: 「シーズン未接続セーブでも常に意味がある画面」を最優先にするか、
  「シーズン進行中セーブに特化した画面」を許容するか。
- 戦術DTOを採用する場合: Team オブジェクトをメモリ上で正規化してよいか（ensure 禁止の厳格モード）か、
  dict コピーのみ正規化で十分か（推奨: 後者）。
- 契約サマリー（D）を採用する場合: finance + roster と被らない「1枚の要約」の定義（KPI 3〜6個など）。
- FA（E）を第10以降に回す場合、Godot カード導線を「人事」列に寄せるかどうか（ナビ設計との兼ね合い）。
