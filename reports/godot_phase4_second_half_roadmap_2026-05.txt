5. この順番を推す理由
================================================================================

1. **player_detail は 3 導線完成済み。** 次に game_detail を最小接続すれば、
   詳細画面基盤が **player / game の 2 系統** になり Phase 4 の設計意図（6950695 / 319d1bb）に合致。

2. **game_detail は「試合を味わう気持ちよさ」** に直結。日程は既に event_id・detail を export 済み。

3. **player_detail 本格化** は正本・専用 export が要り **重い**。最小 game_detail の後回しでも 3 導線は使える。

4. **Python 本体復帰は重要** だが、game_detail 最小まで作ると
   「Godot 読み取り詳細導線 Phase 4 第 1 段」として **きれいに区切れる**。

5. **細 UI 調整** は最近継続済み。Steam 品質では **中身優先** のタイミング。

6. **安定性:** D-0 仕様レポート → D-1 scene のみ → D-2 接続のみ、の分割は Phase C と同型で実績あり。

7. **push 運用:** 各 milestone で commit + push し、ChatGPT コネクター等と GitHub 同期を維持。


6. やらないこと
================================================================================

- 今すぐ **player_detail 専用 export** を作らない
- 今すぐ **player_detail を大改修** しない
- 今すぐ **schedule の既存「詳細」** を消さない
- 今すぐ **Godot から Python を自動起動** しない
- 今すぐ **save / 編集 / 進行操作** を Godot 側につながない
- **細かい余白調整** を主目的にしない
- game_detail 実装前に **仕様レポートをスキップ** しない
- RiskRows / 契約リスク行から player_detail へ **無理に接続** しない
