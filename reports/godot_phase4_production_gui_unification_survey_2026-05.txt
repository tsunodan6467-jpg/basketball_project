6. 共通化できているパターン
--------------------------------------------------------------------------------
- JSON 読込: FileAccess → get_as_text → JSON.parse_string → Dictionary 検証。失敗時は次候補へ（各 .gd でほぼ同型）。
- 候補パス配列: `res://data/<screen>_from_python.json` を先頭にした Array。
- DataSourceLabel（%DataSourceLabel）: 読み込み成功 URI を人間向けキャプションに変換して表示。
- レイアウト骨格: Background ColorRect → Margin → RootCol(VBox) → HeaderCard(Panel) → Scroll → 本文。
- StyleBoxFlat: header / chip / summary / card などの sub_resource が各 .tscn に繰り返し定義（色・角丸・margin が近い値でコピペ類似）。
- ScrollContainer + VBox 系のスクロール領域。
- 1280×720 前提（project.godot）。
- UID: 主要シーンは uid 行と ext_resource の script uid を持つ画面がある（過去に UID 参照事故の修正コミットあり）。方針としては「.gd.uid と .tscn の対応を崩さない」ことが重要。
- change_scene_to_file のみで画面遷移（autoload ナビ無し）。


7. まだ画面ごとに散らばっているパターン
--------------------------------------------------------------------------------
- StyleBoxFlat の sub_resource が各 .tscn に独立定義（Theme リソース化されていない）。色数値の微差・更新漏れリスク。
- ヘッダー構造の2系統（上記 §5）。共通化・部分テンプレ化の障壁。
- 動的 UI: Label.new() / add_child を多用する画面（roster, club_history, standings, schedule, facility, finance, owner_mission, tactics, contract）。行レイアウト・区切り線の付け方が画面ごとに微妙に異なる。
- schedule は Panel を組み立てるなど表現がやや独自。
- home_dashboard のみが「ダッシュボード用の多数の @onready と _apply_snapshot」で他画面と役割が大きく異なる。
- finance_summary_view.tscn の ext_resource は path のみで script に uid 行が無い例あり（他画面とエディタメタの揃い具合が不均一）。


8. 現在の本番 GUI 化に向けた課題
--------------------------------------------------------------------------------
- 共通 UI 部品（シーン片・スクリプト基底クラス）が未整備。複製編集が基本。
- 二重導線（Header 5 vs カード）: 本番ではどちらを「正」とするか未決。現状は意図的な足場だが、ユーザー認知コスト。
- 経営列 3 ボタン: 1280 幅で縦にコンテンツが伸びると、ホームスクロール内のカード列の縦圧力が増える。第11画面をカードに足す余地は狭い。
- Godot エディタの再保存で .tscn のプロパティ順・uid 周りの差分が出やすい運用（チームルールで diff を抑える必要）。
- データ更新は PowerShell 手動。本番体験とはギャップ。res:// 配下への書き込みと配布ビルド時の user:// 論点は別タスク。
- ビジュアルは「暗めフラット＋カード」のプロトタイプ域。90年代 GM 風の高密度・ロゴ・情報階層は未着手。
- Steam 品質（入力・フォーカス・ローカライズ・解像度スケール・アクセシビリティ）は未検証レベル。


9. 候補 A：現行10画面を維持し UI 共通化だけ進める
--------------------------------------------------------------------------------
利点:
  - 既存シーンの責務（JSON → 表示）を壊しにくい。
  - 小さな PR に分割しやすく、1 コミットで revert しやすい。
  - Theme 化の前段として「定数テーブル（色・フォント）」だけ共通化も可能。

欠点:
  - 本番「らしさ」の向上は限定的。ユーザー体験のインパクトは B/E に劣る。

リスク:
  - 共通化の途中でノードパス（% 名）を変えると全画面差分が爆発しうる。

今やるべきか:
  ◎ 最優先クラス。一本化の前に「安全な土台」を厚くするのに最適。

16. 共通化できそうな UI 部品
--------------------------------------------------------------------------------
- ヘッダーカード一式: ReadonlyBadge + ModeStrip +（HomeNavButton の親行）。2系統を1つの PackedScene に寄せる候補。
- DataSourceLabel 更新ロジック: 小さな static ヘルパまたは mixin 相当（GDScript なら基底クラス）に集約可。
- 「ホームへ戻る」ボタン: 文言・tooltip・シグナル接続のテンプレート化。
- スクロール本文の「見出し Label + 下線 HSeparator」パターン。
- 注意・警告用の Panel（warn Style）の sub_resource 共通化。
- JSON ローダ: 候補配列を渡して Dictionary を返す共通関数（型チェック方針は要合意）。
- エラー表示: 「読込失敗」ラベル＋本文クリアの共通処理。


17. 共通化すると危険な部分
--------------------------------------------------------------------------------
- 各画面固有の JSON キー参照・整形（_txt, _join_lines 等の微妙な差）。
- 動的生成ブロック（選手行・ミッション行・履歴行）: 1つの「汎用テーブル」に押し込むと読めなくなる。
- JSON スキーマ差分を無理に一般化すると DTO 変更時に全画面が連鎖破綻。
- UID と unique_name_in_owner（%）のリネーム: 触ると全シーンの手直し。
- scene ツリーの深さ変更: シグナル接続パスが一斉にズレる。


18. 90年代〜PS1/セガサターン風 GM 画面との距離

  - ホーム JSON に次戦・順位・資金・オーナー信頼・タスク・ニュース等の「束ね」は既にある（密度はまだ仮）。

足りない点:

  - 1画面への情報圧縮（ウィジェット密度・タイポ階層・グリッド）は本番水準ではない。
  - 操作フィードバック・サウンド・演出は未接続。
  - 「レトロ感と現代的な可読性」のトーン設計（フォント・コントラスト・アンチエイリアス方針）は未文書化のコード状態。


19. Steam 販売品質に向けて足りない要素（高レベル）
--------------------------------------------------------------------------------
- 一貫したビジュアルガイド（色・タイポ・余白・コンポーネント命名）。
- 多言語・長文ラベルでのレイアウト破綻テスト。
- 解像度・UI スケール・フルスクリーン時の挙動。
- 入力デバイス（キーボードフォーカス・ゲームパッドは要方針）。
- アクセシビリティ（色だけに依存しない警告表現など）。
- エラー時のユーザー向け文言とリカバリ導線（JSON 欠損・部分キー欠落）。
- 実データ負荷（選手数・履歴件数）でのスクロール性能と UI ストレス。
- ビルドパイプライン（エクスポートテンプレ・user:// データ・Steam デポ構成）。
