4. 候補 A：契約・人事サマリーへ限定適用
--------------------------------------------------------------------------------
利点:
  - シーン構造が finance / tactics 等と同型（HeaderTopRow + ReadonlyBadge + ModeStrip + HomeNav、複数 Summary 系 Panel、Scroll）。
  - phase4_readonly_core のバリエーション（Header / Chip / Summary / Warning / DataSource / Table 系）と **対応関係が取りやすい**。
  - ホーム・他画面への **波及が限定的**（ハブの見た目は変わらない）。
  - 直近追加で「第 10 画面として」検証対象として自然。

欠点:
  - .tscn 内の `theme_override_styles/panel = SubResource("StyleBoxFlat_*")` が **多数**（summary カードが複数ブロック）。正本化には削減 or 置換作業が必要。
  - contract_personnel_summary_view.gd に **動的 Label**（リスク行・選手行）と `add_theme_*_override` あり。Theme だけでは表部分の統一は **限定的**（§11）。

触りそうなファイル:
  - 主: godot/scenes/contract_personnel_summary_view.tscn
  - 任意（表の色を Theme に寄せる場合）: godot/scripts/contract_personnel_summary_view.gd
  - 任意（トークン微調整）: godot/themes/phase4_readonly_core.tres

差分規模（実装時の目安）:
  - **最小**: ext_resource（Theme）追加 + ルート Control に `theme = ExtResource(...)` のみ → **見た目はほぼ不変**（各 Panel の `theme_override_styles/panel` が優先されるため。§9〜10 参照）。
  - **効果あり**: ヘッダー Panel 等から `theme_override_styles/panel` を外し `theme_type_variation = &"Phase4HeaderCard"` 等に置換 + 未使用 SubResource 削除 → **中規模 .tscn diff**（数十行〜）。
  - **フル**: 全 Panel の override 撤去 + Label の per-node override をバリエーションへ移行 → **大**。

安全性:
  - 導線（ホーム ↔ 契約人事）は .gd のシーンパスに依存。Theme 適用のみなら **壊しにくい**。
  - UID は既存 uid://c8contractpsview1。編集で先頭行が揺れるリスクは **全シーン共通**だが対象ファイルは 1 本に限定しやすい。

今やるべきか:
  ◎ **第 1 本番シーンへの適用先として最優先クラス**（preview と構造が近く、ホームより安全、ロスターより .tscn 中心で進めやすい）。


5. 候補 B：ロスターへ限定適用
--------------------------------------------------------------------------------
利点:
  - GM 体験の中核。表主体の本番化への **商品価値が最大級**。
  - .tscn 内 StyleBoxFlat は **2 本のみ**（header, chip）。summary 用 SubResource が無く、カード類が薄い。

欠点:
  - 表本体は **GDScript で Label 行生成**し、**コード側で font_color / font_size を上書き**。Theme の Label 既定や Phase4TableCell を活かすには **.gd 変更がほぼ必須**。
  - 行数・列数がデータ依存のため、**1280 での折り返し・高さ**の回帰テストが重い。

触りそうなファイル:
  - godot/scenes/roster_view.tscn（ルート theme + ヘッダ周りの override 整理）
  - godot/scripts/roster_view.gd（動的ラベルの theme_type_variation 化 or override 削除）

差分規模:
  - .tscn 単体は小さくできるが、**見た目を変えるなら .gd 同梱が現実的** → 1 PR の表面積は A より大きくなりがち。

安全性:
  - ゲーム体験の要。表示崩れの **体感インパクトが大きい**。
  - HeaderNavRow は .tscn 内にあり、Theme 適用で **意図せず Button 見た目が変わる**可能性（ルート theme が子 Button に伝播）。要確認。

今やるべきか:
  △ **技術的には第 2 候補**（A で Theme 運用を固めた後、表 + コードのセットで取り組む方が安全）。

9. Theme を root に適用した場合の効果見込み
--------------------------------------------------------------------------------
- Godot の Theme は子孫に伝播するが、**各ノードの `theme_override_*` は親 Theme より優先**される。
- 現状の各 readonly シーンでは、PanelContainer に **`theme_override_styles/panel = SubResource("StyleBoxFlat_*")`** が明示されているケースが多い。
  → ルートに `theme = ExtResource("phase4_readonly_core.tres")` を付けた **だけ**では、これらの Panel は **従来の SubResource の見た目のまま**の可能性が高い。
- **効果が出やすいノード**: override を付けていない Label/Button、または override を削った後のノード。
- **Phase4*** バリエーションは `theme_type_variation` を付与したノードにのみ適用される（Theme 定義済み）。


10. 既存 StyleBoxFlat との衝突・差分リスク
--------------------------------------------------------------------------------
- **衝突というより「上書き優先」**: 同じ見た目を Theme に寄せるには、シーン側 SubResource 参照の **解除が必要**。
- SubResource を削除する場合、**load_steps と SubResource ブロック全体**の整合を取る必要があり、.tscn の **構造 diff が読みにくくなる**リスク。
- 段階戦略: (1) ext_resource + root theme のみ（見た目ほぼ据え置きで「配線確認」）→ (2) HeaderCard だけ variation + override 削除の **パイロット** → (3) 他 Panel へ展開。
- Theme の PanelContainer 既定（summary）と既存 summary の StyleBox は **数値同一**のため、override 解除後も **見た目差分は小さく抑えられる**設計になっている（preview で確認済みの方向性）。


11. script 変更が必要になりそうな箇所
--------------------------------------------------------------------------------
- **必須になりやすい**: roster_view.gd（動的 Label の色・サイズを Theme / variation に寄せる場合）。
- **任意**: contract_personnel_summary_view.gd（動的行を Phase4TableCell に揃える場合。現状も `_style_cell` 的なヘルパで add_theme しており、Theme だけでは二重管理になりうる）。
- **ほぼ不要**: home_dashboard.gd（grep で Label.new / add_theme 無し。主に .tscn の整理）。
- finance_summary_view.gd / owner_mission_view.gd は動的 Label あり。**表・履歴を Theme 化するなら** contract と同様に .gd 触りうる。


12. UID / scene 再シリアライズの注意点
--------------------------------------------------------------------------------
- 先頭 `[gd_scene load_steps=N format=3 uid="..."]` がエディタ保存で **load_steps 欠落・uid 再採番**される事例あり（ユーザー報告: theme_preview）。本番シーン編集時も同様の可能性。
- 対策案: **手編集とエディタ保存の運用ルール**、レビューで先頭行を必ず確認、必要なら `git checkout --` で戻す。
- ext_resource に Theme を足すと **load_steps インクリメント**が必要。数え違いでロード失敗しうるため、保存後に Godot で再オープン確認を推奨。
