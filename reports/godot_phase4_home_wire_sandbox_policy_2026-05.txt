3. home_production_wire_preview.tscn の位置づけ（sandbox としての意味）
--------------------------------------------------------------------------------
- **本番 GUI 研究用 sandbox**: 本線を壊さずに、左ナビ＋上部クラブ帯＋ホーム密度・色味を
  **目視・ピクセル単位で試す場**。`theme_preview.tscn` が Theme variation の第 0 段であるのと
  同型で、本ファイルは **レイアウト／情報設計の第 0 段** と位置づける。
- **契約のないプロトタイプ**: JSON・Python・シグナル・遷移を持たないため、失敗しても
  本線・export・pytest に波及しない。
- **合意形成用アーティファクト**: スクリーンショット・レビューで「本番に近い骨」を共有する。
- **正式名称の呼び方（推奨）**: 会話・PR・将来 README では「本番ホームワイヤー sandbox」
  または「home_production_wire_preview（sandbox）」と明示し、`home_dashboard` と混同しない。


4. 本線 home_dashboard.tscn との役割分担
--------------------------------------------------------------------------------
| 観点 | 本線 home_dashboard | sandbox home_production_wire_preview |
|------|---------------------|--------------------------------------|
| 目的 | 10 画面導線・JSON 表示の**正** | レイアウト・密度・ナビ骨格の**研究** |
| script | あり（読込・遷移） | なし |
| データ | from_python / mock | 固定文言のみ |
| main_scene | 既定の起動シーン | 変更しない（F6 単体実行） |
| pytest / export | 対象・運用の基準 | 対象外 |
| Theme | Header 等 限定適用＋Scroll 従来 | 全体に Theme、variation とシーン内 Style で実験 |
| ドキュメント上の役割 | NAVIGATION の「仮ハブ」 | 長期左ナビ案の**視覚たたき台** |

本線は **破壊しないことを最優先**。sandbox は **意図的に破ってよいが本線に飛ばさない**。


5. sandbox で試してよいこと
--------------------------------------------------------------------------------
- 左レール幅（180〜220px 帯）・余白・縦間隔。
- 大分類ラベルの並び・文言短縮・「現在地」強調表現（Chip / 枠 / フォント）。
- 上部クラブ帯の行数・1 行に詰める指標の取捨（節・順位・資金・信頼の優先順位）。
- 中央カードの数・面積・2 列 vs 2.5 列・Scroll 内の縦長さ。
- 低密度／中密度の A/B（同シーン内でノード並べ替え、または別 sandbox シーンを増やす検討）。
- 右サマリー列「あり」版のワイヤー（別 .tscn に分けると diff が追いやすい場合あり）。
- 色・質感: シーン内 StyleBoxFlat の bg/border、Theme variation の組み合わせ（**phase4_readonly_core.tres は触らず**）。
- ロゴ／アイコン用の空 Panel（プレースホルダ）・スポーツ感の帯ライン。
- 下部メッセージ帯の有無・高さ・トーン。
- 左レールを Button 風に見せる（押下シグナルは付けない／disabled 固定でも可）。


6. sandbox でまだ避けること
--------------------------------------------------------------------------------
- **実データ接続**（home_dashboard_from_python.json 等を読む .gd 追加）。
- **Python 自動起動**・一括 export のトリガ結合。
- **本線への遷移**（change_scene、本線への「開く」ボタンを sandbox に置くと誤認・保守負荷）。
- **main_scene の差し替え**（全員の起動画面が変わる）。
- **複雑な script 化**（状態管理・共通ナビ Autoload 相当まで sandbox に詰め込まない）。
- **home_dashboard の改造を sandbox の結果で同時に行う**（反映は条件付き・別コミットで）。
- **10 画面共通の親レイアウト**を sandbox だけで先に確定させ、本線と二重管理にしない
  （共通シェルは本線移行フェーズで設計）。


7. 本線へ反映する条件
--------------------------------------------------------------------------------
以下を満たしたうえで、**別タスク・別コミット**で本線へ移植する。

1. 1280×720（および想定最小解像度があればそれも）で破綻しない。
2. 当事者による目視合意（情報の優先順位・左カテゴリ名・帯の行数）。
3. Godot 保存後の **UID / load_steps の差分運用**が手元ルールに乗る（§12）。
4. 左レールの **大分類が確定**（10 画面を左に全部並べない方針と整合）。
5. ホームに載せる／載せない情報が INFORMATION_ARCHITECTURE の方針と矛盾しない。
6. from_python の **home DTO に必要なフィールド**が整理済み（表示追加が本線で必要なら
   Python export は別 PR）。
7. 本線移行を **1 コミット単位**に分割できる（例: 左レールのみ .tscn → 次コミットで帯文案 等）。

反映時は **本線の script・connection・JSON 契約を一度に変えない**。


8. 左サイドナビ研究方針
--------------------------------------------------------------------------------
- **大分類のみ左**（ホーム／チーム／リーグ／経営／クラブ）を維持し、10 画面は中央または
  第 2 階層で選ぶ前提を sandbox で検証する（NAVIGATION・production_home_layout 調査と一致）。
- HeaderNavRow（本線 5 ボタン）との **二重導線解消**は本線移行時にまとめて決める。sandbox では
  「左だけの世界」を先に理想形として描く。
- 将来: 左を Button + 最小 script にするか、本線移行時に共通シーンに切り出すかは未決。sandbox
  段階では **見た目優先**でよい。


9. ホーム情報密度研究方針
--------------------------------------------------------------------------------
- ユーザー評価「現状くらいの密度が良さそう」を前提に、**削る候補**（長文ニュース・タスク全文・
  詳細財務）と **残す候補**（次戦・順位要約・短タスク・ヘッドライン）を sandbox で入れ替え試験。
- INFORMATION_ARCHITECTURE の「ホームに残す／分ける」と照合し、sandbox のカード集合を
  タグ付け（must / nice / later）すると本線移行が楽。


10. 色・質感研究方針
--------------------------------------------------------------------------------
- **白固定にしない**（ユーザー要件）。濃背景＋帯・カードは既存 Theme + シーン内 StyleBox で試す。
- phase4_readonly_core.tres を広く変える前に、sandbox の SubResource と Label variation で
  **パレット候補を 2〜3 系統**試し、確定後に Theme へ昇格する流れが安全。
- スポーツ感・クラブ感はタイポグラフィ（サイズ階層）・帯の区切り線・プレースホルダ枠から先に
  付ける。画像素材は最後でもよい。


11. 実データ接続をまだしない理由
--------------------------------------------------------------------------------
- sandbox の価値は **レイアウトと優先度の高速反復**。JSON 接続すると export・DTO・失敗時表示と
  結びつき、本線と同じレビュー負荷がかかる。
- 実データを入れる段階では **本線または本線に近い script 付きシーン**で行い、pytest で守るのが
  正本の規律（README・NAVIGATION の位置づけと一致）。
- どうしても DTO 形を試す場合は、将来案 B（別 mock JSON + 薄い .gd）を**別ファイル**で検討し、
  本線のパスと混同しない命名にする（過去調査レポート参照）。**今回の sandbox 方針では必須ではない**。


12. UID再シリアライズ運用
--------------------------------------------------------------------------------
- Godot エディタで sandbox を開き保存すると、**先頭行の uid:// や load_steps** が変わることがある。
- 作業フロー推奨:
  1. 編集後 `git diff -- godot/scenes/home_production_wire_preview.tscn` を必ず見る。
  2. **意図したノード変更以外**（先頭行のみ等）なら、コミット前に
     `git checkout HEAD -- godot/scenes/home_production_wire_preview.tscn` で当該ファイルを戻し、
     エディタ上では「先頭行を手で元に戻す」か、**安定版を正**として再適用する。
  3. **コミット済みの安定形**をチームの「sandbox 正本」としてタグ付けするか、PR 説明に
     「意図する先頭 uid」を書く。
- sandbox は本線より編集頻度が高い想定のため、**小コミット・差分レビュー**を徹底する。
