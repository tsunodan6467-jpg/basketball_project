4. 参考画像から読み取れる色・質感（推定・画像なし前提）
--------------------------------------------------------------------------------
一般的な「90年代〜PS1/セガサターン期スポーツ GM シミュ」の参照方向としてよく挙がる要素:
  - 画面外周: 濃紺〜黒に近い紺（チーム／リーグの重厚感）。
  - 情報カード: ベージュ・クリーム・薄茶の紙質感（印刷物のスタッツ画面の比喩）。
  - アクセント: 金・琥珀・チームカラー 1 色（見出し帯・選択行）。
  - タイポ: 太めタイトル＋等幅に近いデータ行（現代の Inter 系より「角」がある印象）。
  - 区切り: 明確な 1px〜2px の枠・帯の水平ライン（質感はフラット～ごく薄い立体）。

本 sandbox はすでに「外周濃色＋カード淡色」なので方向性は A に近く、B（ベージュをもう一段「紙」に寄せる）
を足すと参考方向に近づきやすい。

11. 既存 Theme（phase4_readonly_core.tres）との相性
--------------------------------------------------------------------------------
- sandbox はルートで Theme 全体を読み込んでいるため、Summary / Header / Warning の「淡色カード」は
  グローバルに効く。濃色部分は主にシーン内 SubResource で上書きしている。
- OnDark 系は左レール・ロゴ枠内など「濃地の上」に相性よし。Phase4Title 等の「濃文字」variation は
  淡カード上で使われている箇所（ClubBand 右列）と、将来ベージュ化したときのコントラスト検証が必要。
- Phase4ChipPanel は未使用だが、将来「現在地」チップを軽く出すときの候補。
- Theme を急に広く変えると、既に Theme 適用済みの契約人事・ロスター・ホーム Header の見え方が連動する。
  sandbox 単体の実験では「シーン側 theme_override_styles」で Theme を局所上書きするのが安全。


12. sandbox 内 SubResource だけで試せる範囲
--------------------------------------------------------------------------------
十分試せる:
  - Background ColorRect の色。
  - StyleBoxFlat_rail / nav_item / nav_active / logo_slot / bottom の bg／border／角丸／余白。
  - ClubBand の Panel に対し theme_override_styles/panel で、HeaderCard の代わりにシーン専用
    StyleBox を差し込む（Theme 非変更で「ベージュ帯」化可能）。
  - BottomStrip ラベルは OnDark のままなので、帯の地色とのコントラスト調整も SubResource のみで可。

やや手数が増えるが Theme なしでも可能:
  - 各 SummaryCard／WarningCard の PanelContainer に、シーン専用の StyleBoxFlat を個別割当＋
    必要なら Label の theme_override_colors（シーン上のみ）でベージュカード上の文字色を微調整。
  - その場合 load_steps が増える・ノード属性が増えるが、本番他画面への副作用は出にくい。

SubResource だけでは限界:
  - タイポグラフィ階層の根本変更（フォントリソース差し替え）はプロジェクト全体方針になりやすい。
  - チーム可変色のデータ駆動（案 D 本番版）。


13. Theme ファイル変更が必要になる範囲
--------------------------------------------------------------------------------
Theme を触るべきタイミングの目安:
  - 同じパレットを **複数シーン**（本線 home_dashboard の Scroll 以下、他閲覧画面）に広げる決定が
    固まったとき。
  - 「Phase4SummaryCard の定義そのものを変える」と README の Theme ロードマップに影響するため、
    **合意＋回帰目視**をセットにする。
  - sandbox で 2〜3 系統試して「これに統一」と決める前に Theme をいじると、A/B 比較がしづらい。

Theme 変更が有利なケース:
  - ベージュ系（案 B）を契約人事・ロスター・ホームカードまで一気に揃え、**白ベース検証版の意味を
    「アイボリー／ベージュベース検証」に更新する**とき。
  - OnDark の色数を増やし、ダークパネル統一（案 E）を本線表にも展開するとき。


14. 本線移植時のリスク
--------------------------------------------------------------------------------
- 本線 home_dashboard は HeaderCard のみ Theme、Scroll 以下は従来 StyleBox。sandbox で美しくても
  **二段構えのデザインシステム**が長期化するリスク。
- pytest は色に関しては無いが、**ノード構造・theme パス変更**で UID／参照破壊が出やすい。
- 「sandbox でだけ通用する override の山」を本線にコピペすると保守不能になりやすい。
  移植時は **Theme か共有 StyleBox リソース化**に一度整理してからが安全。


15. 比較用に別 scene を作るべきか、既存 sandbox を上書きするべきか
--------------------------------------------------------------------------------
方針レポート（sandbox_policy）の考え方と整合する推奨:

  ■ パレット A/B を「並べてスクショ比較」したい場合
    - 別シーン（例: home_production_wire_preview_palette_b.tscn）に fork するか、
      同一シーン内に「切替用の非表示ブランチ」を置くか。
    - **色の系統が大きく違う**なら別シーン推奨: git diff が読みやすく、UID 事故時の checkout もしやすい。
    - デメリット: ファイル増・同期漏れ。最大 2〜3 個程度にキャップする運用が現実的。

  ■ 微調整（案 F や案 A の微差）
    - 既存 sandbox を上書きでよい。1 コミット 1 意図を守る。

  ■ 中間案
    - 同一 .tscn 内で「PaletteRootA / PaletteRootB」をトップで切替えるのは script なしだと運用が重い。
      非推奨に近い（手動で visible を切り替える運用は人為ミスが出やすい）。


16. 商品価値・Steam 品質への寄与
--------------------------------------------------------------------------------
- Steam ページのキャプションでは「一目でジャンル（スポーツ経営）と時代感が伝わる」色が有利。
  案 B・C はサムネ／キャプションでの訴求力が高くなりやすい。
- 案 A・F は「信頼できる UI」には寄与するが、差別化は弱い。
- 案 E はスクリーンショットが暗く見え、インディー経営シムとしてはありだがマーケ上は要検証。
- 案 D はストーリー性は高いが、実装・ライセンス・チーム差分のコストと見合いを取る必要あり。


17. Cursor としてのおすすめ順（試す優先度）
--------------------------------------------------------------------------------
1. 案 F … 最も安全。質感・階層の改善だけで全体の完成度を上げつつ、後続の色替えの土台にする。
2. 案 C … 既存の青紫・琥珀アクセントを「意図したパレット」として整理。SubResource のみで試しやすい。
3. 案 B … 参考方向に最も寄せる本丸。まず sandbox の ClubBand＋1〜2 枚のカードだけベージュ化して
         目視合意を取るのがリスク低。
