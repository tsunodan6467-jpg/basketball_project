# `_calculate_offer`: `payroll_before >= soft_cap` 即 0 返却の方針メモ（オフFA文脈）

**作成日**: 2026-04-06  
**文書の性質**: **設計・方針メモ（コード変更なし）**。根拠観測: `docs/FA_CALCULATE_OFFER_CAUSE_PLAYCHECK_2026-04.md`（**S1** と **S6** の分離）。実装: `basketball_sim/systems/free_agency.py` の `_calculate_offer`。

---

## 1. 文書の目的

`payroll_before >= soft_cap` による **即 `return 0`** は、**`payroll_budget` 同期や buffer（S6 対策）では到達しない最上流の 0 要因**である。本メモは **オフFA文脈を含め**、この厳しさを **維持／緩和／別表現**のどれに寄せるべきか判断するための材料を整理する。**今回はコードを変えない**。

---

## 2. 現在の soft cap early return の意味

- **位置**: `_calculate_offer` 冒頭、`payroll_before`・`soft_cap` 取得の直後。  
- **条件**: **`if payroll_before >= soft_cap: return 0`**。この時点で **ベース年俸・ボーナス・ハード跨ぎ・`payroll_budget`・贅沢税**など**以降の分岐は一切実行されない**（診断関数 `_calculate_offer_diagnostic` でも同順）。  
- **守っているもの（設計上の読み）**  
  - **サラリー総額がソフトキャップに達したチームから、新規FA契約によるさらなるペイロール積み上げの「交渉オファー芯」を出さない**、という **強いキャップゲート**。  
  - 同一関数の後段には **`payroll_after > soft_cap` への押し戻し**や **ハード超え時の低額のみ**など、**「まだ soft 未満なら」動く**細かいクリップがあるが、**既に soft 以上なら入口で遮断**される。  
- **明示コメントは薄い**ため、**「AI暴走防止」単体と断定はできない**が、結果として **キャップ超過局面でのオファー生成をゼロに固定**している。

---

## 3. オフFA文脈で見たときの違和感 / 妥当性

- **CPU 本格FA・インシーズンFA**: `_calculate_offer` を共有するため、**soft 到達チームは本格式のオファー芯が常に 0**。`conduct_free_agency` 側の他ロジックが別途動く可能性はあるが、**この関数単体では「もう張り付いている」チームに芯を立てない**というメッセージが一貫する。  
- **オフ手動FA**（`offseason_manual_fa_offer_and_years`）: 主経路は同じ `_calculate_offer`。**S1 で `core_offer == 0`** のとき、**`get_team_fa_signing_limit` に正の余地があり**、かつ **他条件を満たせば** **estimate フォールバック**で表示＝契約を立てうる。  
  - つまり **「キャップ上はまだ指一本」だが本格式は 0** と **「キャップ上も実質ゼロ」** の両方が起きうる。前者は **フォールバックで救済**、後者は **UI 上も獲得不能に近い**。  
- **違和感の論点**: **ソフトキャップ「ライン」到達を、FA 交渉の入口で完全停止とみなすのが、オフの「補強の最後のチャンス」という体験と両立するか**。プレイチェック系メモでは **S6 型は buffer で緩めたが、S1 は無傷**。

---

## 4. 取りうる方針の比較

| 方針 | 概要 | 守れるもの | 失いやすいもの / リスク |
|------|------|------------|-------------------------|
| **現仕様維持** | soft 到達なら **常に即 0** | **ルール単純・説明容易**・下流との役割分担が明確 | **オフ・CPU で「見かけ上の枠はあるのに芯だけ 0」**の説明負荷。フォールバック依存。 |
| **緩和A: 低額例外** | 例: **MIN_SALARY 帯だけ**は後段に流し `room_to_soft` 等でクリップ | **「完全停止」ではなく最低限の動き** | **例外の線引き**・テスト爆増・CPU が低額連発する見え方。 |
| **緩和B: オフ手動のみ別扱い** | 手動経路だけ **S1 をスキップ**または別芯 | **CPU 本格の厳しさは維持** | **手動と CPU の非対称**の説明・二重実装。設計書で既に「同型」を重視しているため **摩擦が大きい**。 |
| **緩和C: 早期 return 撤廃し下流へ** | S1 をやめ、**常に後段の `payroll_after > soft_cap` 等に任せる** | **単一パイプライン**でクリップの列だけ見ればよい | **現状よりオファーが立ちやすく**、**意図せぬ高額化**を tax/他クリップだけで抑えられるかの検証が要る。 |
| **別表現: 仕様として「到達＝補強不可」と明文化** | コードは維持し、**UI/ヘルプで期待値を揃える** | **実装リスクゼロ** | **プレイ不満**は残る。 |

比較の軸は **「何を守りたいか」**: **(a) キャップの絶対ゲート**、**(b) オフの体験**、**(c) 手動と CPU の対称性**、**(d) 実装の単純さ**。

## 5. 今の段階での推奨判断

- **本体 `_calculate_offer` をいますぐ書き換える段階にはまだ進めない**のが安全。先に **プロダクト側で「soft cap 到達チームに、FA で何を許すか」を1行で言語化**し、**受入基準（CPU 挙動・手動表示・セーブ互換）**を添えるべき。  
- **方向性のたたき台（断定ではない）**:  
  - **「到達＝新規の意味のある補強はしない」**を維持するなら、**現仕様維持＋別表現（説明・UI）**がコスト最小。  
  - **「到達でも最低限の契約は成立させたい」**なら、**緩和A（極小例外の定義を先に固定）**か **緩和C（下流統合）**のどちらかを、**数値シミュレーション付き**で選ぶ。  
  - **緩和B**は、**手動＝同型**という既存方針と衝突しやすく、**最後の手段**に近い。

## 7. 次に実装で触るべき対象（1つだけ）

**プロダクト短文（1行）＋受入基準を、コードではなく `docs` に確定させる短い「決裁メモ」（次コミット・1ファイル）。** 記載内容の例: 「soft cap 到達時: FA 新規オファーは **常に不可** / **○円未満のみ可** / **下流クリップに統一** のいずれか」と、「そのとき CPU 本格FA とオフ手動FAは **対称を維持するか**」。

- **なぜ最も妥当か**  
  - S1 は **最上流**であり、**buffer・S6 調整と独立**。いきなり分岐をいじると **意図しないチーム全体の FA 分布**が変わる。**文言で合意がないまま実装するとロールバックコストが大きい**。  

- **何がまだ残るか**  
  - 合意後の **`_calculate_offer` 最小差分**、**診断関数の同期更新**、**回帰テスト**、必要なら **実セーブでの S1 頻度の目安**（合意を裏付けるか検証するため）。
