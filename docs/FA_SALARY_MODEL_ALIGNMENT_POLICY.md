## 0. この文書の使い方

- **何のためか**: 監査で判明した **二系統の年俸モデル**（`estimate_fa_market_value` と `_calculate_offer`）について、**全面統一で飛ばず**、**第1調整のスコープと止めどころ**を1本に決める。
- **コード変更ではない**: 本書の作成・更新のみ。係数変更・関数移植は **別コミット**。
- **監査メモとの分担**: `FA_SALARY_ESTIMATE_AUDIT.md` が **「いま何が起きているか」**、本書が **「次にどこを揃えるか」**。事実と矛盾する方針は取らない（監査を正とする）。

## 1. 現在の不整合の整理

| 経路 | 年俸の決定（監査メモの要約） |
|------|------------------------------|
| **手動FA（オフGUI・インシーズンGUI）の表示** | `estimate_fa_market_value` |
| **手動FAの実契約** | `sign_free_agent` → **同じく** `estimate_fa_market_value` |
| **インシーズン CPU 補強** | `sign_free_agent` → **estimate** |
| **オフ CPU 本格FA** | `_calculate_offer`（**estimate 不使用**） |

**プレイヤー体験で違和感になりやすい箇所（推測・未検証を含む）**

- **オフ**で、直前に手動で取った選手の **表示／契約年俸**が **estimate 系**なのに対し、**直後の CPU 本格FA のログ**は **`_calculate_offer` スケール**で動くため、**「同じFAなのに金額の感じが違う」**と感じうる。
- ロスター上の他選手の年俸（別ルールで生成されうる）と **estimate** を比べると **手動FAだけ安く見える**可能性（**未確認**: セーブ依存）。

## 2. 揃え方の候補

| 案 | メリット | デメリット | 分かりやすさ | 既存コードへの影響 | 今の段階に合うか |
|----|----------|------------|--------------|-------------------|------------------|
| **A. 手動FAを `_calculate_offer` 系へ寄せる** | オフ本格FAと **同じ「チーム×選手×財務状況」**のオファー観。違和感の主因に直撃しうる。 | `sign_free_agent` や GUI・`precheck_user_fa_sign` が **estimate 前提**の箇所を **見直し**（**範囲次第で中〜大**）。`free_agency` ↔ `free_agent_market` の **依存**に注意。 | **高**（オフ文脈では「CPUと同じ土俵」と説明しやすい） | **中〜大**（触る範囲を **オフ手動だけ**に限定すれば **中**） | **第1弾はスコープを狭めれば適合** |
| **B. オフCPU本格FAを estimate 系へ寄せる** | `sign_free_agent` 側を広げずに **一系統**にできる。 | **`conduct_free_agency` のバランス・ログ・抽受**に波及。**市場全体の再調整**になりやすい。ユーザー指示の **「CPU本格FAを壊さない」**と相性が悪い。 | **中** | **大** | **初手非推奨** |
| **C. モデルは分けたまま GUI 説明のみ** | 実装最小。**回帰リスク**が低い。 | **金額の低さ自体は変わらない**（体感の主訴が残りうる）。 | **低〜中**（「なぜ安いか」は説明で一部緩和） | **極小** | **緊急の誠実さだけなら可** |
| **D. オフ手動だけ `_calculate_offer`、インシーズンは estimate** | **オフでの「CPU直前」との比較**にフォーカス。**インシーズンCPU補強**や人事GUIは **当面そのまま**で回帰を限定。 | **オフとレギュラーでルールが二種類**と説明が必要。 | **中**（ラベル・案内で **オフ専用**と切る） | **中**（オフGUI＋契約経路に集中） | **第1調整として自然** |
| **（参考）共通関数1本に集約** | 長期の重複排除。 | **設計・移行コスト**が第1弾としては重い。 | 高（完成後） | 大 | **第2段以降** |

## 3. 最小修正候補の比較

| 候補 | 小さく切れるか | プレイヤー価値 | 回帰リスク | 今の優先度に合うか |
|------|----------------|----------------|------------|---------------------|
| **オフ手動FAだけ CPU 本格FA に揃える**（案 A×D） | **中**（ファイル数は少なめだが **`precheck` と表示と契約の三点一致**の実装が要る） | **高**（オフの **手動 vs 直後CPU** の違和感を狙い撃ち） | **中**（`conduct_free_agency` は **触らない**前提なら抑えられる） | **高** |
| **手動FA全般（オフ＋インシーズン）を揃える** | **大** | 高 | **高**（`run_cpu_fa_market_cycle`・人事GUI・テスト広い） | **第2弾向け** |
| **表示文言だけ先に直す** | **最小** | 低〜中 | **極小** | **本格調整の前座**には可 |
| **数値変更はまだしない（方針のみ）** | — | — | — | **本メモの位置づけ** |

## 4. 推奨する最小方針

**結論（1本）**: **第1調整は「オフ手動FA（`offseason_full_fa_tk` 経路）」に限定し、年俸（およびそれに紐づく年数）を `_calculate_offer` / `_determine_contract_years` の系に揃える。** **インシーズンFA（人事GUI・`run_cpu_fa_market_cycle`）は当面 `estimate_fa_market_value` / `estimate_fa_contract_years` のまま** とする（**案 D**）。

| 項目 | 内容 |
|------|------|
| **今回の調整対象** | **オフ・本格FA直前の手動1人**における **一覧表示・事前チェック・最終契約**が **同じオファー額**になること（**表示＝契約**を維持）。 |
| **今回やらない** | **`conduct_free_agency` ループ本体の変更**、**オフCPUを estimate に寄せる**（案B）、**estimate の係数だけの調整でごまかすこと**（根本の **二系統**は残る）、**インシーズンまでの同時統一**。 |
| **なぜこの順か** | 違和感の主因が **「オフ・手動とオフ・CPU本格のスケール差」**であることが監査上の **主候補**であるため。**CPU市場の中核式をいじらず**、**手動側だけ**を寄せるほうが **回帰を局所化**しやすい。 |

## 5. 実装時の注意点（別タスク向けメモ）

| 注意 | 内容 |
|------|------|
| **`sign_free_agent` をどう扱うか** | **単一の契約反映の正**は保ちたい。**案**: オフ手動専用に **明示年俸（＋年数）を渡せる**オーバーロード、または **薄いラッパ**で `player.salary` 設定前に `_calculate_offer` 結果を代入してから既存処理を流用（**重複実装は避ける**）。 |
| **`_calculate_offer` をそのまま使えるか** | **チーム・選手の状態**はオフ手動タイミングで **CPUがオファーする時と同型**とみなせるか **要コード確認**（**未確認**: 例えば FA プール選手の `salary` が 0 に近いと `_calculate_offer` は `ovr*10_000` フォールバック等。**監査メモと整合**）。 |
| **年数を揃えるか** | 年俸を `_calculate_offer` に寄せるなら、**見た目・契約の一貫**のため **年数は `_determine_contract_years(player, team, offer)`** を **同じ offer** で使うのが自然（**estimate 年数との混在はオフ手動では避ける**）。 |
| **オフ手動とインシーズンを分けるか** | **第1弾は分ける**（§4）。インシーズンを寄せる場合は **別タスク**で回帰範囲を再見積り。 |
| **CPU市場を壊さない** | **`conduct_free_agency` 内の `_calculate_offer` 呼び出しを変えない**（手動側が **同関数を読む**だけ）。 |
| **GUI表示が契約とズレない** | **一覧・確認・`precheck` が参照する年俸**を **契約に使う値と同一ソース**にする（監査で指摘の **「表示専用はない」**方針を維持）。 |
| **依存・循環** | `free_agent_market` と `free_agency` の **import 循環**が出る場合は **共通サブモジュールへの切り出し**等を検討（**実装タスクで判断**）。 |
| **`precheck_user_fa_sign`** | 現状は **estimate ベース**。**オフ手動**では **揃えた後の年俸**で **所持金・サラリー余地**を再判定する必要（**実装で整合**）。 |

## 6. 次に切り出せる実装タスク候補（2件以内）

### タスク 1: オフ手動FAのみ年俸・年数を `_calculate_offer` / `_determine_contract_years` に統一

| 項目 | 内容 |
|------|------|
| **目的** | オフ手動1人の **表示・可否・契約**を **オフCPU本格FAと同じオファー模型**にし、**表示＝契約**を維持する。 |
| **触る範囲** | `offseason_full_fa_tk.py`、`free_agent_market.py`（`sign_free_agent` 拡張またはラッパ）、必要なら `free_agency.py` の **公開用薄ラッパ**（循環回避）。`precheck` の **オフ専用分岐**または別関数。 |
| **触らない範囲** | **`conduct_free_agency` のアルゴリズム・ループ**、**インシーズンFA GUI／`run_cpu_fa_market_cycle` の estimate 経路**（第1弾）、**estimate の係数いじりのみの対症療法**（方針外）。 |
| **完了条件** | オフ手動で **GUIの年俸＝契約後 salary**、かつ **同一選手・同一チーム状態**で **CPUログのオファー計算と同型の式**になっていることを **テストまたは手動**で確認、`--smoke` 通過。 |

### タスク 2（後続）: インシーズン手動／CPU を `_calculate_offer` 系に寄せるかの再評価

| 項目 | 内容 |
|------|------|
| **目的** | オフ整合後のプレイ感を見て、**レギュラー中も一系統**にするか判断する。 |
| **触る範囲** | 人事インシーズンFAウィザード、`run_cpu_fa_market_cycle`、`precheck` 全般（**要設計**）。 |
| **触らない範囲** | 第1弾完了前に **同時着手しない**。 |
| **完了条件** | 方針が本書 §4 と矛盾しないこと、回帰テスト整備。 |
