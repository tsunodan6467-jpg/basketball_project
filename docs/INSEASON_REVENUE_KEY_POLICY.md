## 0. この文書の使い方

- **目的**: 「シーズン中のこの加算」を **`inseason_*` のどのキーで呼ぶか**、**いつ正本に載せるか**を一文書で固定し、実装・他 doc の表記ぶれを防ぐ。
- **本書と実装**: キー名・B/A の**合意**は本書が正。`breakdown_revenue` や `finance_history` への載せ替えは別タスク。
- **役割分担**: 収益モデルの**因果の物語**は `SEASON_REVENUE_MODEL_NOTES.md`。**正本外の総分類**は `ECONOMY_NON_LEDGER_MONEY_POLICY.md`。本書は**キー命名と B/A ロードマップ**に特化する。

## 2. キー名の候補

用途は **「ラウンドあたりの、リーグ源のキャッシュイン（分配・放映相当・中央営業按分を束ねた塊）」** を指すキー。**単位は円／ラウンド**（実装で週に寄せる場合は doc とコードの両方を更新する別タスク）。

| 候補キー | 長所 | 短所 | プレイヤー説明との整合 | 将来の拡張しやすさ |
|----------|------|------|------------------------|---------------------|
| **`inseason_league_distribution`** | 「リーグからの分配」が主で、CLI の「リーグ分配」と直結しやすい。 | 「放映」「中央営業」が名前に出ず、**束ねの中身**は doc で補足が必要。 | **良い**（括弧で「放映等」と説明済みの塊と一致させやすい）。 | 第 2 項を `inseason_matchday_estimate_round` 等で**並列追加**しやすい。 |
| **`inseason_media_distribution`** | メディア・放映権のイメージが明るい。 | リーグ運営分配・非放映の中央営業まで含む**現状の塊**には**狭すぎる**誤解を招きやすい。 | やや**狭い**（「等」で吸収している範囲とズレうる）。 | 将来、放映だけ切り出すなら使いどころはある。 |
| **`inseason_central_revenue`** | 「中央からの収入」でリーグ・本部イメージを含められる。 | **抽象的**で、プレイヤー用語（分配・放映）と距離がある。 | **中**（説明文を別途必須）。 | 名前が広すぎて、主場項を足すと**何でも入る箱**になりうる。 |
| **`inseason_round_revenue`** | ラウンド粒度が名前から読める。 | **中身が不明**（チケットなのか分配なのか判別不能）。 | **弱い** | 接頭辞だけ共通にして下位キーを増やす手はあるが、**第 1 項の主キーとしては曖昧**。 |
| **`inseason_league_media_central_round`**（例） | 意味を列挙で固定できる。 | **長い**、キー変更コストが高い。 | **良い** | 細分化時に**名前が破綻**しやすい。 |

**補足候補（第 2 項・将来用・今回はコード未使用）**: `inseason_matchday_estimate_round` — 主場レギュラー試合数に比例した門前・協賛の概算用（`SEASON_REVENUE_MODEL_NOTES.md` §3 の「従」）。

## 3. 推奨するキー構成

**結論（1 本）**

| 項目 | 推奨 |
|------|------|
| **v1 で正とする内訳キー（1 キー）** | **`inseason_league_distribution_round`** |
| **意味（固定文）** | **そのラウンドに入る「リーグ・リーグ運営に由来する収入の按分」**。**放映権相当・中央営業（リーグ単位で配られる部分）を含み**、現行コードの `INSEASON_LEAGUE_DISTRIBUTION_ROUND_YEN_BY_LEVEL` 1 本に対応する。 |
| **1 キーか 2 キーか（当面）** | **当面は 1 キー**で足りる。金額はコード側 1 式のまま。 |
| **将来 2 層に分ける前提** | 主場概算を足すときは **第 2 キー** `inseason_matchday_estimate_round` を**追加**し、**第 1 キーは分割しない**（分配塊と門前塊の二重にならないよう）。放映だけ切り出す必要が出たら **その時点で** `inseason_league_distribution_round` を分割する案を再評価（**未確定**）。 |
| **運用名と拡張名** | **JSON・内訳 dict では snake_case の上記キー**。**プレイヤー向け表示**は既存 CLI 文「シーズン中収益（リーグ分配・放映等）」を正とし、詳細画面では「内訳キー → 短い日本語ラベル」の対応表を別タスクで足す。 |

## 4. 当面 B／将来 A の正本化方針

| 論点 | 方針 |
|------|------|
| **いま B 寄りとする理由** | ラウンド**毎**に `record` を増やすと、`finance_history` が肥大し、**年次物語とノイズのバランス**が崩れやすい（`GM_MANAGEMENT_MENU_SPEC_V1.md` §0.3）。いまは **CLI 1 行で説明**できており、**正本未連携でも矛盾する数値は出していない**。 |
| **将来 A に寄せる条件（候補）** | (1) **週次（またはラウンド）レジャー**が用意され、**小計の和＝締め正本**の二段整合が取れる。(2) **オフの `breakdown_revenue`** と **同じ収入を二重に語らない**合意が `ECONOMY_DESIGN_NOTES.md` と整合。(3) **プレイヤーが一覧で追える UI またはログ**がある。 |
| **正本に載せる前に必要なもの** | **表示**: レポートまたは専用一覧で `inseason_league_distribution_round` の累積・ラウンド内訳が見えること。**内訳**: `breakdown_revenue` に載せるなら **キー名は本書 §3 に従う**。**二重説明防止**: オフの `provisional_central` 等と**同じストーリーの収入を二重計上しない**設計を監査メモで確認。**オフ締めとの整合**: シーズン中に「分配」として既に現金化した物語を、締めの収入項で**別名目で再度全額載せない**（差額・補正のルールを別タスクで書く）。 |

**断定しないこと**: 正本化は **締め 1 回に合算で載せる**か、**週次エントリを永続**するかは、週次会計タスク着手時に決める。
