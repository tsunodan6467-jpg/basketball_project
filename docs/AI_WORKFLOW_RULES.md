# Cursor / AI 作業フロー（正本）

**位置づけ**: `.cursorrules` から参照される。**実装・調査・報告の手順**をここに集約する。  
**言語**: チャット返答・ゲーム内表示は日本語（コード・識別子は英語可）。

---

## 最重要運用ルール（1〜10）

### 1. いきなり実装しないこと

- 中規模以上の変更は、**原則として即実装に入らず、先に調査と計画**を出す。
- 実装前に整理する項目:
  1. 原因候補  
  2. 触るべきファイル  
  3. 触ってはいけないファイル  
  4. 想定副作用  
  5. 確認方法  
  6. 実装方針  
- **例外（即実装可）**: 明らかなタイポ、import 修正、**1 ファイル内の安全な小修正**。

### 2. 修正は最小変更・高保全を徹底すること

- **ついで修正禁止**。依頼と無関係な箇所に広げない。
- 既存のメソッド名・呼び出し契約・セーブ契約・UI 導線を、必要性なく変えない。
- 「きれいに整理したくなる」場面では **安全・確実・回帰防止**を優先。
- 大規模整理が必要なときは、実装前に **「今回は応急修正」「後で整理」** の二段構えを提案する。

### 3. 実装前に必ず完了条件を言語化すること

- 「どうなったら完了か」を**観察可能な形**で書く（曖昧なまま着手しない）。
- 例: ウィンドウを開いた瞬間に現在値が見える／`team is None` でも refresh で落ちない／土日 2 試合は同一相手かつ同一 H/A、など。

### 4. 実装後は必ず自己レビューすること

1. 依頼箇所が直っているか  
2. 近接機能が壊れていないか  
3. import / 初期化 / None 系で落ちないか  
4. イベント順・表示順・保存値に副作用がないか  
5. ログ・テスト・起動確認が通るか  

「直した箇所」だけでなく **巻き込まれやすい箇所**も見る。

### 5. UI 修正時の追加確認

- 初回表示で空欄にならないか、長文で崩れないか、スクロール可否、ボタン押下可否、仮文言の露出、例外で描画が止まらないか。
- 可能なら **変更画面に加え隣接メニューを 1 つ以上**確認する。

### 6. ロジック修正時の追加確認

- 今回のケースだけでなく **同系列の別ケース**も確認する（例: 日程なら 2 試合週 / 3 試合週 / H/A / 大会週 / 並び順）。

### 7. 調査・実装・レビューを頭の中で分離すること

- 調査 → 実装 → レビュー → 追加修正。違和感や危険を見つけたら **追加の最小修正**まで責任を持つ。

### 8. 1 発で終わらない前提で、再修正しやすい報告をすること

報告では必ず分ける:

1. 今回直したこと  
2. 今回あえて触っていないこと  
3. 残るリスク  
4. 確認してほしい観点（**手動確認は 3 個以内**に絞って示す）  

「全部直りました」と雑に言わない。

### 9. Cursor 内レビューを前提に行動すること

- 実装後に差分・テスト・影響範囲を見直し、**セルフレビュー済み**で報告する。
- レビュー指摘は「なぜ起きたか」を短く整理し、再発防止の観点を添える。

### 10. 無関係な箇所が壊れた可能性を常に疑うこと

- 共通処理・UI 共通部品・season / match 進行・save/load・team/player 契約を触るときは波及を前提に考える。
- 影響が大きいほど **変更範囲を縮める工夫**を優先する。

## コード変更の進め方

- 無関係な整形や import 順の並べ替えなど、ノイズになる変更は避ける。
- 長いコードはチャットに大量貼り付けず、**ワークスペース上で直接更新**を優先する。
- **変更前**に「どのファイルを触るか」、**変更後**に「どのファイルを変えたか」を明示する。

---

## 実装前の標準手順（中規模以上）

1. **要求整理** — 要望・期待仕様・制約・危険点を短く。  
2. **調査** — 原因候補、関連ファイル、既存仕様書（`docs/`）の正本、過去の修正の巻き戻し防止。  
3. **計画** — 最小変更の方針、完了条件、確認コマンド / テスト方針。  
4. **実装** — 計画に沿って必要最小限。  
5. **自己レビュー** — 依頼箇所・副作用・近接箇所。  
6. **報告** — 変更ファイル、実行コマンド、確認コマンド、確認観点、残リスク。

---

## ログ・確認ルール

- ログを求めるときは **抽出コマンドを併記**する。ターミナルに収まらない場合を考慮する。
- 実行コマンド / テスト / ログ確認は**セット**で示す。コマンドは可能な限り **コピペ 1 行**。
- 長いログは output ファイル出力の例も併記。「このキーワードで検索」等の探し方を短く添える。

---

## 手動操作の指示ルール（GitHub / Steam / Steamworks 等）

- クリック順を **1→2→3**（分岐は最大 2 択）。  
- 開くページは **URL を 1 つ**。  
- ボタン・タブ名は画面表示の文言どおり。  
- 迷いやすい操作は「探す場所」まで書く。  
- コマンドは **コピペ 1 行**。必要なら「フォルダ空白で Shift+右クリック → ターミナルをここで開く」を先に案内。

## チャット返答の締め（`.cursorrules` と同一）

会話の**最後**に、次の 2 つを**見出しまたは箇条書きで分けて**書く（混ぜない）。

1. **私に手動でやってほしいこと**  
   - Steamworks・実機・法務など AI が代行できない作業。該当なしなら「**なし**（今回はリポジトリ／説明のみで足りる）」。

2. **次の一手**  
   - ロードマップに沿った **最も推奨する 1 つだけ**（具体タスク名レベル）。

本文では目的・変更・判断理由・影響・コマンド等を書いてよいが、**最後は必ず上記 2 セクションで締める**。

---

## 自律補強と「いきなり実装しない」の関係（整合メモ）

- **小さく安全な修正**（タイポ、単一ファイルの明確なバグ修正など）は即提案・即修正してよい。  
- **中規模以上**（複数モジュール、契約変更の可能性、仕様判断が要るもの）は **§1 の調査・計画を先に**行う。  
- **面白さの根幹・大規模仕様変更**はロードマップに沿いユーザーと相談（`docs/PRODUCT_ROADMAP_AND_VISION.md` 参照）。
