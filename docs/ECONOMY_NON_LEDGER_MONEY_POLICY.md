## 0. この文書の使い方

- **何のためか**: 正本外の `money` 更新を「すべてすぐ `record` に入れる」前提にせず、**種類ごとに**（正本化・別履歴・当面放置）の方針を揃え、次の実装タスクを安全に切るための**合意たたき台**とする。
- **コード変更はしない**: 本書単体ではリポジトリの `.py` を変更しない。実装は別タスク・別コミットとする。
- **役割分担**: 経路の**事実**は監査メモ、**べき論の全体**は設計メモ・メニュー仕様、**正本外の扱いの整理**は本書。矛盾する場合は監査を事実の正とし、方針は本書を更新する。

## 1. 対象となる正本外 money 変動

`ECONOMY_MONEY_FLOW_AUDIT.md` §2 の整理をベースに、プレイヤー向け説明の観点で要約する。**未確認**: grep で拾えない経路（動的 `setattr` 等）は本表に含めない。

| 変動 | 主な発生箇所 | 文脈 | プレイヤーに見せたい変動か |
|------|----------------|------|---------------------------|
| **広報（PR）施策コスト** | `pr_campaign_management._commit_pr_campaign_core` | 施策実行時のコスト支払い | **高**（意思決定と直結） |
| **グッズ開発・フェーズ進行コスト** | `merchandise_management._advance_merchandise_phase_core` | 開発ラインの工程を進めたときの支出 | **中〜高**（経営メニュー利用者） |
| **オフ大会賞金（国内杯等）** | `_apply_offseason_asia_cup_rewards` → 締め `record` 合流（リーグ所属） | 大会成績に応じた賞金 | **高** |
| **洲际杯などの大会報酬** | `_apply_intercontinental_cup_rewards` → 同上 | 同上 | **高** |
| **FINAL BOSS 報酬** | `_apply_final_boss_rewards` → 同上 | イベント結果に応じた加算 | **高** |
| **トレード現金** | `trade_logic.TradeEngine.execute_multi_trade` | トレード条件としての現金の授受 | **高**（編成の大きな選択） |
| **ラウンド仮営業収入** | `season.Season._apply_temporary_round_operating_income` | シーズン中ラウンドごとの仮キャッシュ注入（`TEMP_*`） | **中**（「なぜ増えたか」の説明が弱いと不透明） |
| **セッション初期所持金** | `main.py`（`TEMP_INITIAL_TEAM_MONEY` 等） | 新規・CLI 開始時の `money` 代入 | 開始時点の**前提表示**として重要。以降の変動とは性質が異なる |
| **FA 市場用 money 欠損ガード** | `free_agent_market.ensure_team_fa_market_fields` | 欠損時のデフォルト代入 | **低**（異常データ向け。通常プレイでは表に出したくない） |
| **オフ締めループ先頭の欠損ガード** | `offseason._process_team_finances` | 極端な欠損データの保険 | **低**（同上） |
| **外部仮想チームの初期 money** | `offseason` 内チーム生成 | 杯・ボス用の非リーグチーム | **対象外**（国内リーグクラブの財務説明とは別枠） |

**補足（正本に近いが別物）**: `offseason` の `record_financial_result` **失敗時フォールバック**は、意図としては正本と同じ現金移動だが、内訳検証をバイパスしうる。本メモの「正本外」とは区別し、**信頼性・テスト**の論点として扱う（§5）。

## 2. 扱い方の分類軸

すべてを即時に `record_financial_result` に載せることが正解とは限らない。以下の **3 分類**で考える。

| 区分 | 意味 |
|------|------|
| **A. 将来、正本（`record` / `finance_history`）に寄せたいもの** | 年次収支・レポートの物語と整合させ、**説明責任が高い**収入・支出。週次レジャーを挟む場合は、**締め時に正本へ合流**する設計も含む（`GM_MANAGEMENT_MENU_SPEC_V1.md` §0.3 の二段整合と両立しうる）。 |
| **B. 正本には載せなくても、表示ラベルや別履歴で明示したいもの** | `money` は動くが、**施策ログ・トレード画面・開発履歴**などで十分説明できるもの。正本の行を増やしすぎない代わりに、**どこか一貫した「見える場所」**を持つ。 |
| **C. 当面は非正本のままでよいもの** | 仮要素の**橋**、開発用ガード、リーグ財務のスコープ外。ただし **JSON 出口や残高説明**では列挙漏れがバグになるものは、C でも **メタデータとしてのタグ**は検討（§5）。 |

**使わない前提にしないこと**: A に寄せない判断も正当（実装コスト・頻度・二重計上リスク）。B は「二流」ではなく、**説明の主戦場をずらす**選択。

## 3. 各変動の暫定方針

| 変動項目 | 現状の経路 | プレイヤーへの説明重要度 | 推奨分類 | 理由 | 実装時の注意点 |
|----------|------------|-------------------------|----------|------|----------------|
| 杯賞金・国内杯 | リーグ所属: `offseason_competition_revenue_pending` → `_process_team_finances` の `breakdown_revenue` | 高 | **A（実装済み・所属クラブ）** | 締め1回の `record` に内訳キーで載る | 外部招待は `money` 直接のまま（締め対象外） |
| 洲际杯報酬 | 同上 | 高 | **A（実装済み・所属クラブ）** | 同上 | 同上 |
| FINAL BOSS 報酬 | 同上 | 高 | **A（実装済み・所属クラブ）** | 同上 | クリア/失敗で額は別内訳行（同一キーで金額のみ変化） |
| トレード現金 | `execute_multi_trade` | 高 | **A（寄せたい）** または **B→A** | プレイヤーが金額を意識して承認する。正本とズレると不信感 | トレード成立タイミングで `record` に載せるか、締めに集約するかは要設計。**即時正本化**と**シーズン集約**でプレイヤー体感が変わる |
| ラウンド仮営業収入 | `Season._apply_temporary_round_operating_income` | 中（説明が伴わないと不透明） | **C（現状）／将来 A の置換対象** | 仮要素。`ECONOMY_DESIGN_NOTES` でも本実装で置換予定。**いま無理に `record` に載せるより**、将来の「正式なシーズン中収益カテゴリ」に吸収する方が自然 | 削除や数値変更は本タスク外。置換時は **週次・カテゴリ別収益** 等の正本設計とセットで切る |
| PR 施策コスト | `_commit_pr_campaign_core`、`management` にログ | 高 | **B**（当面）／長期は A に寄せうる | 頻度が高く、すべてを年次正本の行にするとノイズになりうる。既存ログを**説明の正**にしやすい | `money` とログの**金額一致**を保つ。将来、週次レジャー → 締め正本へ載せるなら §0.3 の整合ルールに従う |
| グッズ開発進行コスト | `_advance_merchandise_phase_core` | 中〜高 | **B** | 工程単位の支出は「施策・開発履歴」で追う方が分かりやすい場合がある。オフの merchandise 内訳との関係は監査コメント参照（深掘りは別タスク） | 売上・正本 revenue との接続は仕様書に「未接続」記載あり。コストだけ先に A に上げると収支バランスの見え方が変わるため**要検討** |
| セッション初期所持金 | `main.py` 代入 | 開始時 | **（分類の外）** | 会計の「取引」ではなく初期状態 | 表示・チュートリアルで明示すれば足りる |
| FA / オフの欠損ガード | `ensure_team_fa_market_fields`、締めループ先頭 | 低 | **C** | 救済用。プレイヤー向け説明よりデータ健全性 | ログに出しすぎない。JSON 化時は「補正フラグ」程度でよい可能性 |
| 外部仮想チーム初期 money | `offseason` チーム生成 | 対象外 | **C（スコープ外）** | リーグ経営の正本外 | 国内クラブの出口と混ぜない |

## 4. プレイヤー視点での説明方針

- **財務レポート（前季収支・`finance_history` ベース）に出るべきもの**  
  - 年をまたいで「自分のクラブの収支の物語」として語りたい項目は **A 寄り**（大会賞金、トレード現金の扱いをどうするか含む）。  
  - ここに載らないが `money` だけ動くと、「レポートの損益と手持ちの増減が噛み合わない」体感になりやすい（監査メモ R2 と同趣旨）。

- **行動ログ・施策ログ・トレード UI で十分なもの**  
  - **B**。正本の行数を増やさず、**プレイヤーが操作した画面**で金額と理由が見えることを優先。

- **見えなくてもよい（通常は）もの**  
  - 欠損ガード、外部チームの初期値。ただしデバッグ・サポート用には別経路があるとよい（未確定）。

- **窮屈さ・不透明さになりやすい候補**（優先的に説明設計したい）  
  - **ラウンド仮収入**（規則的だが正本に出ない）。  
  - **トレード現金**（金額が大きく、正本に出ないと違和感が残りやすい）。**大会賞金（所属クラブ）**は正本内訳に載るようになった（外部招待は別枠）。  
  - **PR・グッズ**（操作はするが、財務画面だけ見るプレイヤーには見えにくい）。

## 5. 将来の正本化 / JSON 出口整理との関係

- **列挙漏れが危険なもの**: `money` を変える**すべての経路**。正本に載らなくても、**出口用スキーマ**では「種別・参照ID・金額・シーズン/ラウンド」の列挙が必要になりうる。特に **トレード現金・大会賞金・仮ラウンド収入** は残高説明の差分要因になりやすい。

- **`record_financial_result` に寄せる場合の注意**  
  - **二重計上**（R1 と同型）を避ける。既に `money` を動かしている経路を `record` に載せるときは、**片方を廃止するか、締めで一括相殺するか**を必ず決める。  
  - `record` の契約（`GM_MANAGEMENT_MENU_SPEC_V1.md` §0.3）を**いきなり変えない**。内訳の和＝正本の合計を崩さない。  
  - 週次レジャー導入時は、**週小計の和＝締め正本**の二段整合を先に満たす。

- **正本化しない（B/C）でもやること**  
  - 最低限、**ドキュメントとコードコメント**で「この変動は `finance_history` に出ない」を追えるようにする（監査メモの更新含む）。JSON 化時は **optional フィールド**（例: `non_ledger_adjustments[]`）で種別だけでも載せる選択肢あり（実装は別タスク）。
