# 国内バスケGM開発 GUI 1対1トレード導線方針メモ

**作成日**: 2026-04-06  
**文書の性質**: **小設計メモ**。人事画面から入る **GUI 1対1トレード（選手のみ）**の**入口・スコープ・フロー**だけを固定する。完成 UI 仕様・`trade_logic` 改修の正本ではない。

| GUI 主導線のギャップと優先度 | `docs/GUI_MAIN_FLOW_AUDIT.md` §5・§6 |
| CLI トレードメニュー構造（1対1 / multi） | `docs/TRADE_MENU_ONE_FOR_ONE_ENTRY_POLICY.md` |
| 1対1 は選手のみ・現金は multi 寄せ | `docs/ONE_FOR_ONE_TRADE_CASH_POLICY.md` |

**コード上の事実（2026-04-06 更新）**: `main.propose_trade`（CLI）と人事の **「1対1トレード（選手のみ）」**はいずれも `inseason_roster_moves_unlocked` 後に `get_trade_candidate_teams` / `get_tradeable_players` → `one_for_one_trade_evaluate_and_ai_gate`（評価＋AI承諾）→ 確認 → `execute_one_for_one_trade`（現金・RB なし）。GUI は `MainMenuView._all_teams_for_trade_gui` で `Season.all_teams` を優先し CLI と同じ母集団を使う。

## 0. この文書の使い方

- **何のためか**: `GUI_MAIN_FLOW_AUDIT.md` の**最優先ギャップ**のうち、**最小・最安全**な **1対1 トレード GUI** を、**どこから・どこまで**実装するかを先に揃える。
- **コード変更ではない**: 本書のみ。ウィンドウ実装は別タスク・別コミット。
- **役割分担**: CLI のメニュー番号・文言は `TRADE_MENU_*` / `ONE_FOR_ONE_*`。**GUI の入口とスコープ**は本書。評価式の変更は本書のスコープ外。

## 1. 現在の 1対1トレードの位置づけ

| 論点 | 内容 |
|------|------|
| **CLI** | `run_trade_menu` の **2** から `propose_trade`。**選手のみ**。**multi（3）** は複数人＋現金＋RB（`TRADE_MENU_ONE_FOR_ONE_ENTRY_POLICY.md`）。相手候補は **全ディビジョン**（`main.get_trade_candidate_teams`、2026-04-06 修正済み）。 |
| **GUI** | 人事ウィンドウに **選手のみ 1対1 トレード**ボタンあり（multi・現金・RB は CLI）。延長・FA 送りは従来どおり。 |
| **1対1 を先にする理由** | 評価・実行経路が **既に `TradeSystem` に閉じ**ており、GUI は **選択 UI とロック再利用**で足りる。multi は **ステップ数・FA・現金**が増え、初手のリスクが大きい。 |

## 2. GUI入口の候補

| 案 | メリット | デメリット | 分かりやすさ | 既存構造への乗り | 将来拡張 |
|----|----------|------------|--------------|----------------|----------|
| **A. 人事画面から入る** | ロスター文脈で **編成の一環**として自然。`trade_fa_wrap` 直下に **ボタン1つ**で足せる。左メニュー増やさない。 | 人事を開かないと気づかないプレイヤーもいる（**ホーム誘導は別タスク**可）。 | **高**（「人を動かす」＝人事）。 | **最も小さい差分**（`main_menu_view.py` の子ウィンドウ追加）。 | 同ウィンドウに **multi 入口を並べない**文言で multi は CLI へ誘導し続ければ衝突しにくい。 |
| **B. 左メニューに「トレード」独立** | **発見しやすい**。製品イメージに直結。 | メニュー項目増、**人事との役割**説明が要る。初手の実装範囲が広がりやすい。 | **高** | **中**（`MENU_ITEMS`・コールバック・新 Toplevel）。 | 長期的には本番向きだが、**最小導線の第1コミットには重い**。 |
| **C. クラブ案内・情報から** | ダッシュボード感。 | **編成操作**との距離が遠く、「なぜここ？」となりやすい。情報画面は**閲覧**が主で責務が混ざる。 | **低〜中** | **低**（責務のねじれ）。 | multi/FA と並べると**雑多**になりやすい。 |

## 3. 推奨する最小導線

**結論（1 本）**: **案 A — 人事ウィンドウから入る。**

| 項目 | 内容 |
|------|------|
| **入口** | `open_roster_window` 内の **トレード・FA ブロック**に、**「1対1トレード（選手のみ）」**ボタン（ラベルは実装で短縮可）。 |
| **今回のスコープ** | **選手 1 対 1**の選択〜評価表示〜最終確認〜`execute_one_for_one_trade` まで。**現金・RB・FA・新規契約は含めない。** |
| **今回やらない** | multi の GUI 化、1対1 への現金入力（`ONE_FOR_ONE_TRADE_CASH_POLICY.md` どおり当面 multi 寄せ）、左メニュー新設、評価閾値・AI ロジックの変更。 |
| **CLI との関係** | **並存**。GUI は **同等のロジック**を呼ぶ（下記 §5）。multi は **引き続き CLI 正本**と人事内の**短文で明示**。 |

## 4. 最小 UI フロー案（ワイヤーなし）

前提: `self.team`（ユーザーチーム）、`self.season`、`all_teams`（`MainMenuView` が保持するリストまたは取得関数）が取れること。**未確認**: 主画面が常に `all_teams` を持つかは実装時に要確認。

1. **ロック** — `inseason_roster_moves_unlocked(season)` が False なら **messagebox** で既存メッセージ相当を出し、**終了**（`propose_trade` と同条件）。
2. **相手クラブ** — `get_trade_candidate_teams(all_teams, user_team)` を **リストボックス or コンボ**で選択（**全 D**、自チーム除外は関数側）。
3. **相手選手** — `get_tradeable_players(ai_team)` を一覧し **1 人選択**。
4. **自チーム選手** — 同様に **1 人選択**（ロスター tree 再利用は任意・最小は別リストでよい）。
5. **評価** — `evaluate_one_for_one_trade` / `should_ai_accept_trade` の結果を **テキストまたは messagebox** で表示（CLI の `print_trade_evaluation_summary` 相当）。
6. **拒否** — AI 拒否なら **終了**（CLI 同様）。
7. **成立確認** — **yes/no**（`messagebox.askyesno` で可）。
8. **実行** — `execute_one_for_one_trade`。**成功時**は messagebox＋`refresh` / `_refresh_roster_window`。

**明記**: 本フローに **現金・RB の入力欄は置かない**（誤解で multi と混同しないため）。

## 5. 実装時の注意点

| 注意 | 内容 |
|------|------|
| **CLI ロジックの流用** | **推奨**: `main.propose_trade` と **重複を避ける**ため、`propose_trade` から **「選択済みプレイヤーで評価〜実行」**部分を **関数化**し、CLI と GUI の両方から呼ぶ（**未確定**: 関数名・配置は実装タスクで決める）。**最小悪**: GUI から `TradeSystem` を直接呼び条件だけ合わせるが、**二重メンテ**に注意。 |
| **`trade_logic` の変更** | 原則 **不要**。GUI は **既存公開 API** のみ。 |
| **CLI と役割重複** | **意図的に並存**。文言で **「multi・現金は CLI」**を維持。 |
| **multi との誤解防止** | ウィンドウタイトル／先頭ラベルに **「選手のみ 1対1」**を入れる。 |
| **構造化ログ** | `execute_one_for_one_trade` 既存の `history_transactions` をそのまま利用（`TRADE_CASH_ACCOUNTING_POLICY.md` の対象は主に multi 現金）。**追加仕様は別タスク**。 |
| **線引き** | 初回マージでは **Tree 装飾・ドラッグドロップ・下書き保存**はやらない。 |

| 項目 | 内容 |

| 項目 | 内容 |
|------|------|

| **触る範囲** | `main.py` のリファクタ（小）、`tests/`。 |
| **触らない範囲** | GUI 見た目の拡張。 |
