# FA：観測行列の差し替え／拡張 — 設計メモ（母集団・再現・成功指標）

**作成日**: 2026-04-08  
**文書の性質**: **決裁前の整理（コード変更なし）**。上位決裁: `docs/FA_PAYROLL_BUDGET_CLIP_THIRD_TRIAL_OR_MATRIX_DECISION_2026-04.md` §5–§7。乖離観測: `docs/FA_PAYROLL_BUDGET_CLIP_GAP_PLAYCHECK_2026-04.md`。実装参照（本メモでは変更しない）: `tools/fa_offer_real_distribution_observer.py`、`tools/fa_offer_diagnostic_observer.py`、`basketball_sim/systems/free_agency.py`。

## 1. 文書の目的

現行の **real distribution 観測行列**では **`offer ≫ room`** が支配的で、**λ や線形クリップ式の良し悪しを分離しにくい**。本メモは、**どの母集団・どの再現手順で行列を組み替えれば「clip 式の比較装置」として機能しうるか**を、**母集団案・再現手順案・成功指標**まで固定し、**次コミット以降の observer 改修の入力**とする。

---

## 2. 現行観測行列の限界

- **構成**: **上位 FA（top N、例: fa-cap=40）× 全チーム**の直積に近い行列（gap メモでは **1920 ペア**）。**「top 側 FA が中心」**である。  
- **`room_to_budget`**: 既定条件では **30M に固定**されやすく、**チーム間・局面間の room 幅が観測に乗りにくい**。  
- **`offer` と `room`**: クリップ直前では **全面的に `offer > room`**（例: **1920/1920**）。ギャップは **数十 M オーダーで一様に正**のため、**λ を変えても `final > buffer` が続きやすい**構造と整合し、**λ 感度が「全面化の有無」では見えにくい**。  
- **帰結**: 現行列は **「高額帯で room を大きく超える人口」に偏ったスクリーン**であり、**clip 式や λ の比較装置としては偏っている**（合成 diagnostic にある **`offer ≤ room` 帯**が本行列にほぼ載っていない）。

---

## 3. 次に観測したい母集団案

**狙い**: **`offer ≲ room` 帯**（未満・近接・わずかに超過）を行列に**意図的に混ぜ**、**`final` が buffer 周辺や中間域に散る**状態を作る。**何を混ぜるとその帯が増えるか**を候補ごとに比較する。

| 案 | 内容 | 長所 | 短所 |
|----|------|------|------|
| **A. 中額 FA 中心** | top 帯だけでなく、**市場年俸の中位〜中額帯**の FA を行列に含める（ランク帯・閾値、または cap 外のサンプル）。 | **`offer` が `room` に近づきやすい**。clip・λ の効きが **0＜final＜buffer** 等で見えやすい。 | 「中額」の定義（閾値・シード依存）の合意が要る。 |
| **B. room が広いチームを意図的に含める** | **payroll に空きが大きい**／**`payroll_budget` が大きい**／ロスターが薄い局面のチームを必ず含める。 | **同じ FA でも `room` が伸び**、`offer ≲ room` が出やすい。固定 30M 偏重を崩せる。 | チーム選定ルール（全チーム維持 vs サブセット）を決める必要がある。 |
| **C. 別 save／シーズン後状態** | **quicksave 以外**、**シーズン進行後**の save で同一スクリプトを実行。 | ロスター・予算の**実プレイ分布**に近づく。**room のばらつき**が出やすい。 | 再現性は **save ファイルの版管理**に依存。自動化は別段。 |
| **D. top40 以外・複数帯サンプル** | **fa-cap を下げる**、または **複数ランジ帯**（例: 11–20 位、21–40 位）を**別実行で結合**して解釈。 | top 独占を避け、**母集団の幅**が出る。 | 実行回数・集計の合算ルールが要る。 |
| **E. D1/D2/D3 混合ルール**（任意） | リーグ段階ごとに **予算・市場規模が異なる**なら、**帯ごとに行列を分けて**比較。 | バランス調整の**代表性**が上がる。 | スコープ拡大。**第一歩では必須にしない**（必要時に拡張）。 |

**比較の軸**: **`offer/room` の分布が 2〜3 未満のゾーンに質量が乗るか**、**`pre_clip ≤ room` が 0 でなくなるか**、**`final` が buffer 一点に張り付かないか**。

## 4. 再現手順の案

- **既存ツールの位置づけ**: 現状の **行列本体**は `python tools/fa_offer_real_distribution_observer.py`（オプションで `--save` 等）。診断対比は `python tools/fa_offer_diagnostic_observer.py`。**本メモ承認後**も、この **呼び出し口は維持**しつつ **母集団だけを差し替える**のが運用しやすい。  
- **実装方針（案・優先度付き）**  
  1. **第一候補**: **同一 observer に CLI 引数で「母集団モード」**（例: FA ランク帯、room 下限／上位チームのみ、fa-cap 上書き）を追加。**既定値は現行行列のまま**にし、**後方互換**を守る。  
  2. **代替**: 母集団ロジックが肥大する場合のみ **薄い別スクリプト**（ラッパ）を1本足し、内部で既存 `_run_matrix` 相当を呼ぶ。**中身の複製は避ける**。  
- **安全・最小差分**: **デフォルトパスは一切変えない**。**新フラグ指定時だけ**フィルタ適用。**スモーク** `python -m basketball_sim --smoke` は従来どおり通す（observer は本体シミュと分離のため）。

## 5. 成功指標

以下を **「行列が clip／λ 評価に使える」ための必要条件群**として整理する（数値閾値は **実装後の第1観測でキャリブレーション**）。

- **`final == buffer`（または表示上の上限）だけでセルが埋まらない** — 一点張り付き回避。  
- **`0 < final < buffer` が一定割合で現れる** — **内側クリップ**が行列上で観測できる。  
- **`final > buffer` だけに全面化しない** — 現行行列の病理からの脱却のシグナル。  
- **`offer / room`（クリップ前）の分布が広がる** — 中央値だけでなく **p10/p90** や **`pre ≤ room` 件数**で見る。  
- **λ 比較が意味を持つ** — 同一行列で **λ を変えたとき**、`final` 分布や **`> buffer` 比率**が **実務上解釈可能な差**を持つ（全面同一ではない）。

## 6. 推奨する第一候補

**第一候補: 「中額 FA 帯を混ぜたうえで、`room` が広いチームを必ず含める」観測行列**（上表の **A + B** を同一次で組み合わせる）。

- **理由（なぜ最初にこれか）**  
  - gap 観測で問題になったのは **「top×全チーム＋ room 固定幅＋ offer 一様に超大」** の三つ組。**A** は **`offer` のレベルを下げ**、**B** は **`room` の上限を上げ**、両方で **`offer ≲ room` 帯を増やす**方向が直交しにくく、**投資対効果が最大**。  
  - 合成 diagnostic に既に **`offer ≤ room` の例**があり、**本番に近い評価**にはその帯が行列に必要 — **A+B が最短でそこに寄せられる**。  
  - **別 save（C）単独**は再現性・説明コストが高く、**第一歩の「装置」としては補助**に回すのが安全。  
  - **D** は **fa-cap や帯の切り口の合意**が要り、**A/B と併用**しやすいが、**単独より A+B を先**にした方が **`offer ≫ room` 脱飽和**に直結する。  
  - **E** はスコープが広い — **第一候補からは外し**、行列が安定してから検討。

- **他案を第一にしない理由（要約）**  
  - **C 単独**: 装置の主因（top・room 幅）を**確実に崩す保証が弱い**。  
  - **D 単独**: top 以外に逃がせるが、**room 固定**が残ると飽和が続く可能性。  
  - **別 observer 新設（実装論）**: 運用は分かれるが、**第一歩の比較装置**としては **既存にフラグ追加**の方が **差分と検証コストが小さい**（§4）。
