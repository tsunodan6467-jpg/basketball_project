# 経営メニュー仕様 第1稿（たたき台）

**状態**: ドラフト v1（確定仕様ではない）。実装は段階的に行い、セーブ互換と `Team` 既存フィールドを最優先する。  
**関連**: `basketball_sim/models/team.py`（財務・施設・オーナー）、`systems/main_menu_view.py`（経営ウィンドウ）、`main.py`（施設投資 CLI）、`offseason`（オーナーミッション評価）。

---

## 0. 先に固定した論点（ユーザー確認済み）

### 0.1 会計の時間軸

- **基本単位は「週」**とする（シーズン進行の週次モデルと整合。`docs/SEASON_SCHEDULE_MODEL.md` 等の週概念に寄せる）。
- 画面ラベルは「今月」より **「今週」／「直近の週」** を優先。シーズン締め・年次集計は既存の「前季」フィールド（`revenue_last_season` 等）と併記してよい。

### 0.2 操作主体と CPU の経営

- **プレイヤーが操作できるのはユーザーチームのみ**でよい（GUI・明示的な経営アクション）。
- **CPU チームも裏で経営シミュを回す**想定とする（施設・簡易スポンサー／人気の更新など）。強豪／弱旅の **初期パラメータ差**に加え、CPU 同士で **ばらつき（確率・閾値・投資頻度）** を付け、リーグ全体の経済が死なない程度に動かす。
- CPU 用は **軽量ルール**（フル GUI と同じ細粒度にしない）でよい。データは `Team` 上の既存スカラー／将来の `management` ネストでユーザーチームと同型に載せ、**読み取り専用の一貫性**を優先。

### 0.3 収支「内訳」の正本（推奨・Cursor 案）

**方針**: 二重計上とセーブ不整合を避けるため、段階的に明確化する。

1. **第1正本（不変）**  
   - クラブの **現金・シーズン締めの収支合計** は、引き続き **`Team.record_financial_result` が更新する `revenue` / `expense` / `money`（および `revenue_last_season` 等）** を正とする。  
   - 既存セーブ・オフシーズン・CLI との互換の根幹。

2. **内訳（表示・分析用）**  
   - チケット／グッズ／スポンサー等の **内訳は「内訳スナップショット」**として、**合計が上記正本の revenue/expense と一致する**形でだけ永続化する。  
   - 推奨フィールド例: `finance_history` の各要素に任意キー `breakdown_revenue: dict[str, int]` / `breakdown_expense: dict[str, int]` を追加（旧セーブはキーなしでよい）。  
   - **組み立てルール**: 内訳を生成するコードは **必ず合計を revenue/expense に合わせる**（端数はルール固定で1カテゴリに吸収）。検証用に `assert sum(breakdown.values()) == revenue` を開発時またはテストで使う。

3. **週次会計を導入した後**  
   - **週次エントリ**（例: `management.weekly_ledger: list[dict]`）を持ち、各週の収支小計を積み上げ、**シーズン締めで既存の `record_financial_result` に同じ合計を渡す**。  
   - 週の内訳の和＝週の小計、週小計の和＝シーズン正本、の **二段の整合**を保つ。

4. **UI 表示**  
   - 内訳が未生成の期間は「内訳は準備中」または単純な推定表示に留め、**正本の数値と矛盾する文言を出さない**。

メイン左の **「経営」は1項目のまま**とし、開いたウィンドウ内で **タブまたは左サブナビ**に以下を配置する（ウィンドウ乱立と既存4パネル実装の競合を避ける）。

| # | 大分類 | 既存コードとの対応 |
|---|--------|-------------------|
| 1 | 財務レポート | `money`・前季収支・`get_finance_report_text`・`finance_history` |

## 3. データ構造・責務（保全）

- **新規のばら撒きフィールド**より **`Team.management: dict`（`version` キー）** で広報・スポンサー・グッズ・週次メタをまとめ、欠損は `_ensure_history_fields` または `normalize_payload` 側で `{}` 補完（`team_tactics` と同型の考え方）。
- **GUI** は `Team` のメソッドと、**`main` から抽出した純関数化した経営コマンド**のみ呼ぶ（CLI と二重実装しない）。
- **シーズン締め・オーナー評価**は `Season` / `offseason` に残す。
- **`owner_missions` のキー契約**を変えない（評価ロジックが依存）。

---

## 4. CPU 経営（裏処理）のたたき台

- **同じ `Team` フィールド**を更新する（表示はユーザーのみ）。  
- **頻度**: 週次またはラウンド終了時の軽量ステップでよい。  
- **内容例**: 低確率で施設1段（上限遵守）、`sponsor_power` 微調整、人気のドリフト。  
- **強弱**: `league_level`・前年順位・`owner_expectation` 相当の簡易ティアで **投資確率・金額上限**を変える。  
- **パフォーマンス**: 全48チームフルシミュは重くならないよう、CPU は簡略式またはサンプリングを検討。

---

## 5. 実装優先順位（推奨）

1. 施設投資の GUI 接続（既存 CLI と同一ロジック）  
2. 財務レポート拡張（履歴・§0.3 内訳の土台）  
3. オーナーミッション表示の充実  
4. スポンサー（保存 → `sponsor_power`／内訳との整合）  
5. 広報（週次ガード＋弱効果）  
6. グッズ  
7. CPU 裏経営（ユーザーチームの週次が安定してから）

## 6. リスク・注意

- 内訳を先に出して **合計と `revenue` がズレる**とプレイヤー不信。**§0.3 の検証**を必須にする。  
- 「月次」表記と週次実装の混在に注意（UI は週優先）。  
- `record_financial_result` の意味を変えない。
