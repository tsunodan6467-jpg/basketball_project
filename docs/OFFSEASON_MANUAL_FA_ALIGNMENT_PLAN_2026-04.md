## 1. 文書の目的

インシーズンFA と FA プールの `player.salary` は、`estimate_fa_market_value` を軸に**円オーダーが揃いつつある**。一方、**オフ手動FA**（`offseason_manual_fa_offer_and_years`）は、歴史的経緯から **`_calculate_offer` 主経路＋estimate フォールバック＋1.35×estimate 下限**という**ハイブリッド**のまま残っている。

本書は、このハイブリッドを**新世界観の上でどう理解し、どう段階的に整理するか**の判断基準をまとめる。いま**削除を断定**するのではなく、**各要素の責務を再定義**し、**最初の1手をどこに置くか**を明確にする。

## 2. 現在のオフ手動FAロジックの構造

実装は `basketball_sim/systems/free_agent_market.py` の **`offseason_manual_fa_offer_and_years`** に集約されている（`conduct_free_agency` 本体は読むだけで変更しない方針がコメントされている）。

### 2.1 処理のおおよその順序

1. **`ensure_team_fa_market_fields` / `ensure_fa_market_fields`** で欠損補完。
2. **`room = get_team_fa_signing_limit(team)`** … キャップ・ソフトキャップ等を踏まえた**サイン可能な年俸余地の上限**（インシーズン系と整合した単一入口）。
3. **`core_offer = _calculate_offer(team, player)`** … **主経路**。`free_agency` 側のロジックで、ベースは **`player.salary`**（>0 でない場合は `ovr * 10_000` 等の別フォールバック）。**`payroll_budget` と実ペイロールの関係**で **`room_to_budget` が 0** となり、**`core_offer` が 0 になりうる**のがコメントで明示されている。
4. **estimate フォールバック**: **`core_offer <= 0` かつ `room > 0`** のとき、`est = estimate_fa_market_value(player)` とし、**`core_offer = min(est, room)`**。つまり「本格式が 0 を返したが、キャップ上はまだ枠がある」局面の**救済**。
5. まだ **`core_offer <= 0`** なら **`(0, 1)` で打ち切り**。
6. **`est` が未設定なら**この時点で **`estimate_fa_market_value` を取得**（主経路が正のときも、下限計算に使う）。
7. **`floor_offer = est * MANUAL_OFFSEASON_FA_OFFER_FLOOR_MULTIPLIER`（1.35×）**。
8. **`final_salary = min(max(core_offer, floor_offer), room)`** … **下限で押し上げ**たうえで **`room` でクリップ**。
9. **`_determine_contract_years(player, team, final_salary)`** で年数。

### 2.2 構造の要約

| 要素 | 由来 | 役割 |
|------|------|------|
| **主経路** | `_calculate_offer` | `salary`（＋チーム財務・キャップ・予算）からオファー芯 |
| **フォールバック** | `estimate` | 主経路が **0** のとき、`room` 内で **表示＝契約**を成立させる |
| **1.35×下限** | `estimate * 1.35` | `core` が正でも **`core` を下限まで引き上げ**（オフ手動専用） |
| **最終クリップ** | `room` | いずれも **`get_team_fa_signing_limit` を超えない** |

## 3. 新しいFAプール年俸世界観との関係

- **以前**: FA プールの `player.salary` が **estimate と桁・単価でズレうる**状態だったため、**主経路の芯**と**目安**が別物になり、**フォールバックが「別スケールへの橋」**としての側面も持っていた。
- **現在（プール正規化後）**: `normalize_free_agents` 経由で **`salary ≈ estimate`** となり、**主経路の入力**と **estimate** の**乖離は大きく縮小**しうる。
- **ただし** `_calculate_offer` が **0 になる主因**は、必ずしも「salary と estimate のズレ」だけではない。**`payroll_budget` が実ペイロールに追いついていない**等の**チーム側条件**が残るため、**フォールバック自体は依然「オファー0の救済」として意味を持ちうる**。
- 一方、**1.35×下限**は、estimate が **数十万円オーダーだった時代**には「極端に安い表示オファー」を防ぐ**安全网**として効いた。estimate が **数千万〜億オーダー**になった今、**同じ 1.35 倍は「常に強い引き上げ」**になり、**`room` に早々に当たる**挙動が増える。**必要性の形が変わった**可能性が高い。

## 4. estimate フォールバックの役割整理

| 案 | 内容 | 利点 | 注意 |
|----|------|------|------|
| **そのまま残す** | `core <= 0 and room > 0` のまま | 実装変更最小、既知の救済維持 | `core` が「小さい正」で不自然な表示になるケースは別問題 |
| **条件を狭める** | 例: **`core == 0` のみ**フォールバック、または **`core` が閾値未満**のみ | 不要な estimate 依存を減らせる | 閾値設計とテストが必要 |
| **役割を変える** | フォールバックを「estimate 単体」ではなく **`max(core, min(est, room))` 型**に再定義 | 主経路を尊重しつつ底を補う | `_calculate_offer` との重複整理が要る |
| **将来消す** | `payroll_budget` 同期や `_calculate_offer` 前処理の改善後に撤去 | 一本化の最終形 | 現時点では**前処理未整備**のリスク |

**推奨（この段階）**: **まずはそのまま残しつつ、発動条件とコメントを「なぜ 0 なのか」に紐づけて明文化**するのが最も安全。次段で **観測**（ログ・テスト）に基づき **条件の狭め方**を検討する。

**理由**: プール `salary` 同期後も **`core_offer == 0` の経路は消えていない**可能性が高く、フォールバックは**依然正当**。いきなり削除すると **オフ手動だけ契約不能**に戻る回帰リスクが大きい。

## 5. 1.35×estimate 下限の役割整理

### 5.1 元々の意図（要約）

- **d995b48 系の文脈**: オフ手動で **`_calculate_offer` が極端に低い／0 に近い**ときでも、**`estimate` を基準にした下限**で「表示＝契約」の**額の品位**を保つ、という **safety net**。
- **旧 estimate スケール**では、1.35× も **絶対額は小さく**、`room` とのバランスが取りやすかった。

### 5.2 プール salary 同期後

- **`core_offer` のベース**は estimate オーダーに近い `salary` になりやすく、**主経路単体でも「それなりの額」**が出やすい。
- **下限は依然「core を estimate より下に落とさない」方向**に効くが、**estimate が大きい**ため **`floor_offer` が `room` に達しやすく**、結果として **`final_salary == room` が頻発**しうる。これは**意図（品位確保）**と**結果（上限張り付き）**のギャップが生まれる可能性がある。

### 5.3 再定義の方向（削除を急がない）

- **まだ必要とするなら**: **「オフ手動において、本格式の結果が市場ベース（estimate 相当）を著しく下回る場合に、プレイヤーに説明できる下限を張る」**という **safety net** として位置づけ直す。
- **強すぎる可能性があるなら**: 第2弾以降で **倍率の見直し**（例: 1.35→1.0〜1.15）、**`min(floor, α * room)` のキャップ**、**`core` が既に `est` 以上なら下限スキップ**等を検討する。

本書では **いますぐ 1.35 を廃止しない**。**役割の再定義と計測**を先に置く。

## 6. `_calculate_offer` 主経路との接続方針

- **主経路の維持**: CPU 本格FA と**同じ `_calculate_offer`**を使うことは、**挙動の対称性**と**説明責任**の面で引き続き妥当。本体をすぐ書き換えなくても、**入力（`player.salary`）がプール同期で揃った**時点で、**オファー芯は既に新世界に寄っている**。
- **フォールバック・下限の位置づけ**: **主経路の補助線**として **subordinate** に置く。すなわち「通常は `core_offer` を信じ、**0 または著しく不自然なときだけ** estimate 系が補う」「下限は **説明可能な最低品位**のための上乗せ」という**階層**を文書・コメントで固定する。
- **`_calculate_offer` 本体を今触らずに済むか**: **第1〜2弾の「整理」フェーズでは触らずに済ます余地が大きい**。オフ手動側の**条件・倍率・クリップ順**の調整で多くの違和感は吸収できる。本体変更は **`payroll_budget` 解釈**や **レガシー `ovr*10000`** 等、**より広いリファクタ**のタイミングに回すのが安全。

## 9. テストと確認観点

- **`_calculate_offer == 0` かつ `room > 0`**: フォールバックが **`min(est, room)`** になり、**`final` が `room` で頭打ち**にならない／なるケースの**意図した動き**。
- **`_calculate_offer` が正の中程度**: **`max(core, 1.35*est)`** が **`room` を超えず**、**表示＝契約**が破綻しないか。
- **`_calculate_offer` が既に `est` 以上**: 下限が**実質ノーOp**か、**無用に `room` 張り付き**を生むか（第2弾の論点）。
- **回帰**: `test_offseason_manual_fa_offer_alignment`、`test_precheck_user_fa_sign`、関連オフFA UI テスト、**スモーク**。
