## 11. シミュ進行単位と試合間介入（ラウンド vs 週内）【旧 §1】

- **現状**: `Season.simulate_next_round` が 1 回の CLI/GUI 操作に対応。`round_config` 上は週あたり複数試合（土日＋水曜等）が同一ラウンドに載ることがある。
- **課題**: GM 視点では「連戦の間に戦術を変えたい」ニーズと、現状の原子的ラウンド進行が噛み合わない。**別系統**として、介入より先に「同一週の試合を 1 試合ずつ見守る」体験を優先する方針もあり得る（**§16**）。
- **次の検討**: 週内サブステップ（例: 試合単位のコールバック）を増やすか、あえてラウンドのまま「進行前プラン」UI を厚くするか。いずれも **年度ループ・セーブ形式への影響が大きい**ため別イテレーション推奨。
- **今回の補助（案A・表示のみ）**: `next_advance_display_hints` / ホーム「次へ進む」周辺 / `next_round_schedule_lines` で、進行1回あたりの自チーム試合数と「まとめシミュ・試合間介入なし」を明示。
- **将来**: 週内サブステップは本 backlog のまま別イテレーション。

## 16. 週次 GM ループと試合逐次「見守り」（将来方針メモ・未実施）

プレイ体験としての**意図の正本**（確定仕様ではない）。コード上の「週」は多くの場合 **1 ラウンド（`simulate_next_round` 1 回）** とほぼ同義で読み替え可能。

### 16.1 言い換え（本質の整理）

- **GM の判断サイクル**: 「週のはじめ（＝次の進行に入る前）」に戦術・起用・人事・経営などをまとめて決める。
- **見せ方のサイクル**: その週に含まれる自チーム試合を、**1 試合ずつ順に見る**（ハイライト／結果のみ／スキップ等）。
- **本質は試合間の細かい介入ではない**: 毎試合のあいだに戦術を差し替えたいことが主目的ではなく、**週単位の意思決定 → その週の流れを試合単位で追体験する → 翌週また判断する**というリズムを気持ちよく回すことが主眼。

短く言うと **「進行・判断の単位は週（ラウンド）、体験の単位は試合」** を両立させたい、という整理。

### 16.2 方針として妥当か（論点1）

- **週次 GM ループ**という語り口は、現状の「1 回の進行＝1 ラウンドまるごと」と矛盾しない。理想像は **GM 画面に戻るタイミングを「週の頭」に固定し、その手前を試合単位の視聴フローで埋める**イメージ。
- 用語メモ: プレイヤー向けは「週」、実装メモでは「ラウンド／`current_round` インクリメント前ブロック」と対応付けて書くと齟齬が減る。

### 16.3 内部一括のまま、表示だけ逐次化できるか（論点2）

**かなり現実的な第一候補**として、次のモデルが成り立つ。

- **1 操作**（例: GM で「次へ進む」）の裏では、従来どおり **`simulate_next_round` を一度実行**し、そのラウンド分の試合結果・ログ・スタッツ更新まで完走する。
- **UI 層**では、その週に関係する試合の **結果セットをキューとして保持**し、プレイヤーには **1 試合ずつ**「ハイライト → 結果画面 → 次へ」で見せる。中身は **すでに確定した試合の再生／要約表示**であり、試合のたびにシミュを再実行しない（B 案・**プレゼンテーション逐次化**）。

成立条件・注意:

- ラウンド内の試合順と、見せる順序を **明示的に定義**する（リーグ・杯・中間イベントの並びは §12・カップ正本と整合）。
- **ニュース・順位表・記録の更新を「いつ見せるか」**: 週末一括で確定しているなら、UI では「週の最後の試合の後に反映」または「バックグラウンドでは更新済みだが表示は試合後にフェードイン」等、**誤解のないルール**が要る。
- **セーブを「試合2の直後」で切った場合**にどうするかで難易度が変わる（**論点5**）。セッション中だけ逐次で、セーブは常に「週の頭」なら **ラウンド内状態はセーブに持たない**設計も可能。

### 16.4 影響範囲の目安（論点3）

実装時に触れそうな層（想定・未確定）:

| 層 | 内容 |
|----|------|
| **UI / フロー** | GM → 試合視聴ループ → GM の状態機械、戻るボタン・「次へ」の意味の分岐 |
| **試合プレゼン** | ハイライト再生、結果のみ、スキップの分岐、1 試合分のデータ供給 |
| **データの受け渡し** | 1 ラウンド実行後に「自チーム関連の試合結果リスト」を並び順付きで取り出せること（既存ログ／`game_results` 等の利用または薄いラッパ） |
| **オーディオ・演出** | スキップ時のテンポ、連続試合時の遷移 |
| **セーブ** | 上記のとおり、**週の途中から再開するか**で要否が決まる |
| **CLI / 自動進行** | 逐次 UI をバイパスする経路の有無 |

`Season.simulate_next_round` の**分割そのもの**は、B 案だけなら **必須ではない**（ただし将来、試合間介入や水曜別ステップで **状態をセーブに載せる**なら §15 の案A 側に寄る）。

### 16.5 第一段階は 3 択で足りるか（論点4）

**まずは十分あり得る。** 初期スコープ案:

- **ハイライト**（既存または簡易版）
- **結果のみ**（ボックススコア／一行サマリー）
- **スキップ**（即結果画面または次試合へ）

試合間の戦術変更は入れず、**週の頭の設定がその週の全試合にそのまま適用される**現行ルールと整合する。介入を足す場合は §15・§11 の「真サブステップ」と設計が接続する。

### 16.6 将来 backlog に残す設計論点（論点5）

- **試合間介入を入れるか**: 体験の本質が「見守り」なら後回し可能。入れるならシミュ分割または「週内インデックス」の保存が絡みやすい（§15）。
- **水曜だけ別ステップにするか**: 表示キュー上は「同じ週の2本目」として自然に並べられる。**別ステップにするのは UX 上の強調か、シミュ順序の変更か**を分けて決める（§15 案C）。
- **セーブにラウンド内状態を持つか**: **持たない**（常に週頭から）なら B 案が最も軽い。**持つ**なら「週内 N 試合目まで消化済み」フラグ／インデックスとマイグレーションが必要。
- **ニュース・メディア・ファン反応のタイミング**: 一括シミュ後に順序だけ遅延表示するか、仕様を固定する。
- **ユーザーがユーザー以外の試合も「週内で」見るか**: スコープを自チームのみに限定するか。

### 16.7 既存節との置き場所

- **§16（本節）**: プロダクトとしての**理想体験・優先順位（見守り vs 介入）**と **B 案（表示逐次）**の記述の正本。
- **§11**: ラウンド vs 週内の**課題の列挙**と、案A寄りの技術論点。§16 と対をなす「体験論点」として参照し合う。
- **§15**: **内部構造・セーブ・状態機械**の大きな変更が必要な線（案A/C/D の本格版）。§16 で「まず B で寄せる」場合の入口は §16、手詰まりで分割が必要になったら §15。

### 17.3 再配置の推奨案（論点整理）

**推奨（段階的統合）**

1. **スタメン・6th・ベンチ（編集）** → **戦術**ウィンドウ内にタブまたはセクション統合（試合に直結する起用の一本化）。
2. **戦術・HC・起用（`apply_team_gm_settings`）** → **戦術**と同一ハブに集約し、GM からは削除または戦術へのショートカットのみ。
3. **サラリーキャップ（閲覧）** → **経営**の財務／キャップ領域へ（既存 `format_salary_cap_text` の表示場所移動）。
4. **ロスター（閲覧テキスト）** → **人事**ウィンドウに「GM ビュー同等のテキスト」または表の補助表示で吸収し、重複閲覧を減らす。
5. **チーム情報（閲覧）** → **情報**の「チーム／クラブ概要」的タブ、または **人事**ヘッダ、または **ホーム**拡張のいずれかに寄せる（複製は最小化）。
6. **CLI 案内・トレード** → 左メニュー「GM」を廃止した後は **人事**（ロスター操作）＋ **システム／ヘルプ** または **経営** への分散が自然。

**代替案**: 左メニュー「GM」を **`クラブ`／`チーム概要`** に改称し中身はダッシュボードのみ（編集ゼロ）に縮小。編集系はすべて戦術・人事・経営へ移す。

**現状維持の合理的理由（短く）**: 開発コスト・ドキュメント・CLI「8.GMメニュー」との語彙同期・プレイヤー既習慣。長期的には情報設計としては中途半端になりやすい。

### 17.4 安全な実装順（将来）

1. **文言のみ**: 各タブ先頭に「将来は◯◯メニューへ移動予定」等は **短く**（ノイズ注意）。またはリリースノートのみ。
2. **複製後に削除**: 戦術側にスタメン編集を**実装・動作確認**してから GM タブを非表示／削除（1 機能ずつ PR）。

### 18.2 最終処遇の候補

| 案 | 内容 | メリット | デメリット |
|----|------|----------|------------|
| **A. 完全削除** | 左メニューから削除。中身は情報・人事・経営・戦術・システムへ再配置 | 情報設計が最もすっきり。「GMの中のGM」解消 | CLI・ドキュメント・習慣との同期コスト。トレード案内の**単一入口**を別途決める必要 |

### 18.3 プレイヤー視点

- **混乱の主因**: 「自分が GM なのに、メニューにも GM がある」＝**役割と UI ラベルの衝突**。
- **改名だけ**でも「ここはゲーム全体ではなく**クラブの窓**」と誤解は減らせるが、**中身が案内の羅列**だと「なぜ独立メニュー？」は残る。
- **最も自然な長期像**: 閲覧・編集は **情報／人事／経営／戦術** に分散し、**ターミナル連携だけ**なら **人事下** または **システム・ヘルプ** の方が筋が通る。
- **初見向け**: 「ロスターを見る＝人事」「数字＝経営」「試合の指図＝戦術」が一本化されている方が学習コストは低い。

### 18.4 安全な実装順（将来・提案）

1. **文言・チュートリアル・`.cursorrules` の整合**（「GMウィンドウ」→「情報・経営・戦術を開く」等）。**コード変更は最小**。
2. **ロスター全文テキスト** — **実施済み（第5段・2026-03）**: 人事ウィンドウへ正本移管、GM「ロスター」は案内のみ。残作業は将来、**GM タブ行ごと削除**など。
3. **スタメン／戦術参照タブ**を削るか、上部ヒントに**一行マージ**（1 PR）。
4. **左メニュー改名（GUI）** — **実施済み（第7段）**: 「クラブ案内」＋閲覧・案内役割の明示。CLI 説明の一括更新は未着手。
5. **メニュー項目ごと削除**— トレード案内を人事へ移し、**最後に左メニュー行を削除**（1 PR）。

一気削除は **非推奨**（案内・CLI・テスト・プレイ動画の一斉更新が必要）。

### 18.5 backlog に残す論点

- **トレード／インシーズン FA**の GUI 正本をどこに置くか（人事専用 vs システム）。
- **CLI「8.GMメニュー」**の名称変更タイミングと**後方互換**（旧ドキュメントリンク）。

- **§13** の「GM 廃止」と **§18** の最終決定を**いつ一本化するか**（プロダクト方針の確定待ち）。

### 18.6 推奨方針（メモ時点・未確定仕様）
