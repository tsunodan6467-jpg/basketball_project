## 0. この文書の使い方

- **何のためか**: 個別の GUI 改善を**寄せ集め**る前に、**主導線の穴と重複**を一覧し、次の **1 タスク 1 目的**の実装に渡す。
- **コード変更ではない**: 本書のみ。ウィンドウ追加・メニュー改修は別コミット。
- **役割分担**: **理想**は `IDEAL_GAME_DESIGN_MASTER.md`、**事実スナップショット**は `CURRENT_STATE_ANALYSIS_MASTER.md`、**順序**は `IMPLEMENTATION_PLAN_MASTER.md`。本書は **GUI 導線のギャップ分析**に限定する。

## 1. GUI主導線の現在地

| 論点 | 内容 |
|------|------|
| **最初に触る場所** | 中央の **「次へ進む」** がシーズン進行のハブ。**未確認**: 新規プレイヤーが「左メニューから始める」かはユーザーテストなし。 |
| **主な分岐** | **日程**（カレンダー・大会表示）、**人事**（ロスター・限定的編集＋CLI 案内）、**経営**（財務・施設・オーナー・スポンサー・広報・グッズの統合ウィンドウ）、**強化**（育成・トレーニング編集）、**戦術**（スタイル・ローテ・プレイブック等のハブ）、**情報**（順位・統計・大会状況等）、**歴史**、**システム**。 |
| **整理が進んでいる箇所** | 経営ウィンドウ内の **財務サマリー／詳細レポート／施設／オーナー／スポンサー／広報／グッズ** のパネル分割（`GM_MANAGEMENT_MENU_SPEC_V1` の並びと概ね対応）。**シーズン中収益**は `inseason_cash_round_log` を財務に表示。**PR・グッズ支出**は財務から **経営パネル履歴**への誘導文を追加済み（`PR_MERCH_MONEY_VISIBILITY_POLICY` タスク1）。**トレード**は CLI で **1対1 / multi の住み分け**を doc 整理済み（GUI 本操作は未）。 |

## 4. 未整理・未実装の高優先 GUI 課題

| # | 課題 | なぜ問題か |
|---|------|------------|
| 1 | **編成の中心操作（トレード・FA・契約）が CLI 正本** | 製品目標の **GUI 主操作**と最もズレが大きい。人事画面は**読めば分かる**が、**遊ぶほど CLI 行き来**が前提になる。 |
| 2 | **「次へ進む」と左メニューの関係が暗黙** | 進行は中央、深い編集は左 — 構造は合理的だが、**初見で「まず何を押すか」**の説明が主画面だけでは弱い可能性（**未確認**）。**緩和（2026-04-07）**: 主画面の進行パネル先頭に**一文案内**を追加（`main_menu_view` の `advance_wrap`）。 |
| 3 | **日程 / 情報 / 歴史の情報の住み分け** | どこで「今週の結果」「シーズン累計」「クラブ史」を見るかが、メニュー名だけではプレイヤーが試行錯誤しうる。**緩和（2026-04-07）**: 各ウィンドウ先頭に**一文案内**（`open_schedule_window` / `open_information_window` / `open_history_window`）。 |
| 4 | **強化 と 戦術 の連続性** | 両方開かないと「育成した選手をどう試合で使うか」が繋がりにくい（**未確認**）。 |
| 5 | **サラリーキャップ・給与の表示箇所の複数性** | 経営・人事・案内文で**一貫して同じ数値**が見えるかは保守次第。誤解の温床になりうる。 |
| 6 | **試合の「見せ方」入口** | ハイライト／結果だけはコード・doc が別。**主画面からの導線**が製品レベルで固まっているかは `PRODUCT`・現状分析で **□ 寄り**（断定は現状分析書に委ねる）。 |

**CLI 前提のまま残っている例（事実）**: `main.py` のシーズンメニュー、**GM メニュー → トレード**（人事案内文で明示）。

## 5. 優先順位づけ（3〜5 件）

| 順位 | 課題 | なぜ高優先か | 先にやると楽になること |
|------|------|--------------|------------------------|
| **1** | **トレード／FA／契約の GUI 導線の最小版** | 理想・PRODUCT の「GUI 主操作」と直結。いま最も **CLI 依存**が目立つ。 | 以降の経済・戦術 UI の説明から「まずターミナルへ」が消える。 |
| **2** | **主画面の「今やること」／進行と左メニューの関係を一言で固定** | 大改修なしで**迷子率**を下げられる（文言・ホームタスクの整理）。 | オフ・レギュラー切替時の混乱を減らす。 |
| **3** | **日程 vs 情報 vs 歴史の「何をどこで見るか」1枚図 or 短いヘルプ** | 重複閲覧・**欠落感**（どこにも無い情報）の両方を防ぐ。 | 新規向け QA を減らす。 |
| **4** | **強化↔戦術のクロスリンク（1 行でも可）** | 実装コストが低く、体験の**連続性**だけ改善しやすい。 | 中核プレイヤーの満足度向上。 |
| **後回し可** | 財務の PR・グッズ**合算行**（タスク2） | 誘導文で **正の場所**は示せている。合算は「便利」層。 | — |

## 6. 次に切り出せる実装タスク候補（3 件以内）

### タスク 1: 人事（または専用ウィンドウ）からの「GUI トレード最小導線」

| 項目 | 内容 |
|------|------|
| **目的** | **CLI なしで** 1 回のトレード意思決定が完結する入口を **1 か所**用意する（範囲は 1対1 限定でも可・別設計）。 |
| **設計の正** | **入口・スコープ・フロー**は `docs/GUI_ONE_FOR_ONE_TRADE_ENTRY_POLICY.md`（人事ウィンドウから・選手のみ）。 |
| **触る範囲** | `main_menu_view.py`、既存 `trade_logic` 呼び出し、必要なら `main.py` の共有ヘルパ。 |
| **触らない範囲** | multi の全面 GUI 化、評価式の本改修、セーブ形式の破壊的変更。 |
| **完了条件** | ドッグフードで **GUI のみ**でトレード 1 回成立、pytest または手動手順が文書化。**実装済み（最小・2026-04-06）**: 人事の 1対1（選手のみ）＋`test_one_for_one_trade_main_helpers.py`・`--smoke`。 |

### タスク 2: ホーム「やること」／進行ラベルと左メニューの関係を doc＋UI 文言で固定

| 項目 | 内容 |
|------|------|
| **目的** | **次へ進む**が「シミュを進める正」であることと、左メニューが「編集・閲覧のハブ」であることを**同一画面内で読める**。 |
| **触る範囲** | `main_menu_view.py` のラベル・`tasks` 生成、必要なら `PRODUCT` の一言。 |
| **触らない範囲** | 左メニュー順序の大型変更、新メニュー追加。 |
| **完了条件** | 新規プレイヤー向けの**1段落説明**がコード上のどこかに単一ソースで存在。**実装済み（最小・2026-04-07）**: `advance_wrap` 先頭の**一文**（進行＝下のボタン、確認＝左メニュー）。 |

### タスク 3: 情報／日程／歴史の各ウィンドウ先頭に「この画面で見るもの」1行

| 項目 | 内容 |
|------|------|
| **目的** | §4 の **住み分け**を開いた瞬間に示す（重複閲覧の削減）。 |
| **触る範囲** | 各 `open_*_window` 先頭の `ttk.Label` 等。 |
| **触らない範囲** | データ取得ロジック、タブ構造の再設計。 |
| **完了条件** | 3 画面すべてに**1 行説明**があり、相互参照文が矛盾しない。**実装済み（最小・2026-04-07）**: 各 `outer` 先頭の `tk.Label` 一文（日程＝カレンダー系、情報＝集計データ、歴史＝通算・記録）。 |
