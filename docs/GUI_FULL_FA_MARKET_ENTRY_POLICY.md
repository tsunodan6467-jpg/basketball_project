# 国内バスケGM開発 GUI 本格FA市場導線方針メモ

**作成日**: 2026-04-06  
**文書の性質**: **小設計メモ**。**オフシーズンの本格FA市場**（`conduct_free_agency` が担う領域）を GUI 化する場合の **入口と第1実装の最小スコープ**だけを固定する。完成 UI 仕様・`free_agency.py` / `Offseason.run` のロジック正本ではない。

|------|------|
| FA / 契約全体（再契約・オフFAの位置づけ） | `docs/GUI_FA_CONTRACT_ENTRY_POLICY.md` |
| **インシーズンFA（レギュラー中・1人）** — 本書と混同しない | `docs/GUI_INSEASON_FA_ENTRY_POLICY.md` |
| GUI 主導線のギャップ | `docs/GUI_MAIN_FLOW_AUDIT.md` |

| 実装順の正本 | `docs/IMPLEMENTATION_PLAN_MASTER.md` |

**コード上の事実（リポジトリ静的確認・2026-04-06、実装追記）**: `Offseason.run` 内で **ドラフト**（`conduct_auction_draft`）→ **`conduct_trades`** → **`conduct_free_agency` 直前**に **`pre_conduct_free_agency_ui_prompt` が設定されている場合のみ** `systems/offseason_full_fa_tk.run_user_offseason_fa_one_pick`（GUI 主画面オフ経路の `main.py` から注入）→ **`conduct_free_agency(self.teams, self.free_agents)`**（フェーズ13付近）。**`conduct_free_agency` 本体**は **全チーム CPU シミュ**のまま。**CLI 直オフ**ではコールバック未注入のため **手動1人UIは出ない**。**オフ再契約**は **`resign_ui_prompt`**。**レギュラー中1人**は人事 **インシーズンFA**（`GUI_INSEASON_FA_ENTRY_POLICY.md`）。

## 0. この文書の使い方

- **何のためか**: 人事 GUI に **トレード／解除／インシーズンFA1人／オフ再契約** まで揃ったあと、残る大きな穴である **本格FA市場（オフ・プール全体の補強フェーズ）** を、**いきなり全面GUI化せず**、**どこから入り第1弾で何をするか**を1本に絞る。
- **コード変更ではない**: 本書の作成・更新のみ。`conduct_free_agency` の改変や新ウィンドウ実装は **別タスク・別コミット**。
- **役割分担**: **インシーズンFA**は `GUI_INSEASON_FA_ENTRY_POLICY.md`。**再契約 y/n**は `GUI_FA_CONTRACT_ENTRY_POLICY.md`。**本書はオフの `conduct_free_agency` 前後に相当する「市場」体験**に限定する。

## 1. 現在の本格FA市場の位置づけ

### 1.1 オフ再契約 GUI との違い

| 論点 | オフ再契約（GUI 済） | 本格FA市場（本メモの対象） |
|------|----------------------|----------------------------|
| **対象** | **自ロスター**の契約満了間近選手 | **FAプール**の選手 |
| **処理箇所** | `Offseason._resign_players` → `evaluate_resign_offer` / `apply_resign_offer` | `conduct_free_agency` |
| **ユーザー操作** | 提示条件への **y/n**（`resign_ui_prompt`） | **なし**（CPU のみ） |
| **タイミング** | オフ前半（ドラフト・トレード・FAより前） | **ドラフト・オフトレードの後** |

### 1.2 インシーズンFA 1人獲得 GUI との違い

| 論点 | インシーズンFA（人事・GUI 済） | 本格FA市場 |
|------|-------------------------------|------------|
| **シーズン** | **レギュラー中** | **オフシーズン**（`Offseason.run` 内） |
| **データ** | `season.free_agents` 等（シーズン文脈） | `Offseason` が保持する **`self.free_agents`**（当該オフのプール） |
| **ロック** | `inseason_roster_moves_unlocked` 等（トレードと同系） | オフ実行中のみ意味を持つ（**未確認**: 専用ガードの有無は実装タスクで照合） |
| **他チーム** | CPU は `run_cpu_fa_market_cycle` が別経路 | **同一関数内**で全チームが順にオファー |

### 1.3 CLI / シミュ上の動き（事実）

| 項目 | 内容 |
|------|------|
| **呼び出し** | GUI 主画面からオフを実行しても CLI 直オフでも、**`Offseason.run` が最後まで進めば** `conduct_free_agency` は **同じ関数**で走る（**未確認**: 全分岐で必ず到達するかは動的網羅していない）。 |
| **ユーザー対話** | **ない**。進行は **print ログ**が主な可視化。 |
| **CPU中心か** | **はい**。チーム順（勝ち数などでソート後）、ロスター13人未満かつ上限内でループし、候補にオファー → 抽受。 |
| **ユーザーチーム** | **CPU と同じルールで自動補強されうる**（指名 UI はない）。 |

## 3. 最小実装候補の比較

| 候補 | 小さく切れるか | プレイヤー価値 | 既存ロジック流用しやすさ | 今の段階に合うか |
|------|----------------|----------------|---------------------------|------------------|
| **一覧を見るだけ** | **最も小さい**（表示のみ） | **低〜中**（情報は増えるが「操作できない」不満が出やすい） | **高**（プールの `normalize` 表示だけ） | **足場確認**には有効だが、製品ギャップの **主訴**だけ埋めにくい。 |
| **一覧から 1 人だけ契約** | **中**（`sign_free_agent` ＋ 既存プレチェック系の再利用が前提） | **高**（初めてオフFAを **意図的に1手**打てる） | **高**（インシーズンFA GUI と **同型**に寄せられる） | **推奨しやすい**（`conduct_free_agency` 本体をいじらず **前後に薄い層**を足すイメージ）。 |
| **再契約の直後に「1人だけ追加獲得」程度** | 文言・タイミングの問題。**技術的には**「再契約フェーズ直後」ではなく **現コードでは** ドラフト・トレード **後**がFAなので、**物語上の「直後」とコード順序は一致しない**（**未確認**: UXコピーで吸収可否）。 | 中（「オフの流れ」の中で1手） | **高**（やはり `sign_free_agent` 寄せ） | **B入口と組み合わせると自然**（オフ進行のステップとして説明）。 |
| **CPU市場を触らず、ユーザー分だけ薄く追加** | **中**（`conduct_free_agency` **無改変**で、**直前に**ユーザー1回だけ `sign_free_agent` を呼ぶ等） | 中〜高 | **最高**（市場ループはそのまま） | **本メモの前提（CPUを壊さない）に最も合う**。 |
| **（参考）`conduct_free_agency` 内にユーザー分岐を埋め込む** | **大** | 高 | **低**（関数が肥大・回帰範囲が広い） | **初手非推奨**（別設計・別フェーズ）。 |

**補足（重要・事実ベース）**: 現状、`conduct_free_agency` **後も**ユーザーチームは **CPU として再度**補強されうる。第1弾で **「手動1人だけ」** を足しても **そのままでは二重に補強されうる**。**初手では許容し、第2タスクで「手動消費」やユーザーチームの CPU スキップ等を検討**する、と明記しておく（**断定**: 初手スコープに **必ず**含める必要はない）。

## 4. 推奨する最小導線

**結論（1本）**: **入口は B（オフシーズン専用フロー内）**。**第1実装のスコープは「FAプール一覧を見て、ユーザーチームが `sign_free_agent` で 1 人だけ獲得できる」まで**（交渉・金額入力・CPUループの差し替えは **含めない**）。

| 項目 | 内容 |
|------|------|
| **入口** | **`conduct_free_agency` 呼び出しの直前**（またはオフ進行上それに相当する1ビート）に、**オプションの UI コールバック**を差し込めるようにするイメージ（**`resign_ui_prompt` / ドラフト注入と同型**。実装詳細は別タスク）。 |
| **第1弾でやる** | プールの **一覧表示**、**1人選択**、既存の **契約可否チェック**、`sign_free_agent` 実行、**キャンセル**。 |
| **第1弾でやらない** | `conduct_free_agency` の **アルゴリズム変更**、ユーザーとCPUの **競合シミュの新設**、オファー額の自由入力、複数人連続指名、独立常設ウィンドウのみの先出し、サラリー/cap/交渉式の変更。 |
| **既知の挙動（第1弾で直さなくてよい）** | 手動獲得の **直後に** `conduct_free_agency` が走ると、**ユーザーチームもCPU補強の対象**のまま。**プレイヤー説明**と **フォロータスク**で吸収する。 |

## 5. 最小 UI フロー案

1. **いつ開くか**: ユーザーが **オフシーズン実行**を進め、**オフトレード終了後〜本格FA（CPU）開始前**のタイミングで **モーダル／子ウィンドウ**が出る（**未確定**: `main_menu_view` から注入するか `main.py` からかは実装タスク）。
2. **候補をどう見るか**: 当時の **`free_agents`** を `normalize_free_agents` 相当で **リスト表示**（名前・位置・OVR・目安年俸程度で足りる）。
3. **誰を選ぶか**: **最大1人**（選ばず **スキップ**可）。
4. **何を確認するか**: インシーズンFAと同系の **キャップ・国籍枠・ロスター**メッセージを **短く表示**（既存ヘルパの流用を想定。**未確認**: オフ専用の追加ガードの有無はコード照合）。
5. **実行 / キャンセル**: **実行** → `sign_free_agent`。**キャンセル / スキップ** → そのまま **`conduct_free_agency`** へ。

**今回やらない（UI上も）**: 他チームのオファー一覧、市場のタイムライン再現、CPUとの同時争奪アニメーション、オフ外からの同一画面オープン。

## 6. 実装時の注意点

| 注意 | 内容 |
|------|------|
| **既存ロジック流用** | **`sign_free_agent` を正**とし、二重実装を増やさない。インシーズンFA GUI のウィザードを **共通化**できるなら最善（**未確定**: 切り出し粒度は実装タスク）。 |
| **オフ再契約との関係** | 再契約は **既に別フェーズで完了**している前提で説明する（「まだ契約してない選手」は再契約対象ではなく本件プール、の整理）。 |
| **インシーズンFAとの役割分担** | **レギュラー中**は人事の **インシーズンFA（1人）**。**オフ**は本導線。**文言で必ず区別**。 |
| **cap / nationality / roster** | 既存の `can_team_afford_free_agent` / `can_team_sign_player_by_japan_rule` / ロスター人数などを **踏襲**。 |
| **CPU市場との整合** | **`conduct_free_agency` は第1弾で変更しない**。手動とCPUの **重複補強**は §4 のとおり **フォローで検討**。 |
| **将来拡張** | コールバック I/F は **狭く**（「スキップ可能な1ビート」）。全面市場GUIや交渉UIは **別doc／別フェーズ**。 |
