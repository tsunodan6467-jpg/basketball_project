## 0. この文書の使い方

- **目的**: いま **仮調整**で支えている経済を、**どの思想・どの順で本モデルへ置き換えるか**を言語化し、あとから **1タスク1目的**で切り出せるようにする。
- **やらないこと**: 本書は**コード変更を伴わない**。数値の最終確定・給与テーブル全面改定・オーナー評価ロジックの再実装・GUI 改修の仕様固定はここでは扱わない（「論点」として触れるのは可）。
- **仮調整の位置づけ**: **否定しない**。長期周回と検証を可能にした**橋**として評価しつつ、**正本の経済モデルではない**と明記する。
- **ゲームの主役**: 経営単体ではなく、**編成・長期物語・試合**を支える**土台**（理想設計書と同旨）。

## 1. 現在の経営システムの整理

### 今ある経営要素（コード・仕様 doc ベース）

- **チーム財務スカラー**: `Team.money`、前季収支・`finance_history`、`record_financial_result` 経由の集計（`GM_MANAGEMENT_MENU_SPEC_V1.md` §0.3 と整合する設計意図）。
- **オフシーズン締め**: 収益・支出・ペイロール等の計算と、シーズン間の `money` 更新（`offseason.py`）。
- **シーズン中**: ラウンド進行時の処理、CPU 裏経営（`cpu_management.py`）、広報施策（`pr_campaign_management.py`）等。
- **施設・スポンサー・オーナーミッション**: `Team` フィールドと評価フロー（`evaluate_owner_missions` 等）。

### 仮調整として入っているもの（事実）

| 名称（便宜上） | 場所（代表） | コメント上の意図 |
|----------------|--------------|------------------|
| 初期資金 20 億円 | `Team.money` デフォルト、`main.TEMP_INITIAL_TEAM_MONEY` | 経営即死回避・本設計へ差し替え想定 |
| ラウンド仮営業収入 | `season.TEMP_ROUND_OPERATING_INCOME_BY_LEVEL`、`_apply_temporary_round_operating_income` | シーズン中の最低限キャッシュ注入。正本収支を壊さないよう **単純加算** |
| オフ締めの集中配分（仮） | `offseason.TEMP_OFFSEASON_CENTRAL_PAYROLL_SHARE`（0.98×ペイロールを収入加算） | modeled 収益が実ペイロール規模に追従していないことの**上乗せ**。本実装で置換予定 |

### 何が「遊べる状態」を支えているか

- **シーズン中の現金枯渇**を、ディビジョン別の **固定額ラウンド加算**で緩和している。
- **オフ締め**で、収益モデルと年俸規模のギャップを **ペイロール連動の一括収入**で吸収している。
- **高い初期資金**が、序盤の契約・トレード・投資の試行錯誤余地を与えている。

### 何がまだ本実装でないか

- 上記 **TEMP_* は「経済の因果」ではなく「バランス用ノブ」**に近い。プレイヤーが理解すべき**収益の物語**（観客・中継・スポンサー・リーグ分配の内訳）としては未整理。
- `GM_MANAGEMENT_MENU_SPEC_V1.md` が目指す **内訳・週次会計・正本との整合**は、**段階導入**が前提で、現状はすべてが満たされているわけではない。

## 2. 現在の主な問題

### 仮調整に依存している点

- 仮要素を外すと、**同じロスター・同じ試合結果**でも経済がすぐに破綻しうる構造になっている可能性が高い（検証は実装タスクで行う。**本メモでは断定せず「リスク」**と書く）。

### 収益と支出のスケールのズレ

- オフコメントどおり、**modeled 収益**と**実ペイロール**のオーダーが噛み合わないとき、0.98 係数が **糊**になっている。本実装では **どちらかのスケールを正**にし、もう一方を**説明可能な項**で揃える必要がある。

### オフ締めの一括負担の扱い

- 年俸・税金・施設維持等が **締め時にまとめて**効く一方、収入も **集中配分（仮）**でまとめて入ると、プレイヤーにとって **「なぜこの金額か」の追跡**が難しくなりうる（内訳表示と設計がセット）。

### オーナー評価とのつながり

- ミッション・信頼度は既に存在し、**軽微赤字の扱い**は調整済み（現状分析書）。ただし **財務の因果**（投資が翌年の収入にどう効くか等）がまだ薄いと、オーナー文脈が **結果の表示**に寄り、**意思決定の手応え**に繋がりにくい。

### プレイヤー体感としての違和感になりうるもの

- **難しい**より先に **窮屈** に感じるパターン: 説明なく現金が増える／締めで一気に動く、中身が見えない。理想は **理解すれば回る難しさ**（理想設計書）。

### まとめ

- 問題の核は「**数値が悪い**」より **「因果と表示が正本として追えるか」** にある、という前提で本実装を設計する。

## 3. 理想の経営ループ

### プレイヤーが理解していれば回るべきこと（概念）

- **キャッシュがどこから入り、どこへ出ていくか**の主な経路（リーグ・観客・スポンサー・分配・年俸・投資・税金／リーグ規定）。
- **シーズン中**と**オフ**で、会計の**主役となるイベント**が違うこと（試合週 vs 契約・締め）。
- **投資**（施設・広報等）が **中長期で何に効くか**（すぐ勝てる保証ではないが、無意味でもない）。

### 収益構造（理想・数値は未確定）

- **試合・人気・市場・ディビジョン**に依存する成分と、**リーグ共通の分配**に依存する成分の**両方**があり、プレイヤーは **編成・戦績・経営施策**でレバーを持つ。
- **内訳の和＝正本**（`GM_MANAGEMENT_MENU_SPEC_V1.md` §0.3 の精神）を満たし、表示とセーブが矛盾しない。

### 支出構造（理想）

- **年俸**が主役だが、それ以外（維持・税金・施策コスト）も **読める粒度**で存在する。
- **給与の支払タイミング**（月次／週次／締め一括等）は **1つに決めて文書化**し、実装と UI のラベルを揃える（**いま確定しない**。論点のみ）。

### 投資は何に効くべきか

- **短期のスタッツ直結だけ**にしない。**人気・収入係数・将来の予算上限**など、長期ループに接続する（既存 `arena_level` 等の延長線上でよい）。

### 経営判断と編成判断の関係

- **お金がないから星を捨てる**だけでなく、**星を取るためにお金を使う／使わない**が戦略になる。経営が主役になってロスターが従属しない程度の**重さ**（理想設計書）。

### 「夢の補強」と「経営の現実」

- 夢のスター編成は **ルールとキャップの壁**で抑えつつ、**経営で少しだけ余地**を作れる（無限甘さは避ける）。

## 4. 本実装で設計すべき主要論点

各項目: **今（事実）** / **理想（方向）** / **すぐには決めないこと** / **実装時の注意**

### 4.1 初期資金の扱い

| 観点 | 内容 |
|------|------|
| **今** | 新規・CLI 初期化で **20 億円規模**（`team.py` / `main.py`）。 |
| **理想** | 難易度・ディビジョン・ストーリーモードで変えうるが、**通常モードの正本**として「なぜこの額か」を説明できる。 |
| **すぐ決めない** | 具体的な円額。 |
| **注意** | セーブ互換・既存キャリアの違和感。変更は **移行または新規シーズンのみ**等を検討。 |

### 4.2 定期収入の扱い

| 観点 | 内容 |
|------|------|
| **今** | ラウンドごとの **固定額加算**（レベル別）。`record_financial_result` 外の **サイドエフェクト**。 |
| **理想** | **チケット・中継・日次/週次の営業**等に分解し、試合結果・人気と接続しうる。 |
| **すぐ決めない** | 式の係数・週次導入のタイミング。 |
| **注意** | 正本を `money` 直書きに増やし続けると二重計上リスク。`GM_MANAGEMENT` の **週次レジャー→締めで正本**パターンを参照。 |

### 4.3 チケット / スポンサー / 分配金 / グッズ等の位置づけ

| 観点 | 内容 |
|------|------|
| **今** | オフの収益計算・内訳 dict、`provisional_central_distribution` 等が存在。 |
| **理想** | 各カテゴリが **プレイヤーに説明可能**で、合計が正本と一致。 |
| **すぐ決めない** | カテゴリの細目と優先順位（全部 v1 必須か）。 |
| **注意** | 「表示だけ増やして合計が合わない」を防ぐ（§0.3）。 |

### 4.4 給与負担のタイミングと重さ

| 観点 | 内容 |
|------|------|
| **今** | オフ締めでペイロールが支出側に集約される経路が主。 |
| **理想** | プレイヤーが **シーズン中のリスク**（残金・来季）を感じられるが、**理不尽な即死**はしない。 |
| **すぐ決めない** | 支払を週次に振るかどうか。 |
| **注意** | FA・トレード・再契約との整合。いきなり全面変更しない（実装計画書と同旨）。 |

### 4.5 オフシーズン締めの考え方

| 観点 | 内容 |
|------|------|
| **今** | `TEMP_OFFSEASON_CENTRAL_PAYROLL_SHARE` で収入を **ペイロール連動**補正。 |
| **理想** | **リーグ分配・放映・罰金・収入分成**など、名前と因果が一致した項に分解。 |
| **すぐ決めない** | 0.98 の置換先の式。 |
| **注意** | 締めはセーブ境界になりやすい。**テストとログ**を先に用意する。 |

### 4.6 投資（施設など）の意味

| 観点 | 内容 |
|------|------|
| **今** | 施設 Lv・投資 API・CPU 裏経営が存在。 |
| **理想** | コストに見合う **中長期の返済**が説明できる（必ずしも数式単調増ではない）。 |
| **すぐ決めない** | 各 Lv の具体的 ROI。 |
| **注意** | CPU とユーザーの **同型データ**（`GM_MANAGEMENT`）を崩さない。 |

### 4.7 オーナー評価との接続

| 観点 | 内容 |
|------|------|
| **今** | ミッション・信頼度・赤字緩和定数あり。 |
| **理想** | **財務行動**（投資過多・連続赤字・予算遵守）が、オーナー文脈に**読み取り可能**に反映。 |
| **すぐ決めない** | 新しいペナルティ／報酬の数値。 |
| **注意** | **本メモでは再調整実装をしない**。接続の**設計**まで。既存緩和の巻き戻しは別タスクでリスク評価。 |

### 4.8 難易度設定（通常 / 将来のイージーモード）との関係

| 観点 | 内容 |
|------|------|
| **今** | イージーモード未実装（理想設計書で将来候補）。 |
| **理想** | **通常モード**を正本とし、イージーは **係数・初期資金・オーナー厳しさ**等のパッケージで後から差し込める余地。 |
| **すぐ決めない** | イージーの有無・初版範囲。 |
| **注意** | 通常の因果が固まっていない状態でイージーだけ作ると二重負債になる。 |

## 5. 本実装への段階的な置換案

**全面改定はしない**。下記は **自然な依存関係**に基づく案（順序はあとから入れ替え可）。

1. **会計正本と内訳の契約を固定する（doc＋フィールドの薄い追加）**  
   - `GM_MANAGEMENT_MENU_SPEC_V1.md` §0.3 に沿い、「何を正本と呼ぶか」「内訳をいつ永続化するか」を実装タスクに落とす。  
   - **TEMP をすぐ消さない**段階でも、**仮要素を内訳のどのラベルに載せるか**を決めると、後の置換が楽。

2. **シーズン中キャッシュフロー（ラウンド仮収入）の置換**  
   - 試合・人気・ホーム試合数等と接続した **説明可能な収入**へ。  
   - 移行時は **同じシーズン中の総流入が大きく暴れない**よう、シミュレーション比較を1タスクにする。

3. **オフ締めの再設計（0.98 集中配分の解体）**  
   - 分配・放映権等の**項**に分け、`provisional_central_distribution` の役割を正式名称へ。  
   - ペイロール規模との整合は **式**でなく **設計上の正**から決める。

4. **初期資金・新規キャリア導入の本設計**  
   - 上記が安定してから。セーブ互換の方針とセット。

5. **オーナー評価の「本接続」（財務イベントの反映）**  
   - 経済の因果が読めるようになってから。数値調整は別タスク。

6. **難易度モード（将来）**  
   - 通常モードが回ることを確認してから。

## 6. 今すぐ実装しないこと

| 対象 | 理由 | どの段階で |
|------|------|------------|
| 給与テーブル・契約ロジックの全面見直し | 波及が大きく、経済の**箱**が先 | オフ締め・キャップ設計の安定後 |
| オーナー信頼の**再調整実装**（本メモだけでは着手しない） | 経済因果とセットでないと迷走 | §4.7 の設計合意後 |
| イージーモード実装 | 通常正本が先 | §4.8 |
| GUI の大規模改修・ボタン配置の確定 | 本メモのスコープ外 | `GUI_MAIN_FLOW`・別 spec |
| 数値の一括チューニングだけのコミット | 構造が先 | 各置換タスクの検証時 |

## 7. 次に切り出せる実装タスク候補

※ **1件ずつ**選び、完了後に次を選ぶ想定。

### 候補 A: 経済内訳ラベルと TEMP の対応表（doc＋最小コード）

- **目的**: 仮収入・仮集中配分を、将来の正式カテゴリに**マッピング**し、表示・ログで名前を揃える。  
- **触る範囲**: `docs/`（本メモへの追記 or 小さな表）、必要なら **表示文字列・内訳キー**のみ。  
- **まだ触らない**: 定数の値、`record_financial_result` の契約変更。  
- **完了条件**: 「ラウンド仮収入」「provisional_central」が **どの正式項に吸収される予定か**が表で一意に決まっている（合意済みとしてチーム内で読める）。

### 候補 B: `GM_MANAGEMENT` §0.3 の「第1正本」検証タスク

- **目的**: 既存の `revenue` / `expense` / `money` 更新経路を洗い、**二重計上リスク**をリスト化する（調査のみ可）。  
- **触る範囲**: 読み取り中心。出力は doc または issue メモ1枚。  
- **まだ触らない**: 新フィールドの本導入、TEMP 削除。  
- **完了条件**: 「どの関数が `money` を直接いじるか」が一覧化され、週次導入時の**禁止事項**が書ける。

### 候補 C: シーズン中代替収入の第1案（仕様1ページ）

- **目的**: `TEMP_ROUND_OPERATING_INCOME` の**置換先**を、文章と粗い式の骨子だけ定義（数値は TBD）。  
- **触る範囲**: `docs/` のみ（本メモへの節追加でも可）。  
- **まだ触らない**: `season.py` の定数変更。  
- **完了条件**: プレイヤー向けに **1段落で説明できる収入の因果**がある。レビューで「窮屈すぎ／甘すぎ」と調整可能な**つまみ**が列挙されている。
