## 0. この文書の使い方

- **何のためか**: `GUI_MAIN_FLOW_AUDIT.md` で最優先ギャップとされた **「トレード／FA／契約の GUI」** のうち、**トレード最小版が人事から実装済み**となった次の一手として、**FA／契約を一気に広げず**、**どこを入口にし、どこまでを第1コミットにするか**を固定する。

- **役割分担**: **CLI メニュー構造の細目**は本書の主題外（必要なら別 doc）。**GUI の入口・最小スコープ**は本書。ロジック変更の正本は触らない。

## 1. 現在の FA / 契約の位置づけ

### 1.1 CLI で何ができるか（整理）

| 種別 | おおよその実装・経路 | 備考 |
|------|----------------------|------|
| **再契約（オフ・満了前）** | `Offseason._resign_players` → ユーザーチームは `_user_team_resign_decision`（**提示条件でオファーするか y/n**）→ `evaluate_resign_offer` / `apply_resign_offer` | **対話あり**（標準入力）。オフ処理の一部。 |
| **契約年の消化・離脱** | `_decrease_contracts` 等（オフ内） | ユーザー選択は別フェーズ。 |
| **オフ FA（市場からの獲得）** | `conduct_free_agency(teams, free_agents)` | **CPU が候補抽出〜オファー〜抽受**。ユーザーが「誰と契約するか」を選ぶフローは**この関数単体には含まれない**（ログ出力中心）。 |
| **インシーズン FA 補強（CPU）** | `Season` 進行から `run_cpu_fa_market_cycle`（`free_agent_market`） | ラウンド締切は `season_transaction_rules`。**CPU 向け**。 |
| **インシーズン FA（ユーザー）** | 人事 **「インシーズンFA（1人）」** → `sign_free_agent`（`main_menu_view`、2026-04-06） | **`run_gm_menu` に FA プール契約項目はなし**（CLI 正本なし）。**GUI で 1 名まで**（交渉・金額入力なし）。詳細は `GUI_INSEASON_FA_ENTRY_POLICY.md`。 |
| **新規契約** | ドラフト落札（`conduct_auction_draft` + **UI プロンプト注入可**）、オフ FA 上記 | 「FA 以外の新規」は主にドラフト側。 |
| **シーズン中の編成** | `run_trade_menu`、人事 GUI の延長・解除・**インシーズンFA（1人）** | トレードは CLI + 人事 1対1 GUI。 |

### 1.2 GUI で何が未実装か（事実ベース）

- **インシーズン FA 手動**は **人事で 1 名まで**（上表）。**市場全面・複数人・交渉 UI は未**。
- **本格 FA 市場の閲覧・交渉・獲得**の GUI は **未**（オフ `conduct_free_agency` もユーザー選択型ではない）。
- **オフ再契約の y/n** は **コンソール入力前提**のため、**GUI モードでオフを実行する場合、標準入力が無い環境では詰まりうる**（**未確認**: 実際の配布形態での再現は手元検証が必要）。
- 人事の **+1年延長**・**解除**は **GUI 済み**（別経路。オフ再契約の「提示額での y/n」とは異なる）。

### 1.3 契約操作の区別（用語）

| 用語 | 本メモでの指すもの |
|------|---------------------|
| **再契約** | 主に **オフ**の **満了間近選手へのオファー可否**（現状 `_user_team_resign_decision`）。 |
| **延長（＋1年）** | 人事 GUI の **同一条件での 1 年延長**（シーズン中の別操作）。 |
| **インシーズン FA** | レギュラー中の **FA プールからの補強**（期限は `inseason_roster_moves_unlocked` 系と人事案内でトレードと同列説明）。 |
| **オフ FA** | オフ処理後半の **市場ループ**（現状は CPU シミュ寄り）。 |
| **新規契約** | ドラフト獲得、FA サイン、特別指定など **チームに選手を新たに載せる**総称（初手では切り分ける）。 |

## 4. 推奨する最小導線

**結論（1 本）**: **オフシーズン実行中の「ユーザー再契約オファー（y/n）」を、標準入力ではなく GUI ダイアログで完結させる。**  
**プレイヤー向けの説明・発見性**は **人事のトレード・FA ブロック**で **「再契約の確認はオフ実行中に表示される」** と **1〜2 行で補足**する（**案 A + 案 C のハイブリッド**）。

| 項目 | 内容 |
|------|------|
| **技術上の入口** | **`Offseason`** にドラフト同様の **オプションコールバック**（例: `resign_ui_prompt`）を注入し、`_user_team_resign_decision` から **`input` 相当を GUI に差し替え**可能にする（**実装詳細は別タスク**）。 |
| **今回のスコープ（第1実装）** | **再契約 y/n のみ**。金額交渉のスライダー化、多段交渉、FA 市場の全面 GUI 化は **含めない**。 |
| **今回やらない** | `conduct_free_agency` の **ユーザー指名型への作り替え**、インシーズン FA の全面 GUI、キャップ／AI 交渉式の変更、サラリー表の再設計、左メニュー新設だけの先出し。 |

---

## 5. 最小 UI フロー案（ワイヤーなし）

1. **開き方**: ユーザーが **「オフシーズンを実行」**（既存フロー）を開始する。**人事**では「再契約確認はオフ中に出る」旨を読める（新規の常時ウィンドウは不要）。
2. **誰を選ぶか**: **システムが提示**（現状どおり `contract_years_left == 1` 等の対象）。ユーザーは **順次ダイアログ**で応答（初手はドラフト入札ほど複雑にしない）。
3. **何を確認するか**: 選手名・ポジション・年齢・OVR・提示年俸／年数・スコア・ソフトキャップ超過可否（**既存 `print` 相当をラベル or スクロールテキストに寄せる程度**でよい。第1コミットは **yes/no だけ**でも可）。
4. **最終確認 / 実行**: **Yes** → 既存 `apply_resign_offer` 経路。**No** → 現状と同様オファーなし。

**今回やらない**: FA 市場の一覧タブ、複数選手の同時比較、オフ外からの再契約画面。

## 6. 実装時の注意点

| 注意 | 内容 |
|------|------|
| **既存 CLI ロジックの流用** | `evaluate_resign_offer` / `apply_resign_offer` / キャップ判定は **変更しない**。差し替えるのは **入力取得部分**に限定するのが最小。 |
| **GUI と CLI の住み分け** | **CLI は残す**（`PRODUCT`・`IMPLEMENTATION_PLAN` と整合）。**同じ成否**になるよう **1 関数に集約**するのが望ましい（**未確定**: 関数名・配置は実装タスク）。 |
| **cap / roster / nationality / lock** | 再契約・FA は既存モジュールのガードを **呼び出し側で踏襲**。新たな緩和・締め付けは **別タスク**。 |
| **広がりすぎ防止** | 「人事に FA 市場ボタン」を先に付けて **中身が空／CLI 飛ばし**になる状態は避ける。第1弾は **オフ中プロンプト**に寄せる。 |
| **将来拡張** | `conduct_free_agency` をユーザー参加型にする場合も、**再契約 GUI 化で得た `Offseason` の UI 注入パターン**を流用しやすいように **プロンプト I/F を狭く保つ**。 |
