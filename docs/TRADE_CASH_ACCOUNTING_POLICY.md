## 0. この文書の使い方

- **目的**: トレード現金について、`record_financial_result` / `finance_history` に載せるか、トレード専用の履歴を正にするか、役割分担するかを**実装前に**揃える。
- **コード変更はしない**: 本書の更新だけでは `.py` を変更しない。実装は別タスク・別コミット。
- **役割分担**: 事実は監査メモ、正本の思想は `GM_MANAGEMENT_MENU_SPEC_V1.md` §0.3、**トレード現金の「どこを正と見せるか」**は本書。

## 1. 現在のトレード現金の扱い

**コード上の事実**（代表: `basketball_sim/systems/trade_logic.py` の `execute_multi_trade` 付近）。

| 項目 | 内容 |
|------|------|
| **`money` の更新** | `team_a.money -= cash`、`team_b.money += cash`（`cash_a_to_b` を整数として扱う）。`record_financial_result` は**呼ばれない**。 |
| **履歴が残る場所** | `Team.add_history_transaction` により `history_transactions` に追加。現金がある場合、`note` に `Cash+…円` / `Cash-…円 received…` のような**文字列**が載る（`player` は `None` の行もある）。 |
| **残らない場所** | `finance_history`（年次正本のペイロード）には**自動では載らない**。 |
| **CLI** | `main.py` のトレードメニューから `TradeSystem` 経由で上記が実行される。**未確認**: GUI 経路でも同じ `execute_multi_trade` に集約されているかは画面単位の照合が必要。 |

**プレイヤー視点（現状）**

- トレード実行時は、コンソール／将来の UI で「成立」のフィードバックは得られる想定。
- **年次の「前季収支」画面だけ**を見るプレイヤーには、トレード現金の増減は**説明されにくい**（`money` は変わるが `finance_history` に行が増えない）。
- 一方、**取引の文脈**では `history_transactions` に金額が**テキストで**残るため、「導線があれば」追跡可能性はある。**未確認**: すべてのクライアントがこのリストを同じ粒度で表示するか。

## 2. 考えられる方針案

### 比較表（概要）

| 観点 | **A. 年次正本へ寄せる** | **B. トレード専用ログを正にする** | **C. 両方（役割分担）** |
|------|------------------------|-----------------------------------|-------------------------|
| **概要** | 成立時またはオフ締めに、`record_financial_result` で収入・支出または専用内訳行として反映 | `money` 更新＋トレード履歴（構造化）を**説明の正**とし、年次レポートは触らないか最小限 | トレード側に**イベントの正**、財務側に**要約または導線**のみ |
| **メリット** | `money` と `finance_history` の乖離が減る（R2 系の違和感緩和）。JSON 出口で「年次1本」に寄せやすい | 取引条件として自然。大会賞金の「成果収入」と概念が混ざらない。実装が局所的になりやすい | レポート派・トレード派の両方に最低限応えられる |
| **デメリット** | トレードはシーズン中に複数回あり得る。**即時に record を積むと `finance_history` が細切れ**になりうる。収入／支出のラベル付けが便宜的になりがち | 年次画面だけでは「なぜ手持ちが動いたか」が**一覧しにくい**（導線設計が必要） | 二重表示の**説明の仕方**を誤ると「二回引かれた」誤解リスク |
| **プレイヤー分かりやすさ** | 「全部財務に出る」一貫性はあるが、**「このトレードでいくら払ったか」**は年次行から逆算しにくい場合あり | **「トレード画面／履歴で完結」**しやすい | 両方見せるなら**役割の言語化**（「取引」と「年次サマリ」）が必須 |
| **実装コスト** | 高め（`record` 契約・タイミング・二重計上防止・CPU トレード頻度） | 中（履歴の**構造化**・表示・永続化の確認） | 高（A と B の設計＋整合ルール） |
| **二重計上／二重説明** | `money` と `record` の両方で現金を動かすと**二重**（R1 と同型の禁止事項）。**どちらか一方だけ**が現金の真実 | 現金は `money` のみなら計上は一系統。**説明**が履歴と二重にならないよう文言設計 | サマリを `record` に足す場合、**トレード履歴と同じ額を二つの「物語」で言い切らない** |

## 3. このゲームに合う推奨案

**推奨: 案 B を「説明の正」の中心にし、案 C は将来オプションとして限定利用する。**

- **正（意味の正）**: トレード現金は**年次の「営業収益／費用」ではなく、クラブ間の取引条件の一部**としてプレイヤーが理解するのが自然。大会賞金（成果に対する収入）と**同じ棚に載せない**方が説明が破綻しにくい。
- **データの正（追跡可能性）**: 現状の `history_transactions` の `note` 文字列に頼り切らず、将来は **トレード ID・相手・金額・日付（またはラウンド）** が取れる**構造化されたトレード専用ログ**（既存フィールドの拡張または別リスト）を**第一の参照元**とする、という方向がよい（**未確定**: フィールド名・保存場所は実装タスクで決める）。
- **年次正本（`finance_history`）**: **いま必須で埋める案（純粋な A）は採用しない**。`GM_MANAGEMENT_MENU_SPEC_V1.md` §0.3 の「第1正本」は維持しつつ、**すべての `money` 変動を無理に `revenue`/`expense` の語彙に落とさない**。
- **案 C の限定利用**: プレイヤーから「前季の手持ちの変化の内訳が知りたい」という要望が強い場合に限り、**重複説明にならない形**で  
  - 財務詳細に **「トレード現金の純増減（サマリ1行）」** や  
  - **トレード履歴へのリンク／参照文**  
  を足す。**同じ取引を「収益」「費用」として二重に語らない**。

**「年次収支の物語」と「取引イベントの物語」**

- **主に見せるべきは取引イベント側**（トレード）。年次は引き続き**オフ締めのモデル化された収支**が主役でよい。
- トレード現金は、その年次物語の**脚注・差分要因**として必要になったら足す、がバランス良い。

## 4. 表示方針の理想

| 場所 | 理想 |
|------|------|
| **財務レポート（年次）** | 既定では**トレード現金の内訳行は必須にしない**。必要なら **純増減サマリ1行**または「詳細はトレード履歴」の**導線**のみ。 |
| **トレード履歴** | **いつ・誰と・選手の出入り・現金の向きと金額**が一覧できる。**説明の正**はここ（案 B）。 |
| **両方に出す場合（案 C）** | 財務側は**集計値のみ**。トレード側は**明細**。同じ金額を「収入」「支出」と「トレード」とで**矛盾するラベル**を付けない。 |
| **混乱を減らすコツ** | 「トレードで払った＝経費」などの**会計語の乱用を避ける**（プレイヤー向け文言）。内部実装で `expense` に入れる場合も、表示ラベルは「トレード支出」など**取引由来**と明示。 |

## 5. 実装時の注意点

- **`money` と表示のズレ**: 正本にサマリを足す場合、**`money` の変動と表示上の合計が一致**すること（監査・R2）。
- **二重計上**: `execute_multi_trade` で `money` を動かした**まま**、同額を再度 `record` で現金増減として扱うと二重。**一方の経路に寄せる**か、**サマリのみ別レジャーで相殺**する設計が必要。
- **二重説明**: 同じトレードを、トレード画面と財務画面で**別の「物語」として二重に語る**と、プレイヤーが「二回引かれた」と感じうる。役割を文書化する。
- **過去セーブ**: `history_transactions` の形式変更・新フィールド追加は、**旧セーブで欠損が出ても破綻しない**ようデフォルト・マイグレーション方針を別タスクで決める。
- **トレード UI / CLI**: 成立時に**金額の確認**が取れることが重要（設計の前提）。
- **JSON 出口**: トレード現金は **`trade_events` 等の列挙可能な配列**として出す価値が高い。年次 `finance_history` だけに寄せると**明細が落ちる**リスク。
