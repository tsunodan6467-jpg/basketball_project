# 国内バスケGM開発 主場概算収益の数え方方針メモ

**作成日**: 2026-04-06  
**文書の性質**: **小設計メモ（docs のみ）**。将来の第 2 キー **`inseason_matchday_estimate_round`** を実装する**前**に、**何を「主場試合数」として数えるか**と**代表戦ラウンド・試合なしラウンドの扱い**だけを固定する。**金額・係数・式は書かない**。

| 収益モデルの物語（主従 2 層） | `docs/SEASON_REVENUE_MODEL_NOTES.md` §3 |
| 経営置換の順序・論点 | `docs/ECONOMY_DESIGN_NOTES.md` |
| money 経路の事実 | `docs/ECONOMY_MONEY_FLOW_AUDIT.md` §2 |
| 代表イベントの理想像（クラブ操作と分離） | `docs/IDEAL_GAME_DESIGN_MASTER.md` §代表イベント |

**コード上の事実（静的確認・2026-04-06）**: `Season.__init__` で `self.season_events = self._build_season_events_from_regular_season()` → `self.events_by_round = self._group_events_by_round(self.season_events)`。`get_events_for_round(round_number)` は **`events_by_round` の参照のみ**。`SeasonEvent` は `home_team` / `away_team` / `competition_id` / `competition_type` / `round_number` 等を持つ。`_build_season_events_from_regular_season` は **`schedule_by_round` の対戦のみ**から生成し、**常に** `competition_id="regular_season"` / `competition_type="regular_season"`。**全日本カップ・東アジアトップリーグ等はこのリストに含まれない**（別関数で同一 `simulate_next_round` 内実行）。`ROUND_CONFIG` の `league_games_per_team == 0` のラウンドでは `_prepare_regular_season_schedule` が当該ラウンドにリーグ枠を積まないため、**通常は `get_events_for_round` は空リスト**。

## 0. この文書の使い方

- **何のためか**: `inseason_matchday_estimate_round` 実装時に「ホーム何試合分か」で**実装者間の解釈が割れない**ようにする。
- **コード変更ではない**: 本書は**ルール整理のみ**。`season.py` の改変は別タスク・別コミット。
- **役割分担**: キー名と第 1 / 第 2 の線引きは **`INSEASON_REVENUE_KEY_POLICY.md`**。シーズン中収益の**物語**は **`SEASON_REVENUE_MODEL_NOTES.md`**。本書は**「1 件＝何試合のホームに相当するか」**に限定する。

## 1. 現在わかっている事実

| 論点 | 内容 |
|------|------|
| **ラウンドとスケジュール** | レギュラー帯は `ROUND_CONFIG` の `league_games_per_team` 等に従い、`_prepare_regular_season_schedule` が **`List[List[Tuple[Team, Team]]]`**（ラウンドごとの全ディビジョン対戦の連結）を構築。 |
| **SeasonEvent の由来** | 上記スケジュール各行が **国内リーグ公式戦 1 試合**として `SeasonEvent` 化される（ホーム／アウェイはタプル順）。 |
| **ホーム試合数を数えられるか** | **同一データ源で可能**。特定チームについて `get_events_for_round(r)` を列挙し **`event.home_team is 当該チーム`** の件数を数える形が、いまの構造と**直交**する。 |
| **代表戦ラウンドの見え方** | `ROUND_CONFIG` に `national_team_window` が付いたラウンドは **`league_games_per_team` が 0**（例: ラウンド 7・20）。その週は **リーグ用 `SeasonEvent` は通常生成されない**。`simulate_next_round` では `_get_round_national_window` が真のとき **リーグ公式戦の `Match` を回さない**（`is_national_team_break and event.competition_type == "regular_season"` で `continue`）。代表側は `_run_national_team_window_log_only` 等の**別系**（**クラブの `SeasonEvent` ではない**）。 |
| **試合なし・別大会週** | 全日本カップ集中週・オールスター週等も **`league_games_per_team == 0`** のため、**当該ラウンドの `get_events_for_round` は空**になりやすい。カップ本体は **`_play_emperor_cup_round`**、東アジアは **`_play_easl_stage`** で処理され、**`season_events` / `events_by_round` には載っていない**。 |
| **第 1 キーとのタイミング** | `_apply_inseason_league_distribution_round` は **EASL・天皇杯の当ラウンド処理の後**、`simulate_next_round` の後半で実行（`season.py` 静的確認）。 |

**未確認・断定しないこと**

- 将来 `ROUND_CONFIG` を変え、**代表ウィークでも `league_games_per_team > 0` かつ `SeasonEvent` が残る**場合に、シミュを skip しつつ「日程上のホーム」だけ数えるべきかは、**そのときの仕様変更とセットで再判断**する（現状のデフォルト構成では起きにくい）。
- EASL・カップの「ホーム」を **別データ構造から数える**必要があるかは、**案 B を採る場合の別調査**（本書 §4 の推奨は案 A のため、深掘りしない）。

## 2. 数え方の候補

| 案 | メリット | デメリット | プレイヤー分かりやすさ | 実装の安全性 | いまの段階との相性 |
|----|----------|------------|------------------------|--------------|-------------------|
| **A. そのラウンドの国内リーグ公式戦ホーム数だけ数える** | `get_events_for_round` と **同一の正本**で説明できる。`SEASON_REVENUE_MODEL_NOTES.md` §3 の「ホームのレギュラー試合が何試合あるか」と**文言が一致**しやすい。二重計上リスクが第 1 キー（リーグ分配塊）と**切り分けやすい**。 | カップ・国際クラブ戦の主場興行は**反映されない**（見た目の収入と体感差が出うる）。 | **高い**（「リーグのホームの週だけ増える」が説明しやすい）。 | **高い**（既存 API・テストパターンに載せやすい）。 | **最も相性が良い**（コードが既にレギュラー日程と代表・杯を分離している）。 |
| **B. カップ戦・国際クラブ戦も含め、クラブのホーム開催なら数える** | 門前・協賛のイメージに**近い**（実際にホームでイベントがある週）。 | **今の `SeasonEvent` 一覧だけでは足りない**。EASL・天皇杯のホーム判定を**別経路で集約**する必要があり、第 1 キー・将来の杯賞金・表示と**役割の被り**を丁寧に避ける必要がある。 | **中〜低**（「なぜこの週だけ増えない／増える」の説明コストが上がる）。 | **低〜中**（結合点が増え、回帰の穴が開きやすい）。 | **早すぎる**（v1 の第 2 キーは単純さ優先）。 |
| **C. 当面は第 2 キーを入れない** | 実装リスクゼロ。監査・GUI は現状のまま。 | 主場の手応えは**分配キーだけ**のまま。 | （該当なし） | （該当なし） | **安全だが**、既に方針上「第 2 キー追加」が次候補として挙がっているため、**数え方だけ先に固定する本書の目的**とは別枠。 |

## 3. 代表戦ラウンドの扱い候補（コード実態との整理）

| 候補の考え方 | 実態との関係 |
|--------------|----------------|
| **代表ウィークでクラブのリーグ試合が無い → ホーム数 0** | 現状の `ROUND_CONFIG` では **リーグ枠が 0** のため `get_events_for_round` は**空**になりやすい。**推奨ルール（§4）と一致**。 |
| **代表ウィークでもクラブ試合があればその分だけ数える** | 現状はシミュ側で **代表ウィークのリーグ公式戦を skip** する分岐がある。**「数える」ならシミュと整合する定義**（例: skip 対象は数に含めない）が必要。**未確認**の変種設定では再検討。 |
| **代表イベント自体はクラブ matchday に含めない** | **IDEAL_GAME_DESIGN_MASTER.md** の趣旨（日本代表はプレイヤーが直接操作しないイベント）と整合。**クラブの `SeasonEvent` / リーグホーム数にも混ぜない**。 |

**結び**: 代表戦は **ナショナル側のログ・バナー**として扱い、**`inseason_matchday_estimate_round` の「件数」の入力には使わない**（クラブのリーグ主場数と**概念分離**）。

## 4. 推奨ルール（1 本）

| 項目 | 推奨 |
|------|------|
| **定義** | ラウンド `r`・チーム `T` について、**国内リーグ公式戦（`competition_type == "regular_season"`）**のうち、**`get_events_for_round(r)` に現れ、`home_team == T` である `SeasonEvent` の件数**を、そのラウンドの **「主場試合数」（第 2 キー用）**とする。 |
| **代表戦ラウンド** | **リーグ `SeasonEvent` が無ければ 0 件**。代表イベント・ログは **この件数に含めない**。 |
| **試合なしラウンド（カップのみ・オールスター等）** | リーグ枠が無いラウンドは **0 件**（カップのホームは **本ルールでは数えない**）。 |
| **なぜ安全か** | 第 1 キーは **リーグ分配・放映等の按分**（全チーム同額系）であり、第 2 キーは **クラブごとのホーム回数**——**物語が被らない**。データ源が **`get_events_for_round` に閉じる**ため、テストで **ラウンドごとに期待件数**を置きやすい。 |

**数値**: 本書では **係数・円換算は定めない**（別タスク）。

## 5. 実装時の注意点

| 注意 | 内容 |
|------|------|
| **二重計上** | 第 2 キーは **門前・協賛概算の塊**として第 1 キーと**併記**するが、**同じラウンド収入を二重に語る**説明・正本設計は避ける（`INSEASON_REVENUE_KEY_POLICY.md` §3・`SEASON_REVENUE_MODEL_NOTES.md` §3）。 |
| **第 1 キーとの役割** | 第 1 は **リーグ源の按分**、第 2 は **ホーム回数比例の概算**。杯・EASL を第 2 に入れると **第 1 の「等」や将来の杯経済**と**説明上競合**しやすい → **案 A を維持**する間は入れない。 |
| **カップ・国際の誤差** | プレイヤーには **「リーグ主場ベースの概算」**とラベルで示し、**全収入の再現ではない**ことを前提にする。 |
| **表示ラベル** | 内部キー `inseason_matchday_estimate_round` をそのまま出さず、**日本語の短いラベル**（例:「主場・門前概算」程度・未確定）を別対応表で付ける（既存の分配キー表示と同じ思想）。 |
| **テスト** | **同一 `Season`・同一 `round_number` で件数が決まる**形。`basketball_sim/tests/test_regular_season_home_count.py` で §4 と一致を固定。 |
| **シミュ skip との整合** | 極端な設定で「イベントはあるが代表週でリーグを skip」が起きる場合、**収入側は「実際にシミュした試合」に寄せるか「日程上のホーム」に寄せるか**を再確認する（§1 未確認）。 |

- **推測を断定にしない**。§1 の未確認は、必要になったらコードと `ROUND_CONFIG` を再読して潰す。
