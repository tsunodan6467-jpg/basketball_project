# soft cap 到達時の FA 新規オファー方針：決裁用短文メモ

**作成日**: 2026-04-06  
**文書の性質**: **意思決定メモ（コード変更なし）**。根拠: `docs/FA_SOFT_CAP_EARLY_RETURN_POLICY_NOTE_2026-04.md`、`docs/FA_CALCULATE_OFFER_CAUSE_PLAYCHECK_2026-04.md`、`docs/FA_CALCULATE_OFFER_CAUSE_CLASSIFICATION_PLAN_2026-04.md`。実装参照: `basketball_sim/systems/free_agency.py` の `_calculate_offer`、オフ手動FA `basketball_sim/systems/free_agent_market.py` の `offseason_manual_fa_offer_and_years`。

---

## 1. 文書の目的

**S1**（`payroll_before >= soft_cap` の即 `return 0`）は **buffer・`payroll_budget` 同期（S6 対策）では到達しない最上流**である。実装の前に、**soft cap 到達チームへの「FA 新規オファー（`_calculate_offer` の芯）」をどう扱うか**をプロダクトとして1行で固定し、次チャット・次コミットの判断材料にする。

---

## 2. 比較する3案

| 案 | 内容 |
|----|------|
| **案A: 完全不可** | **`payroll_before >= soft_cap` なら現仕様どおり即 `return 0`**。以降のベース年俸・ボーナス・budget・贅沢税クリップは実行されない。 |
| **案B: 低額例外可** | soft cap 到達でも、**最低年俸帯や所定の低額上限まで**は後段ロジックに流し、オファー芯を立てうるようにする（閾値・CPU の見え方は別途固定が要る）。 |
| **案C: 下流クリップ統一** | **S1 の early return を廃止**し、**`payroll_after > soft_cap`・budget・tax 等の既存クリップ列のみ**で抑える。単一パイプラインになる一方、到達後のオファー分布が現状より広がりうる。 |

---

## 3. 各案の長所・短所

| 観点 | 案A（完全不可） | 案B（低額例外） | 案C（下流統一） |
|------|-----------------|----------------|----------------|
| **ルールの分かりやすさ** | 最強。「ラインに張り付いたら芯は出ない」と説明しやすい。 | 中。**例外の定義**（額・年数・CPU 上限）を言語化しないと説明が崩れる。 | 弱め。**複数クリップの合成結果**が答えになり、プレイヤーが頭内シミュレーションしにくい。 |
| **手動と CPU の対称性** | 高い。両者とも主に同一 `_calculate_offer` を通るため、**入口で止めるなら一括で揃う**。 | **同一関数内で例外を定義すれば対称は維持可能**。閾値設計を誤ると片系だけ別経路になりやすい。 | **理論上は対称**だが、クリップ列の変更は **CPU FA 全体の分布**に波及し、手動だけ救済する余地が残りにくい。 |
| **実装の安全性** | 最高。**挙動変更ゼロ**で方針確定だけできる。 | 中。**条件分岐・境界テスト**が増え、低額連発・変な契約の回帰確認が要る。 | 低め。**入口を外すと税・budget・bridge との相互作用を再検証**しないとリスクが残る。 |
| **既存設計との整合** | 高い。現コードは **「soft 以上なら入口遮断」＋「未満なら下流でクリップ」**の二段構え。 | 中。**「遮断に穴を開ける」**ため、設計コメント・診断キー（S1）の意味を更新する必要がある。 | 低〜中。**S1 の役割を設計から外す**ことになり、`FA_SOFT_CAP_EARLY_RETURN_POLICY_NOTE` で整理した役割分担と衝突しうる。 |
| **プレイ感の自然さ** | 厳しい。**オフの「最後の補強」**と張り合うと、芯0＋手動の estimate フォールバック依存が説明負荷になる（既知）。 | 比較的良好。**「張り付きでも端の契約はありうる」**に寄せられる。 | 変化が大きい。**張り付き後も一定の芯が立ちやすく**なり、キャップゲームの緊張感が変わる可能性。 |
| **将来の調整しやすさ** | 高い。**「緩めるなら B か C」**の二択が明確。 | 中。**閾値ノブ**が増える。 | 低め。**クリップの組み合わせ調整**になり、原因追跡（診断）が複雑化しやすい。 |

---

## 4. 推奨案

**推奨: 案A（完全不可の維持）**を、**現プロジェクト段階の第一候補**とする。

- **向いている理由**: S1 は **buffer では触れない最上流**であり、ここをいきなり緩めると **意図しない FA オファー分布**のコストが大きい。**案A は実装リスクゼロで決裁でき**、手動・CPU の **経路対称性も維持**される。体験上の違和感は **`offseason_manual_fa_offer_and_years` の estimate フォールバック**や説明整備で **既に一部緩衝**がある（方針メモ参照）。
- **案B が今すぐ第一候補でない理由**: **例外の線引き・CPU の見え方・テスト範囲**を詰めないまま実装すると、**案A よりロールバックと説明が重い**。採るなら **閾値と受入基準の短設計を先に別コミット／別文書**で固定すべき。
- **案C が今すぐ第一候補でない理由**: **変更面が広く**、下流だけで意図したキャップ体験を再現できるかの **検証コストが最大**。ユーザー感覚どおり **「いきなり広く変えすぎる」**寄りの選択。

**補足**: プレイテストで **「張り付き後の端契約が必須」**と判明した段階で、**案B を第二候補**として閾値付きで再検討するのが自然。案C は **シミュレーションと受入基準が揃った後**の選択肢として残す。

---

## 5. 決裁用の1行結論

**soft cap 到達チームの FA 新規オファー（`_calculate_offer` のオファー芯）は、現仕様どおり完全不可とする。**
