# CPU 自動トレード: `_position_need_bonus` 仕様整理と最小調整案

対象: `basketball_sim/systems/trade.py` の `_position_need_bonus`（`conduct_trades` → `_evaluate_trade_value` 経由）。

観測: `tools/cpu_auto_trade_value_breakdown_observe.py`（seed 424242, 3 年）で、成立 gain の平均内訳が **position_need_bonus 最大**。

---

## 1. 現仕様要約

- **見ているもの**: 対象選手と**同じ `position` 文字列**の選手を `team.players` から列挙した人数 `same_pos_count`。
- **outgoing（`is_incoming=False`）**: 上記リストから**自分自身を除外**してから人数を数える（手放したあとのイメージ）。
- **incoming（`is_incoming=True`）**: まだロスターにいないため、**現ロスター上の同ポジ人数**をそのまま数える（加入後のイメージは粗いが「今の穴／余り」に対応）。
- **返却値（段階関数）**:
  - `same_pos_count <= 1` → **+5.0**（同ポジが極端に薄い）
  - `== 2` → **+2.5**
  - `== 3` → **0.0**
  - `>= 4` → **-2.0**（同ポジが厚い）

### incoming / outgoing の非対称

- 同一トレードでも、`gain_a` 計算では「`p_a` を team_a から出す」と「`p_b` を team_a に入れる」で **数え方の基準ロスターは同じ team_a** だが、**除外の有無**と「列挙に相手選手が含まれるか」が異なる。
- 片方が **薄いポジ（+5）**、もう一方が **厚め／普通（0〜+2.5）** の組み合わせだと、`OVR` 差が小さくても **差分だけで +2.5〜+5 級**が乗りやすい。

### OVR より position_need が勝ちやすい典型

- 交換後の「同ポジ人数の段」が **incoming と outgoing で 1 段以上ズレる**（例: 出す側は除外後に薄くなる／入る側は現状まだ薄い等）。
- `MIN_TRADE_OVR` 付近同士で OVR 差が小さいが、**ポジション構成だけで incoming 側に大きな +** が付くケース。

---

## 2. 問題の言語化（観測との接続）

- `_position_need_bonus` が **gain の平均寄与で最大**だったため、**実力差（OVR）よりロスター形状**が成立判定を押しやすい。
- 閾値（`min_gain` や `|gain_a - gain_b|`）は緩めのままだと、**「穴埋めロジック」だけで双方プラス**が取りやすい。
- 見た目として **低 OVR 同士・ロースター都合の入れ替え**が増えやすい（必ずそうとは限らないが、説明負荷が上がるリスク）。

---

## 3. 最小調整案（2〜3 案）

### 案1: 段階値の軽い圧縮（推奨の次点）

- 例: `+5 / +2.5 / 0 / -2` → `+4 / +2 / 0 / -1.5` のように **全段を少しだけ狭める**。
- **長所**: 閾値の「薄い／厚い」の意味はそのまま。**短所**: 定数が複数あり、後から微調整の粒度が粗い。

### 案2: 戻り値に単一スケール係数（第1候補）

- 内部で今と同じ段階を求め、最後に **`raw * SCALE`**（`SCALE` はモジュール定数、**既定 1.0** で現状維持、検証時のみ 0.85〜0.92 など）。
- **長所**: 変更箇所が一か所に寄せやすく、観測スクリプトで **1 ノブの感度試験**がしやすい。段階の意味も保持。**短所**: ペナルティ側も同率で弱る（厚み過多の忌避感はやや弱まる）。

### 案3: incoming 側だけスケール

- 例: `is_incoming` のときだけ `raw * IN_SCALE`（outgoing は現状維持）。
- **長所**: 「獲得で穴が埋まる喜び」だけ抑え、手放し評価はそのまま；非対称の意図が明確。**短所**: 係数バランスの設計負荷がやや高い；`gain` の解釈が「片側だけ弱体化」になり、調整根拠の説明が必要。

---

## 4. 第1候補（まず試すなら）

**案2: 単一スケール係数** — 実装は `basketball_sim/systems/trade.py` の **`POSITION_NEED_BONUS_SCALE`**（未設定時 **0.90**、多 seed 比較で選定）。import 前に環境変数 **`BASKETBALL_SIM_POSITION_NEED_BONUS_SCALE`** をセットすると上書き。段階ロジックの後にのみ乗算。

- 既存の段階ロジック・incoming/outgoing の違いを**壊さない**。
- 「需要は残すが振幅だけ少し下げる」が表現しやすい。
- 実装時は **定数 1 つ＋ return の形** に収めやすく、回帰テスト・観測差分の切り分けが容易。

---

## 5. 副作用の見込み（短く）

- **成立件数**: やや減る可能性（双方 `gain` が境界付近の取引が落ちやすい）。
- **ロスター穴埋め**: 極端な薄さへの反応は残るが、**最大 +5 が +4.4（×0.88）**程度になり、以前ほど `gain` を支配しにくい。
- **厚みペナルティ（-2）**: 同率で弱るため、**同一ポジ過多の忌避**はわずかに鈍る（必要なら案1と併用検討）。

---

## 6. 未解決 / 保留

- 最適な `SCALE` は **観測（多 seed・年数）でしか決めにくい**。
- 案3 のように incoming のみ弱める方が、見た目の「入れ替え感」には効くが、**調整ノブが増える**ため第 2 段階向き。
