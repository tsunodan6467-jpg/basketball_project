# 国内バスケGM開発 1対1トレード現金方針メモ

**作成日**: 2026-04-06  
**文書の性質**: **小設計メモ**。CLI で 1対1 に任意 `cash_a_to_b` を載せる**前**に、評価・UI 意味・multi との住み分けを整理する。実装仕様の正本ではない。

| 参照 | 文書 |
|------|------|
| トレード現金の会計・ログ方針 | `docs/TRADE_CASH_ACCOUNTING_POLICY.md` |
| money 経路の事実 | `docs/ECONOMY_MONEY_FLOW_AUDIT.md` |
| フェーズ・優先度 | `docs/IMPLEMENTATION_PLAN_MASTER.md` |

**コード上の事実（調査時点）**: `execute_one_for_one_trade` は `cash_a_to_b` を**キーワードで受け取れる**が、`evaluate_one_for_one_trade` / `should_ai_accept_trade` は**現金を見ていない**。CLI `propose_trade` は現金入力なし・実行も `cash_a_to_b` 未指定。multi は `evaluate_multi_trade_for_team` 内で `_calc_cash_rb_score` により現金・RB をスコアに反映。

## 0. この文書の使い方

- **目的**: 「1対1でも現金を許すか」「許すなら評価をどうするか」「multi とどう分けるか」を**短く決め、次の実装タスクに渡す**。
- **コード変更はしない**: 本書のみ。実装は別コミット。
- **役割分担**: 構造化ログ・正本の話は `TRADE_CASH_ACCOUNTING_POLICY.md`、本書は**導線・評価・AI 受諾**に限定。

## 1. 現在の 1対1 トレードの扱い

| 項目 | 内容 |
|------|------|
| **前提** | ユーザー `team_a` と AI `team_b` の**選手1対1交換**が中心。CLI は「相手選手」「自チーム選手」の2択入力。 |
| **現金入力** | **なし**（`main.propose_trade`）。 |
| **評価** | `evaluate_trade_for_team` ×2（ユーザー側・AI 側）。**選手価値・ポジション・国籍等のみ**。`cash` パラメータなし。 |
| **AI 受諾** | `should_ai_accept_trade` は上記 AI 側 `TradeEvaluation` の `accepts` を見るだけ。 |
| **実行** | `execute_one_for_one_trade(..., cash_a_to_b=0)` が既定。**現金ありでも評価が現金を知らない**ため、CLI から現金だけ載せると **AI は現金を織り込まずに承諾済み**になりうる（不整合リスク）。 |
| **multi との違い** | multi は `evaluate_multi_trade_for_team` が **`cash_change` / `rb_change` を `_calc_cash_rb_score` でスコア加算**し、閾値 `threshold` と一体で受諾判定。現金・選手・RB が**同じ評価関数**に入る。 |

## 2. 考えられる方針案

| 観点 | **A. 1対1では現金を許さない** | **B. 許すが評価は最小（現金をスコアに足す）** | **C. 許し、multi と同等の考え方へ寄せる** |
|------|------------------------------|-----------------------------------------------|------------------------------------------|
| **概要** | 現金は **multi 専用**。1対1 CLI は選手のみのまま。 | 1対1 に `cash_a_to_b` を足し、AI 側スコアに **`_calc_cash_rb_score` 相当のみ**加算（RB は 1対1 では引き続き扱わない等）。 | 1対1 を multi の薄いラッパに近づける、または共通ヘルパで評価・閾値を統一。 |
| **メリット** | 実装・説明が単純。**評価と実行のズレが起きない**。multi が「現金ありの本命」で一貫。 | プレイヤーが**軽い導線で現金付き1対1**をしたい場合に応えられる。数式は既存 `_calc_cash_rb_score` と揃えやすい。 | 経路間の**思想・閾値・拒否理由**が最も揃う。 |
| **デメリット** | 「1対1だけで現金を付けたい」人はメニュー切替が必要。 | 1対1 独自の閾値（0.5 固定系）と multi の `stance` 付き閾値が**まだ完全一致しない**可能性。 | **リファクタ・テスト範囲が大きい**。「最小」から外れやすい。 |
| **プレイヤー分かりやすさ** | 「お金は複数人トレード」と説明すれば明快。 | 評価画面に「現金分」を**数値で見せる**必要あり（ないと納得感が落ちる）。 | 揃えば説明は一貫しやすいが、UI 文言の整理が要る。 |
| **実装コスト** | **最低**（案内文・ヘルプの追加程度。コードは現状維持でも可）。 | **中**（評価関数に現金引数、CLI 入力、受諾・表示の整合）。 | **高**。 |
| **将来保守** | トレード種別が2系統と明確。 | multi と「似て非なる」閾値が残るなら**ドキュメントで差分を書く**必要。 | 長期的にはきれいだが、いまの段階では重い。 |

## 3. 推奨案

**推奨: 当面は案 A（1対1 CLI では現金を扱わない。現金は multi 導線に集約する）。**

- **理由（安全）**: multi は既に**現金・評価・資金不足チェック**が一連になっている。1対1 は `execute` だけ現金対応済みで**評価が追随していない**ため、ここに CLI から現金を入れると **AI 受諾と実際の取引内容がプレイヤー期待とズレる**リスクが高い。
- **理由（役割）**: 1対1は**最短の選手交換**、multi は**条件付きの大型取引**という住み分けは、現段階のコード構造とも整合的。
- **前進の仕方**: 案 A を採る場合でも、**「現金や RB を付けたいときはメニューで複数人トレードへ」**の一言（将来 CLI / GUI）で十分なことが多い。**入力前案内の改善方針を壊さない**範囲で足す。

**案 B は次の段階の候補**: プレイヤーから「1対1 だけで現金」の要望が強い場合、**`evaluate_one_for_one_trade` に `cash_a_to_b` を追加し、AI 側 `evaluate_trade_for_team` の後に `_calc_cash_rb_score(+cash, 0)` を足す**等の**最小パッチ**として切り出す（§6 タスク2）。その際は**ユーザー側表示にも同じ現金項**を出し、誤解を防ぐ。

## 4. UI / 操作上の注意

- **案 A 採用時**: 1対1メニュー内で multi を**差し込み誘導**する文を置くなら、**確定後の y/n の前**か、**メニュー見出し近辺**が分かりやすい（**未確定**: 実際の CLI 文言は実装タスクで調整）。
- **案 B 採用時**: 入力順は「**両選手の指定 → 現金（0 可）→ 評価表示 → 確認**」が自然。multi と**用語**（「あなたから相手へ支払う金額」＝ `cash_a_to_b`）を揃える。
- **混同防止**: 「1対1＝選手のみ」「複数人＝現金・RB 可」を**短いラベル**で固定するとよい（実装時）。

---

## 5. 実装時の注意点

- **評価**: 現金を入れるなら **必ず AI（と理想はユーザー）スコアに同じスケールで反映**。`trade_logic._calc_cash_rb_score` の再利用を優先検討。
- **資金不足**: `execute` 前に **評価段階でも `cash_insufficient` を返せる**と親切（multi と同型）。
- **AI 受諾**: `should_ai_accept_trade` のシグネチャに `cash_a_to_b` が入るなら、**拒否理由**に `cash_insufficient` / 価値不足を区別。
- **構造化ログ**: `execute_one_for_one_trade(..., cash_a_to_b=…)` は既に multi と同型ログを残す。**評価と同額の現金だけ**が実行に渡ることを担保（CLI バグでズレないよう）。
- **現金ゼロ**: 既定 0 のまま **既存挙動を維持**。
