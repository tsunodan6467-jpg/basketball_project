# 国内バスケGM開発 PR・グッズ money 可視化方針メモ

**作成日**: 2026-04-06  
**文書の性質**: **小設計メモ（docs のみ）**。広報（PR）施策とグッズ開発の **`money` 減算**について、プレイヤーに**どこを「説明の正」にするか**を固定する。会計仕様・`finance_history` 契約の正本ではない。

| シーズン中収益（第1・第2キー） | `docs/INSEASON_REVENUE_KEY_POLICY.md` |

**コード上の事実（静的確認・2026-04-06）**:  
- **PR**: `pr_campaign_management._commit_pr_campaign_core` が `team.money` を施策コスト分減算し、エントリを `team.management["pr_campaigns"]["history"]` に追加（`cost`・`label`・`round_key` 等）。`record_financial_result` / `finance_history` は**経由しない**。  
- **グッズ**: `merchandise_management._advance_merchandise_phase_core` がフェーズ進行時に `team.money` を開発費分減算し、`team.management["merchandise"]["history"]` に追加。オフ締めでは `management_merchandise_revenue_bonus` 経由で **収入内訳 merchandise にボーナス**が載りうる（**開発費の cash と売上の物語は別経路**）。  
- **GUI**: `main_menu_view.py` の経営まわりに広報パネル・グッズパネルがあり、`format_pr_history_lines` / `format_merchandise_history_lines` で履歴テキストを表示。**財務サマリー**（`format_inseason_cash_round_log_lines` 等）には **PR・グッズの支出は混在していない**。

---

## 0. この文書の使い方

- **何のためか**: PR・グッズによる `money` 変動を、**年次財務正本に寄せず**に説明するとき、**プレイヤーがどこを見れば納得するか**を先に揃える。
- **コード変更ではない**: 本書は**表示方針・導線の整理のみ**。実装は別タスク・別コミット。
- **役割分担**: 分類 **B**（別履歴で説明）の根拠は `ECONOMY_NON_LEDGER_MONEY_POLICY.md`。本書は**「その別履歴をどう見せるか・財務画面とどう並べるか」**に限定する。

## 1. 現在の PR・グッズ money の扱い

| 論点 | 内容 |
|------|------|
| **どこで `money` が動くか** | PR: 施策確定時に一括減算。グッズ: フェーズを 1 段進めるたびに減算（`ADVANCE_COST`）。CPU 経路も同コア関数。 |
| **`finance_history` / 正本** | **載らない**（監査メモどおり）。年次レポートの内訳スナップショットとは**別物語**。 |
| **履歴が残る場所** | **`Team.management` 配下** — `pr_campaigns.history` / `merchandise.history`。金額はエントリの `cost` と `money` 減算が**一致する設計**。 |
| **プレイヤーから見えるもの** | **経営メニュー内**の広報・グッズ各パネルで、実行結果メッセージ＋履歴テキスト。**所持金**は画面上部等で常時表示され、施策直後に変化は見える。 |
| **見えにくいもの** | **財務サブ画面だけ**を見るプレイヤーには、「いつ・何に・いくら使ったか」の**一覧の正**が経営パネル側にあることが**自明ではない**（NON_LEDGER §4 の「財務画面だけ見る人」リスクと同型）。 |

**未確認**: 全 GUI 経路で `management` が欠損セーブでも必ず補完されるかは、セーブロード経路の網羅テストに依存（断定しない）。

## 2. 考えられる見せ方の候補

| 案 | メリット | デメリット | 分かりやすさ | 実装コスト | 二重説明リスク |
|----|----------|------------|--------------|------------|----------------|
| **A. 財務サマリー／財務サブに出す** | 「お金の話は財務に集約」という直感に合う。残高と並べて**不足感**を説明しやすい。 | `inseason_cash_round_log`（ラウンド収益）と**並ぶとカテゴリが混ざる**。年次内訳と**二重に「支出」**を語りかねない。一覧が長くなりノイズ化。 | 財務派プレイヤーには**高**、施策派には項目が埋もれうる | 中（ウィジェット追加・ソート・空状態） | **高**（同じ減算を「財務ログ」と「施策履歴」の二箇所で詳述すると齟齬管理が必要） |
| **B. PR／グッズ専用の行動ログ・履歴を正にする** | **既存データ**（`management.*.history`）と一致。**「行動の結果」**として説明が自然。NON_LEDGER の **B 分類**と整合。 | 財務画面だけの人は**導線を知らないと見落とす**。 | 施策を打つプレイヤーには**高い**（いついくらかが行単位で残る） | **低〜中**（文言・導線強化が中心。一覧の再設計は任意） | **低**（正を1か所に置ける） |
| **C. 両方に出すが役割を分ける** | 財務側は**残高・注意・合算1行**、詳細は経営パネル — **迷子防止と深さ**の両立を狙える。 | 設計を曖昧にすると **A の二重説明**に戻る。文言の役割分担を doc で固定する必要がある。 | **高**（両タイプのプレイヤーに届けやすい） | 中（財務は最小差分に抑えれば可） | **中**（「サマリのみ／詳細は経営」と明記すれば抑止可能） |

## 3. このゲームに合う推奨案

**結論（1 本）**: **案 C を採るが、内訳の「正」は B（専用履歴）に置く。**  
財務側に載せるなら **第 2 層以下**に限る — 例: **「施策・開発支出の詳細は経営メニュー（広報・グッズ）の履歴を正とする」**という**一文導線**、または **当期の合算のみ**（任意・後続タスク）。

**なぜ自然か**

- 本作では PR・グッズは **「ラウンド収益」ではなく「GM の行動」**に近い。シーズン中収益（`inseason_*`）は **カレンダーラウンドに同期したキャッシュイン**であり、**同じ財務一覧に並べるとカテゴリが混線**しやすい（ユーザー指示の「混ぜない」と整合）。
- **トレード現金**は `TRADE_CASH_ACCOUNTING_POLICY.md` で**取引履歴を説明の正**に寄せる思想がある。PR・グッズも **`management` 履歴を正**にするのは同型の一貫性。
- **年次収支**はオフ締めの物語が中心。グッズは開発費（シーズン中 `money`）と、オフの **merchandise ボーナス**（別物語）が共存しており、**すべてを「年次収支の1行」に押し込めない**方が誤解が少ない（`merchandise_management` コメント・NON_LEDGER の注意と整合）。

**「年次収支」と「行動の結果」**

| 見せ方の軸 | 向いているもの |
|------------|----------------|
| **年次収支・正本スナップショット** | オフ締め後の revenue/expense 内訳（グッズ**売上ボーナス**はここに接続しうる） |
| **行動の結果（いつ誰が何をして幾ら）** | PR 実行・グッズ工程進行の **`management` 履歴**＋実行時ダイアログ |

## 4. 表示方針の理想

| 場所 | 理想 |
|------|------|
| **財務画面** | `inseason_cash_round_log` は**ラウンド収益専用**のまま。PR・グッズを**同じリストに混ぜない**。足すなら **(1) 静的な説明文**で経営パネルへ誘導、**(2) 任意**で「本シーズン（または直近）の施策・開発支出合計」など**1行サマリ**（データは `management` から集計・新規永続フィールドは不要でも可）。 |
| **行動ログ／履歴** | **現状の `format_*_history_lines` を強化する方向**が第一（日時・金額・ラベルの一貫性、空状態の文言、CPU 実行のタグは既存）。**正本はここ**。 |
| **両方やる場合の分担** | 財務＝**導線・合算・残高文脈**、経営＝**内訳・納得の正**。同じ履歴を財務に**丸ごと複製**しない（複製するなら自動生成で**単一ソース**に限定）。 |
| **混乱を減らすコツ** | 「財務レポート＝年次公式」「シーズン中一覧＝ラウンド収益」「広報・グッズパネル＝施策支出」の**三層**をプレイヤー向け一文で区別できるとよい。 |

## 5. 実装時の注意点

| 注意 | 内容 |
|------|------|
| **`money` と表示のズレ** | 履歴エントリの `cost` と実際の減算が**常に一致**することをテストで守る（既存コアの前提）。 |
| **二重説明** | 財務に長文化した一覧を足すと、`management` 履歴と**二重の正**になりうる。**単一ソース**（集計関数が `management` を読む）を推奨。 |
| **途中段階** | グッズはフェーズ単位で複数回支出がありうる。**1 施策＝1 ログ行（PR）**、**1 進行＝1 ログ行（グッズ）**の粒度を崩さない。 |
| **JSON 出口** | 将来クライアント移行時は `management.pr_campaigns` / `management.merchandise` を**そのまま系列化**しやすい。財務用に別名で二重保存しない方がよい（要検討時に本書を更新）。 |
| **GUI 最小差分** | 財務ウィンドウなら **既存 `ScrolledText` の下に短い `Label` または数行**で導線を足す程度が第 1 候補。レイアウト大改造は不要。 |

## 6. 次に切り出せる実装タスク候補（2 件以内）

### タスク 1: 財務サマリーへの「導線のみ」追加

| 項目 | 内容 |
|------|------|
| **目的** | 財務画面だけ見るプレイヤーが、**施策支出の正が経営パネルにある**ことに気づけるようにする。 |

| **完了条件** | 財務画面に **1 箇所**、経営の広報・グッズへの誘導が読める。 |

| 項目 | 内容 |
|------|------|
| **目的** | 財務文脈で**いくら使ったかの感触**を与えつつ、**詳細は履歴**に残す（§4 の 1 行サマリ）。 |
| **触る範囲** | 小さなヘルパ（例: `pr_campaign_management` / `merchandise_management` に集計関数）、財務表示 1 行、pytest。 |

## 7. 更新ルール

- **表示の正・導線**を変えたら本書を更新し、`ECONOMY_NON_LEDGER_MONEY_POLICY.md` の PR・グッズ行の説明と突き合わせる。
- **実装は別タスク・別コミット**。
- **推測を断定にしない**（GUI の全経路網羅は本書のスコープ外）。
