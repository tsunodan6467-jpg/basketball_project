## 0. この文書の使い方

- **目的**: 「1対1をメニューに戻すなら **何番・何ラベル・何分岐**が自然か」を決め、**次の実装 1 コミット**に渡す。
- **コード変更はしない**: 本書の作成・更新のみ。`run_trade_menu` の改修は別タスク。
- **役割分担**: 現金・評価の思想は `ONE_FOR_ONE_TRADE_CASH_POLICY.md`、会計は `TRADE_CASH_ACCOUNTING_POLICY.md`、本書は **CLI トレードメニューの構造**のみ。

## 2. 考えられる導線案

| 観点 | **A. トレードメニューに 1対1 を別項目で追加** | **B. 「トレード提案」内で 1対1 / multi に分岐** | **C. 当面 1対1 を実導線に戻さない** |
|------|-----------------------------------------------|------------------------------------------------|--------------------------------------|
| **概要** | 例: **2**＝1対1（選手のみ）、**3**＝multi、**4**＝戻る（**1** は一覧のまま）。ラベルで役割を並列表示。 | **2** 選択後にサブプロンプト（例: `1` 1対1 / `2` multi）で `propose_trade` と `propose_multi_trade` を振り分け。 | 現状維持。`propose_trade` は内部・検証用。 |
| **メリット** | **一目で両経路がある**ことと違いが分かる。誤って multi と思って 1対1 に入る／その逆を減らしやすい。番号が固定で説明しやすい。 | **トップレベルの行数**は 3 のままにしやすい。「トレードはまず 2」と覚えられる。 | 実装ゼロ。**リグレッションなし**。 |
| **デメリット** | メニュー項目が **1 つ増える**（4 択）。番号の**再割当**で習慣ユーザーに一度違和感。 | **毎回 multi する人も 1 キー増**。サブメニュー文言を誤ると **今の「2＝multi」習慣と衝突**。 | **案内文だけ入れた `propose_trade` が実プレイで無意味**に近い。CLI で 1対1 を試したい人が迷う。 |
| **プレイヤー分かりやすさ** | **高い**（ラベルで並列比較）。 | **中**（階層理解が要るが、トレードを「1ドア」にしたい人には馴染む）。 | **低**（機能が存在しないように見える）。 |
| **CLI 操作のしやすさ** | 目的別に **1 回で直行**。 | トレード回数が多いと **キーストローク増**。 | 変更なし。 |
| **方針との相性** | **良い**。項目名に「選手のみ」「複数人数＋現金＋RB」を載せれば、**現金は multi** が構造で固定化される。 | **良い**が、サブ画面の見出しと `propose_trade` 先頭案内の**重複**に注意。 | 方針ドキュメントと**体験が乖離**（説明はあるが導線がない）。 |

---

## 3. 推奨案

**推奨: 案 A（トレードメニューに 1対1 を別項目で追加し、multi は別番号にする）。**

- **自然さ**: いまの **「2. トレード提案」が multi 専用**であることが、ラベルと中身のずれを生んでいる。1対1 を戻すなら **並列で明示**するのが、`ONE_FOR_ONE_TRADE_CASH_POLICY.md` の「1対1＝選手のみ／multi＝現金・RB」と**同型の情報設計**になる。
- **住み分けが崩れない理由**: 分岐は **メニュー番号**で固定し、**multi にだけ**現金・RB の STEP を持たせる現状を変えない。1対1 は既存 `propose_trade` をそのまま呼ぶだけにできる（**評価・execute は触らない**）。
- **案 B を次点にする条件**: トップレベルを **3 行厳守**にしたい場合のみ。そのときはサブメニュー文言を **「1＝選手のみ（1対1）／2＝複数人数・現金・RB」**のように固定し、**ロック時**は 1・2 ともガードすることをセットで実装タスクに書く。
- **案 C**: **短期でメニューを触れない**ときの保留。ただし **`propose_trade` 案内の投資対効果**は低いままなので、方針メモとしては「暫定」と明記するのがよい。

## 4. UI / 文言上の注意

- **メニュー項目名**: multi 側は既存の **`トレード提案（複数人数＋現金＋RB）`** と揃う表現をトップメニューにも出すと、`propose_trade` 内の案内文と**用語が一貫**する（実装時に表記ゆれを避ける）。
- **1対1 のラベル例（未確定）**: 「**選手のみのトレード（1対1）**」など、**現金・RB がない**ことが名前から分かる方向。
- **`propose_trade` 先頭の案内**: メニューで既に区別できても **削らず残す**方が安全（テスト直叩き・将来 GUI 共有・コピペ導線）。**完全に同じ文をメニューでも繰り返さない**程度に抑え、**重複感**は実装タスクで 1 行に圧縮するか判断する。
- **入力前 UX**: `propose_trade` は既に **相手チーム選択後・[1/2] 前**に `flush` 付きで案内している。**メニュー追加後もその順序は維持**し、メニュー側は「どちらの経路に入ったか」だけ示すとよい。

## 5. 実装時の注意点

| 項目 | 内容 |
|------|------|
| **既存メニュー番号** | 案 A では **2 が multi から 1対1 に変わると既存ユーザーが誤操作**しうる。安全策は **新規番号に 1対1 を挿入し、multi を繰り下げ**るか、**一覧の直後に 1対1 を置く**など、**「以前 2 だったのが何か」**を README / リリースノートで一言触れる（**未確定**: 最終並びは実装タスクで決定）。 |
| **入力案内順** | `propose_trade` / `propose_multi_trade` 内部の STEP 順は**本タスクでは変更しない**。 |
| **案内の重複** | メニュー 1 行＋`propose_trade` 2 文は**許容**しつつ、長文化したら **メニューを短く・詳細は画面内**に寄せる。 |
| **誤操作防止** | multi を選んだ人が「現金がない」と混乱しないよう、**multi の見出しに現金・RB が入っている**現状を活かす。 |
| **実プレイで見えるか** | 実装後、**実際に `run_trade_menu` → 新番号 → `propose_trade`** で先頭案内まで到達できることを手動で 1 パス確認する（自動テストは別判断）。 |
| **`free_agents`** | `propose_multi_trade` は `free_agents` を取る。`propose_trade` は**シグネチャ上不要**。分岐実装時に **引数の渡し間違い**がないかだけ確認する。 |

## 7. 更新ルール

- **方針変更**（例: 案 A から B へ、または C で長期凍結）したら本書を更新し、`ONE_FOR_ONE_TRADE_CASH_POLICY.md` の参照表に本書へのリンクがあるなら**役割が矛盾しないか**を見る。
- **実装は別タスク・別コミット**。本書だけの更新では `.py` を変更しない前提。
- **推測を断定にしない**（例: 「プレイヤーが必ず multi を多く使う」等は書かない）。メニュー番号の最終並びは**実装コミットで確定**でよい。
