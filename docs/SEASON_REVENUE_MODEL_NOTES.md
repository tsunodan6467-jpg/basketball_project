## 2. 正式シーズン中収益として考えられる要素

「名前を付けられるか」「v1 で持てる複雑さか」で分類する。**数値・式は本節では確定しない**。

| 要素 | v1 で入れる候補 | 将来候補 | 今は見送り |
|------|-----------------|----------|------------|
| **チケット / 試合日収入** | 試合内容に**即さない**「**当ラウンドのホーム試合数などに比例した概算**」として**弱く**入れる余地あり | 得失点・対戦相手・PO 等まで反映した**精算** | v1 での**1 試合ごとの会計**、詳細チケット単価モデル |
| **スポンサー / 営業（協賛・物販・ローカル）** | **リーグ共通ベースに束ねた「営業・協賛のラウンド分」**として 1 行説明に含める | 施策・広報・グッズ実績と**連動した増減** | カテゴリ過分割（スポンサー A/B/C…） |
| **リーグ分配 / 放映相当** | **主役候補**。安定収入として「リーグからの分配・放映権相当のラウンド按分」と説明しやすい | 交渉イベント・特別契約 | 実在リーグの契約再現レベル |
| **人気・クラブ格・ディビジョン差** | **ディビジョン差**は既に `league_level` で実装済みの事実を**説明に載せる**（係数の見直しは別タスク） | **人気 `popularity`** の係数化、市場規模との組み合わせ | v1 で**多変数の同時最適化** |
| **施設投資との接続** | 原則 **見送り**（説明と式が一気に重くなる） | **アリーナ Lv** 等を「観客収入の上限・係数」に接続 | 施設ごとの個別収益モデルを v1 から全部載せること |

## 3. 推奨する収益モデルの骨格

**推奨（1 本）**: ラウンドごとに、プレイヤーへ **1 段落で説明できる**次の構造に寄せる。

1. **主役（ベース）** — **「リーグ分配・放映・中央営業のラウンド分」**  
   - リーグの格（`league_level`）で**規模が変わる**ことは、現状の D1/D2/D3 差と**説明上一致**させる。  
   - これは **「毎ラウンド入る安定キャッシュ」**の物語を与える。

2. **従（オプション・v1 は単純でよい）** — **「主場開催に応じた門前・協賛の概算」**  
   - **そのラウンドでホームのレギュラー試合が何試合あるか**（既存スケジュールから数えられるなら）だけで**比例**させる程度に留める案を第一候補とする。  
   - 細かい対戦カードまでは **v1 では必須にしない**。

**粒度**: **ラウンド単位**（現行フックと同じ）。試合シミュの直後に載せる現在位置は、「試合結果の次にキャッシュが動く」リズムと合わせやすい。

**内訳キー（docs 固定）**: 第 1 項に対応する正キーは **`inseason_league_distribution_round`**（`INSEASON_REVENUE_KEY_POLICY.md` §3）。**実装**: `Team.inseason_cash_round_log` に `amount`・`round_number` とともに記録（正本外）。

**オフ締めとの関係**: シーズン中は **「レギュラー帯のキャッシュイン」**として説明し、**年俸の大きな塊・集中配分**は引き続きオフの物語に寄せる（**0.98 の再設計は本書スコープ外**。二重に「同じ収入」を語らないよう、将来オフ側を触るときは **本メモと `ECONOMY_DESIGN_NOTES` を突き合わせる**）。

**プレイヤー向け一文（例・未確定）**:  
「このラウンドの収入は、**リーグからの分配と放映相当**、および**ホームでの試合があった分の門前・協賛の概算**です。」（ホーム項を v1 で省略する判断もありうる — その場合は分配のみの一文にする。）

## 4. このゲームに合う理由

- **`IDEAL_GAME_DESIGN_MASTER.md` §7**: 経営はスプレッドシートではなく**判断の手応え**へ。分配＋概算の 2 層なら、**「強いリーグ／ホームがある週は少し潤う」**が読み取れ、理不尽な無根拠加算より**見落としに気づきやすい**。
- **経営を主役にしすぎない**: v1 は**カテゴリを絞り**、試合・編成の主役を奪わない。シーズン中は**キャッシュの最低限の物語**で足りる。
- **夢・長期積み上げ**: ディビジョン昇格や（将来）人気・施設と係数で結ぶ余地を**骨格に空け**つつ、初手では**説明負荷を増やさない**。

## 5. 実装時の注意点

| 項目 | 内容 |
|------|------|
| **`money` 直接加算の乱立** | 置換後も**原則は 1 関数（または 1 ヘルパ経由）**に集約し、他経路の追加は `ECONOMY_MONEY_FLOW_AUDIT.md` を更新する習慣にする。 |
| **正本 / 非正本** | 当面は **`record_financial_result` に毎ラウンド載せない**選択肢も正当（`GM_MANAGEMENT_MENU_SPEC_V1.md` §0.3 の二段整合・ノイズとの兼合い）。**B（別履歴・ラウンドサマリ）**や**シーズン末の合流**は別タスクで決める。 |
| **表示との整合** | CLI 1 行を **正式ラベル**に差し替え、可能なら**内訳の短い内訳**（分配 / 主場分）を括弧で示すと不透明感が下がる。 |
| **オフとの二重説明** | 「シーズン中に既に取り込んだ項」をオフの `revenue` で**もう一度物語る**と混乱。**どちらを正に説明するか**を `ECONOMY_DESIGN_NOTES` とセットで見る。 |
| **テスト** | 同一 `round_number`・同一スケジュールで**再現可能な期待額**が書ける形（固定シード or モック）を優先。 |
