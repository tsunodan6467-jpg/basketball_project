# 情報メニュー仕様 第1稿（たたき台）

**関連**: `basketball_sim/systems/information_display.py`（Tk 非依存の行・テキスト生成）、`basketball_sim/systems/main_menu_view.py`（`open_information_window`・`_refresh_information_window`・`_get_news_items`）、`basketball_sim/tests/test_information_display.py`、`models/season.py`（`get_standings`・`leagues`・`awards_by_division`・`_calculate_and_print_awards_by_division`）、`systems/player_stats.py`（`PlayerStatsManager`）、`models/team.py` / `models/player.py`（シーズン累計フィールド）。

**位置づけ**: 確定仕様ではない。**読み取り専用**の表示拡張を前提とし、順位計算・試合シミュ・表彰の**正本ロジックは Season / 既存モジュール側**に残す。

## 0. 全体所見・保全上の前提

### 0.1 現状の情報 UI（実装ベース・2026-03-28 時点）

- **情報ウィンドウ**は **`ttk.Notebook` の6タブ**：**概要**（シーズン進行・次ラウンド予定・大会進行）、**順位表**、**個人成績**、**チーム成績**、**リーグニュース**、**表彰**。D1/D2/D3 切替は **複数タブで共有**する `StringVar`（コンボ同期）。
- **一覧**は **順位・チーム・個人**が **Treeview**。自チーム行は **タグ `user`** で背景強調＋「自」列（`build_standings_rows` / `build_team_summary_rows` の `is_user_row`）。
- **表示データの正本**は `information_display.py` 経由で **`Season.get_standings`**・**`PlayerStatsManager.get_leaderboard`**・**`awards_by_division`**・**`build_information_news_lines`**（`external_news_items` があれば主画面ニュースと同じリストをそのまま表示）に接続。GUI は結線と整形のみ。
- **主画面**のニュースパネル（`_get_news_items`）は従来どおり。**情報タブ「リーグニュース」**は上記 `build_information_news_lines` で **主画面と同内容を再表示**（`external_news_items` 注入時は同期）。**概要と「日程」ウィンドウの役割分担**は **§0.15** を参照。
- **CLI** の `print_midseason_leaderboards` 等は従来どおり。情報ウィンドウの個人タブは **同じ `PlayerStatsManager` を参照**（重複実装なし）。

### 0.2 全体所見（5分類案との関係）

ユーザー案の **順位表 / 個人成績 / チーム成績 / リーグニュース / 受賞・表彰** は、概念として**自然**で、既存データへの載せ方も段階的に可能。ただし **「すでにある画面」との役割分担**を仕様で決めないと、実装が二重化しやすい。

**実装**: 上記 Notebook 構成は情報ウィンドウに反映済み。主画面ニュースは **要約1〜4行**、情報の「リーグニュース」タブは **同じソースをタブ内で全文表示**（将来、履歴・カテゴリを足すなら情報側を厚くする想定）。

### 0.15 日程ウィンドウとの役割分担

| 画面 | 役割 |
|------|------|
| **情報 → 概要** | シーズン進行・**次ラウンド予定の要約**（`information_panel_schedule_lines` 等）・大会進行の**短い行一覧**。一覧の深掘りはしない。 |
| **左メニュー「日程」** | **次戦詳細**、**今後の日程（Treeview＋フィルタ）**、**過去結果**（`schedule_display`）。試合単位の一覧・詳細の正本。 |

ユーザー向けメッセージでも **「概要＝要約／日程＝一覧・過去結果」** と明示し、情報ウィンドウのフッターヒントから「日程」導線を維持する。

### 0.3 現状構造に対して危険そうな点

1. **順位表の「ゲーム差・連勝・直近・PO/昇降格帯」**  
   - `Season.get_standings` は **勝数 → 得失点差** のソートのみ。`Team` に **連勝・連敗・L10・GB** の永続フィールドは見当たらない。  
   - 仕様に書くなら **第1段は「勝・負・勝率・得点・失点・得失差」まで**（`regular_wins` / `regular_losses` / `regular_points_for` / `regular_points_against` から派生）。GB 等は **データ追加 or 別モジュール計算**が必要＝**後段扱い**が安全。

2. **個人成績の FG% / 3P% / FT% / 出場時間**  
   - `PlayerStatsManager.STAT_DEFINITIONS` は **得点・RB・Ast・Blk・Stl** のみ。`Player` のシーズン集計も同系が中心。  
   - シュート率・分は **ボックス集計が別系**なら未接続。**第1段は PlayerStatsManager 準拠の指標のみ**と明記しないと仕様と実装が衝突する。

3. **チーム成績の詳細指標**  
   - チームに **平均得失点** は試合数から**導出可能**（`games = wins+losses`）。チーム全体の **R/Ast/TO/命中率** は **リーグ集計の正本が無ければ**都度全選手合算などが必要になり、重い・欠損しやすい。**第1段は得失点系＋順位表との整合**に留めるのが安全。

4. **受賞・表彰レース**  
   - `awards_by_division` と本番 MVP ロジックは **`_calculate_and_print_awards_by_division`（シーズン終盤の流れ）**で埋まる想定。**シーズン途中はバケツが空または未更新**の可能性。  
   - 途中から「MVPレース」を UI に出すには、(a) **同じ関数を途中で呼ぶ**（副作用・パフォーマンス・プレーオフ枠前提の論理に注意）、(b) **別の「参考スコア」**を情報専用に定義、のどちらか。(b) は **本番表彰と二重定義**になるため **UI ラベルで「参考」**と必ず区別する。

5. **リーグニュース**  
   - `_get_news_items` は **デフォルトが固定文＋自チーム順位1行**。永続ニュースログは**現状ほぼ無し**。  
   - 「保存型」は **セーブスキーマ・整合・パッチ**が要る。**第1稿は「その場生成（読み取り）」または `external_news_items` 拡張**が低リスク。

### 0.4 より安全にするための前提（仕様に明記）

- **表示は正本データの投影**に徹し、**順位・スタッツ・表彰候補をユーザーが編集しない**。
- **集計ロジックの複製を避ける**：可能な限り `get_standings`・`PlayerStatsManager.get_leaderboard(s)`・（終了後）`season.awards_by_division` を呼ぶ。**GUI は整形のみ**。
- **欠損は正常系**：`season` / `leagues[level]` 空、選手0、`games_played==0` 等は **「—」「データなし」**。
- **シーズン終了前後**で表示内容が変わる（表彰・ニュース）ことを **UI 文言で許容**。

## 2. Cursor 視点：質問への回答

1. **構成は自然か**  
   自然。既存の **情報ウィンドウ＋PlayerStatsManager＋get_standings** に直結。**ニュースだけ主画面と役割分担**を決めると安全。

2. **先に作ると最も安全か**  
   **純関数レイヤー → S1 順位表 → S2 個人**。

3. **表示だけ先 / 後回し**  
   **GB・連勝・シュート率・チーム詳細スタッツ・ニュース保存・途中MVP本番同等**は後回し。

4. **初期から入れても壊れにくい項目**  
   勝敗、勝率、得失点、順位、PlayerStats の5種目、D切替、自チームハイライト。

5. **曖昧・危険・衝突しそうな箇所**  
   - 「表彰レース」＝**本番 MVP 式と同一**と誤解されること。  
   - 順位の**ゲーム差**を `get_standings` だけで約束すること。  
   - ニュースを**セーブ正本**にすること（スキーマ負債）。

6. **データ参照の切り方**  
   - 順位: **`Season.get_standings(season.leagues[level])`** のみ。  
   - 個人: **`PlayerStatsManager.get_leaderboard(teams, stat_key, ...)`**。  
   - 表彰: **`season.awards_by_division`**（存在・キーを `get`）。  
   - ニュース: **`build_*` が `season` / `team` を読むだけ**。

7. **責務分離**  
   - **information_display**: 行 dict 列表の生成。  
   - **main_menu_view**: UI。  
   - **Season / PlayerStatsManager**: 正本計算（変更しない）。  
   - **（将来）ニュース**: `systems/information_news.py` のような生成専用でも可。

8. **ニュース・表彰：保存型 vs その場生成**  
   - **ニュース**: 第1稿は **その場生成＋optional 外部リスト**が安全。保存はニーズ確定後。  
   - **表彰結果**: **本番は Season 終了処理が既に `awards_by_division` に書く**なら **読むだけ**が正。途中「レース」は**保存しない**方が一貫（毎回同じ入力なら再計算でよい）。

9. **より安全な画面分け**  
   - **概要 | 順位 | 選手 | チーム | ニュース | 表彰** の6タブ。  
   - または **概要に順位トップ3だけ**載せ **「全順位は順位タブ」** で重複を減らす。

## 5. 手動スモークチェックリスト（情報ウィンドウ）

リリース前や UI 変更直後に、実ゲームまたは開発用セーブで **5〜10分**確認する想定。**読み取り専用**で状態が壊れないことのみ見る。

1. **開閉**: 左メニューから情報を開く／閉じる／再オープンでクラッシュしない。
2. **概要**: ラウンド・FA 表示・次ラウンド行・大会行が以前と同様に埋まる（欠損は「—」等でよい）。
3. **順位表**: D1/D2/D3 切替で行が変わる。自チーム行が強調される（ユーザー運用時）。
4. **個人成績**: 区分・項目コンボでランキングが更新される。注記（出場試合数と表彰規定の別）が表示される。
5. **チーム成績**: 区分切替で平均得失点が整合する（0試合チームは除算ガードで破綻しない）。
6. **リーグニュース**: 主画面ニュースと矛盾しない（`external_news_items` 使用時は同一文言になること）。
7. **表彰**: シーズン途中は「未確定」系メッセージ、終了後は `awards_by_division` があれば表示される。
8. **区分の同期**: 順位表で D2 にしたあと表彰タブを開き、同じ D2 が選ばれている（共有 `StringVar`）。
9. **導線**: フッターの「日程」案内から日程ウィンドウが開き、情報の概要と役割が分かる。
