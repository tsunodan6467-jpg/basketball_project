## 1. 文書の目的

第1弾で **`generate_teams` 経路のみ**により、ディビジョン別（D1 / D2 / D3）の**クラブ総年俸帯の段差**は確保できた。次は、**その総額 `T` を変えずに**、同一ロスター内で **上位5人 / 中位5人 / 下位3人**の**厚薄（メリハリ）**を付け、正本 §5 の「階層感」に**近い見え方**を作るための、**安全な再配分方針**を実装担当がそのまま使える形で整理する。

本書は **FA・再契約・CPU 本格 FA・経済本体**には踏み込まない。開幕ロスターの **見た目と説明可能性**を第2弾のスコープとする。

---

## 2. 第1弾実装後に残っている課題

第1弾では、各チームについて **`calculate_initial_salary` による素年俸を重み**として、目標総年俸 `T` へ**比例スケール**する `_rebalance_team_initial_salaries_to_target` が導入された（`systems/generator.py`）。これにより **D1 / D2 / D3 の平均総年俸**は分離されうる（例: seed 42 で D1 平均おおよそ **10.8億**、D2 **7.3億**、D3 **5.3億** 前後）。

一方、**同一チーム内**の配分は、いまだ **「総額を `T` に合わせた単一の比例配分」**の色が強い。第1弾の意図どおり、**層別の追加ウェイトはまだ掛けていない**。

その結果、

- **上位5 / 中位 / 下位3**の**年俸のメリハリ**は、正本 §5 が示す「スター帯・ローテ帯・控え帯」の**コントラスト**より**平坦**になりやすい。
- プレイヤーがロスター画面で並べたとき、**スター感**（上位数名の突出）や**控え感**（下位枠の抑えめ）が、まだ**弱い可能性**がある。
- **総年俸帯（ディビジョン差）**は優先して確保できたが、**ロスター内の物語性**は第2弾の主題として残っている。

---

## 3. 第2弾の到達目標

- **総年俸帯は原則維持する**。すなわち、第1弾で定めた **チームごとの `T`（または同等のディビジョン別レンジ抽選）を変えない**ことを最優先ガードとする。再配分は **`T` 固定のまま**個人間で行う。
- **同一チーム内**で、**上位選手ほど厚く、下位ほど薄い**構造を、**数値と並び順の両方で読み取りやすく**する。
- **極端な一極集中**（1人が総額の大半）や、**階層の逆転**（明らかに低 OVR が高年俸など）は避け、**「自然」**の範囲に収める。
- 将来、**FA・再契約**が共通の base 年俸観へ接続するときに、**ロスター上の階層ラベルと整合しやすい**並び（OVR 順と年俸順の一致度が高い）を土台にする。

---

## 4. 現在の実装のどこに第2弾を乗せるか

**現状**: `generate_teams` は 13 人を `generate_single_player` で積んだあと、

1. `_initial_roster_target_team_payroll(league_level)` で **`T`** を決める  
2. `_rebalance_team_initial_salaries_to_target(team.players, T)` で **合計 `T`** に収める  

という **二段のうち後段**が第1弾の本体である。

**第2弾の乗せ方（推奨）**: **同じ呼び出し順・同じ `T` を維持**し、**再配分の中身**だけを「比例一発」から「**層別ウェイト付き**」へ拡張する。

| 案 | 内容 | 利点 | 欠点 |
|----|------|------|------|
| **A. `_rebalance_team_initial_salaries_to_target` 内部を拡張** | `T` と `players` を受け取り、層ラベル付け → 層別予算 → 層内配分までを一関数で完結 | 呼び出し箇所は `generate_teams` のまま。**局所化**されテストしやすい | 関数がやや長くなる（コメントと小さな内部ヘルパで抑える） |
| **B. 第1弾関数の直後に tiering 専用ヘルパを追加** | 第1弾は「比例」、第2弾は「`T` 維持のまま形状変更」と二段に見える | 責務がファイル上で並ぶ | **`T` を二重に保証**する必要があり、実装ミスで合計がズレるリスクが増える |
| **C. 第1弾処理をスキップして tiering のみ** | 省略 | — | **第1弾のディビジョン差が壊れる**ため不採用 |

**結論**: **案 A（既存 `_rebalance_team_initial_salaries_to_target` の拡張、またはその直前処理を吸収した「単一の再配分関数」への整理）**が最も安全である。`generate_teams` の行数増は最小で、**「`T` は第1弾と同じ入口から来る」**という前提を壊さない。

補足: ファイル内に **`_assign_salary_tiers_by_ovr`** のような **5行程度のラベル付け**だけを切り出すのは可。ただし **合計 `T` の責務は再配分関数に一本化**する。

## 5. 層分け基準の案

| 基準 | 概要 | 利点 | 注意 |
|------|------|------|------|
| **OVR 順** | 13 人を OVR 降順で並べ、1〜5 / 6〜10 / 11〜13 を層とする | **開幕時点でプレイヤーが最も見る指標**と一致。正本 §5 の典型 OVR 帯と**説明がつながりやすい** | OVR 同率が複数いると順序が不安定 → **安定ソート**（player_id 等のタイブレーク）が必要 |
| **素年俸順** | `calculate_initial_salary(ovr)` の値で並べる | 第1弾の重みと**完全に同じ情報**であり、実装の一貫性が高い | OVR と単調でないケースは実質ほぼ無いが、将来素年俸に別要素が入ったときにズレうる |
| **スターター想定ロール順** | `infer_role_expectation` 等で rank 付け | 「試合での役割」に寄せられる | **開幕直後のロスター生成のみ**では、並びが期待とずれる可能性。**依存が増える** |
| **複合** | OVR 主、同率は素年俸、など | 細かく制御可能 | 第2弾では**過剰**になりやすい |

**第2弾の推奨（1つ）**: **OVR 降順（安定タイブレーク付き）**を正とする。

**理由**: 正本 §5 は階層と **典型 OVR 帯**を対にしており、UI 上も OVR が最も参照されやすい。素年俸は現状 **OVR の線形関数**であり、第1弾の比例配分と **層境界の解釈がプレイヤーに説明しやすい**。スターター推定は第3弾以降の「役割プレミアム」に回してよい。

## 6. 年俸再配分モデル案

### 6.1 枠組み

- チーム総額を **1.0** とみなしたときの **層別シェア**は、`INITIAL_ROSTER_SALARY_REDESIGN_PLAN` §6.3 の目安を踏まえる。
  - **上位5**: 合計 **0.55〜0.65**
  - **中位5**: 合計 **0.28〜0.35**
  - **下位3**: 合計 **0.08〜0.12**

### 6.2 層内の配分

- **層内**は第1弾同様、**素年俸（または当該層に属する選手の素年俸）の相対比**で配分する。一様割り当ては、OVR と年俸の相関を弱めるため第2弾では避ける。
- 層ごとに **層予算 `B_top`, `B_mid`, `B_bot`** を決め、各層で **最大剰余法**等により整数化し、**全員の合計が `T`** になるよう調整する（第1弾と同型のパターン）。

### 6.3 複雑度の上限（第2弾）

- **クラブごとの性格差**（エース寄り／均タイプ）の複数プロファイルは **第2弾では入れない**か、**定数1組のみ**に留める。
- 国籍・ポジション補正は **入れない**（将来 §6 属性補正へ）。

### 6.4 int / 合計 / 下限

- 各 `salary` は **int**。**合計は必ず `T`**（第1弾と同一）。
- **下限**は既存の `MIN_SALARY_DEFAULT`（`contract_logic` と整合）を維持。層別後に下限で足りない場合は、第1弾と同様 **高額側からの再配分**で吸収する方針とする。

### 6.5 安全案と強め案（2案＋推奨）

| 案 | 層シェアの例（上位5 / 中位5 / 下位3） | 印象 |
|----|----------------------------------------|------|
| **安全案** | **0.58 / 0.32 / 0.10** | 第1弾の平坦さから**少しだけ**上位へ寄せる。逆転リスクが低い |
| **強め案** | **0.62 / 0.30 / 0.08** | メリハリはっきり。テストで「下位の最大が中位の最小を超えない」等を確認したい |

**推奨**: 実装はまず **安全案**で入れ、seed 固定の見た目とテストを見て **強め案**へ定数だけ上げる「微調整フェーズ」に回す（§8）。

## 7. 総年俸帯を壊さないためのガード方針

- **ディビジョン別段差**: `_initial_roster_target_team_payroll` の**レンジ定義を第2弾で変えない**（本書の前提）。変更するのは **同じ `T` の内訳**のみ。
- **チーム総額**: 再配分の**最終出力**で `sum(salary) == T` を**必ず満たす**（単体テストまたはアサートで担保）。
- **下限**: `MIN_SALARY_DEFAULT` 未満にしない。丸めで総額がズレる場合は、第1弾と同様の **差分を上位へ加算／減算**のパターンに収める。
- **極端な歪みの禁止**: 例として、下位層の最大が中位層の最小を上回る、上位1人が `T` の 40% を超える、などは **閾値またはソート後チェック**で検知し、定数を安全案側に戻す。
- **テストで確認すべきこと**（例）:
  - 複数 seed で **D1 / D2 / D3 の平均総年俸順序**が第1弾と**同等**（退行しない）。
  - 各チームで **OVR 順と年俸順の Kendall 的な一致**、または **上位5の最小 ≥ 下位3の最大** など**緩い単調性**。
  - `get_team_payroll(team) == T`（再配分直後）。

## 8. 安全な段階実装案

### 第2弾本体

- `_rebalance_team_initial_salaries_to_target`（または後継の単一再配分関数）に **OVR 順の層ラベル**と **層別予算**を導入する。
- **定数は安全案**（§6.5）から開始。

### 微調整フェーズ

- 層シェア定数のチューニング（安全案 ↔ 強め案）。
- OVR 同率時のタイブレーク規則の固定化（`player_id` 昇順など）。
- 必要なら **1チームあたりのログ（開発用）**で上位5と下位3の円額を確認。

### 将来フェーズ

- **FA / 再契約**の base と §5 の接続。
- **役割・国籍プレミアム**（正本 §6）。
- **架空プール・国際 FA** への同方針の波及検討。

## 9. 第2弾実装で「やること / やらないこと」

### やること

- **開幕ロスター**（`generate_teams` 経路）において、**総額 `T` を維持**したまま **上位5 / 中位5 / 下位3** に年俸を再配分する。
- **OVR 降順（安定ソート）**による層付けと、**層別シェア → 層内素年俸比例**の配分。
- **整数・合計一致・下限**のガードと、**既存のディビジョン別ペイロールテストの退行防止**。

### やらないこと

- `calculate_initial_salary` の**式の変更**。
- `GENERATOR_INITIAL_SALARY_BASE_PER_OVR` の**変更**。
- **ディビジョン別の目標総年俸レンジ**の再定義（第1弾の段差を壊す変更）。
- **ユーザーチームを 4 億に固定**する特例。
- **FA / 再契約 / CPU 本格 FA / 経済 / d995b48** の変更。
- **`generate_single_player` の能力生成**の変更。
- **`generate_fictional_player_pool` 等**への波及（別タスク）。

## 10. 次に実装で触るべき対象（1つだけ）

**`systems/generator.py` の `_rebalance_team_initial_salaries_to_target`（単一関数への集約を含む）**

### なぜ最初の1手として最も安全か

- 第1弾の **合計 `T` の唯一の確定箇所**がここに既にある。**第2弾は「内訳の作り方」だけを差し替え**ればよく、`generate_teams` のループ構造を増やさない。
- **新規ヘルパを「`T` 確定後の別ステップ」として乱立させない**方が、合計ズレ・二重スケールのリスクが低い。

### そこだけ触ると何が改善するか

- **同じ `T` のまま**、ロスター画面で **上位と下位の年俸差**が読みやすくなる。**スター感・控え感**が出る。

### 何はまだ残るか

- **FA 表示の estimate**や**再契約希望額**の正本接続。
- **役割・国籍**による帯内の上下振れ。
- **ユーザーチーム専用の 4 億寄せ**。
