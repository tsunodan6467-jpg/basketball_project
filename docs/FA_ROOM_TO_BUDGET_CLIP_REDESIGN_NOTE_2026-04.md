# FA：`room_to_budget` クリップ構造の見直しメモ

**作成日**: 2026-04-06  
**文書の性質**: **設計メモ（コード変更なし）**。背景: S6 buffer 段階試行（3M / 10M / 30M）と `tools/fa_offer_real_distribution_observer.py` の観測。決裁例: `docs/FA_S6_BUFFER_30M_DECISION_MEMO_2026-04.md`。実装参照: `basketball_sim/systems/free_agency.py` の `_calculate_offer`、同期: `basketball_sim/models/offseason.py` の `_sync_payroll_budget_with_roster_payroll`。

---

## 1. 文書の目的

**`payroll_budget` 同期 buffer を上げても**、主要観測系では **`final_offer == buffer` の全面化**が続き、**`0 < final_offer < buffer` が 0** であることが確認された。本メモは、**buffer 増額だけでは解けない**局面で、**`_calculate_offer` 内の `room_to_budget` クリップを今後どう見直すか**の方向を整理し、**次のコード実装前の合意形成**に使う。

---

## 2. 現在の `room_to_budget` クリップ構造

- **位置**: `free_agency._calculate_offer` は、**soft cap 調整の後**（hard cap / soft cap による `offer` 圧縮のあと）、**贅沢税圧縮の前**に **`payroll_budget` によるクリップ**を行う（`_calculate_offer_diagnostic` も同順で `offer_after_budget_clip` として記録）。  
- **式**: `payroll_budget = int(getattr(team, "payroll_budget", soft_cap) or soft_cap)`、`room_to_budget = max(0, payroll_budget - payroll_before)`、**`offer = min(offer, room_to_budget if room_to_budget > 0 else 0)`**。  
- **収斂**: ここまでの `offer`（年俸＋ボーナス等で組み立てた芯）が **`room_to_budget` を超えると、最終的にオファー額は room 天井に張り付く**。`room_to_budget` が **オフ同期 buffer によって「典型的に一定幅」**に押し上げられると、**観測行列全体が同じ天井値に揃う**ことが起きうる。  
- **段階試行との関係**: 3M / 10M / 30M では、**張り付きの数値が buffer に追従**し、**`0 < final < buffer` や `final > buffer` が出ない**条件では、**同じクリップ構造のスケーリング**に見える。  
- **性質**: 現状は **`payroll_budget` を経営上の上限として扱う設計**であり、観測結果は **単純なバグというより「天井の見え方」**に近い（ただしプレイ体感としては不自然になりうる）。

---

## 3. 取りうる見直し案

| 方向 | 概要 | メリット | リスク・コスト |
|------|------|----------|----------------|
| **現状維持** | S6 を「経営ガイドライン下での厳しい上限」として受け入れ、buffer は運用ノブのまま | 実装・決裁が単純。S1 との分離も維持 | CPU/ユーザー交渉の芯が長期間天井張り付きのまま |
| **クリップ式の緩和** | 例: `min(offer, room)` を **線形以外**（フロア付き残差、段階的 cap、滑らかな減衰）に変更 | 同一 `room` でも **芯の分布が開く**可能性 | 式の説明責任・回帰範囲が広がる。贅沢税段との順序も要検討 |
| **適用条件の変更** | 例: **一定額以下の room だけ**強く `min` を効かせる、**高額 FA 帯だけ**別ルール | 低 room の破綻回避と高額の柔軟性の両立を狙える | 閾値設計と「なぜここで別扱いか」の決裁が必要 |
| **別指標の導入** | `payroll_budget` とは別に **「交渉用 room」** を定義する | 経営表示と交渉芯を分離できる | **大工事**になりやすい。同期・GUI・CPU 一貫性のコスト大 |
| **切り出し＋テスト固定（準備のみ）** | クリップを **純関数化**し挙動をテストで固定、**中身は現状のまま** | **次段の式差し替えが安全**。いきなり本体全面書き換えを避けられる | 挙動変更は別コミット |

**比較の結論（次の最小差分）**: いきなり **式の意味を変えるコミット**より先に、**現行 `min(offer, room)` ブロックを局所化しテストで固定**する方が、プロジェクト段階として **安全かつ比較可能**。

---

## 4. 今の段階での推奨方向

- **いきなり `_calculate_offer` 本体の意味変更（新しい min 式の本番投入）は推奨しない**。観測上 **全面張り付きが構造として明確**になった直後であり、**経営・手動 FA・CPU 全体**への波及を一度に抱えるとロールバックが重い。  
- **先に行うべきこと**: **budget clip 部分の切り出し（純関数化）と、現挙動の回帰テスト**、または **診断経路で「候補クリップ関数」を並列評価できる小さな比較枠**（本番経路は未切替）のどちらか。ユーザー感覚の「**式を書き換える前に比較メモ／局所設計**」と整合する。  
- **最も安全な第一歩**: **挙動不変のリファクタ**（クリップの1箇所化＋テスト）→ **その後**に決裁付きで **候補式の1本**を試す。

---

## 5. 今回はまだやらないこと

- **`_calculate_offer` 本体の意味変更**（`min(offer, room_to_budget)` の式・順序の変更を含む）  
- **`_calculate_offer_diagnostic` のロジック変更**（現状は本体と同一維持が前提）  
- **floor 条件の変更**（`offseason_manual_fa_offer_and_years` 等）  
- **オフ手動 FA の全面再設計**  
- **低額／高額例外ルールの本体追加**  
- **buffer 定数のさらなる無決裁変更**  
- **generator / GUI / 経営収支のついで改修**

---

## 6. 次に実装で触るべき対象（1つだけ）

**`free_agency.py` において、`payroll_budget` による `offer` クリップ（`room_to_budget` の算出と `min`）を、現行挙動のまま **単一の純関数（またはプライベート関数）に切り出し**、**ユニットテストで入出力を固定するリファクタのみ**を行う（本番の数値結果は変えない）。**

- **なぜその1手が今もっとも妥当か**  
  **クリップ構造の「差し替え点」が1箇所に明示**され、**後続の候補式比較・段階投入**が **小さな差分**で済む。いきなり交渉式を変えずに、**変更境界を制御**できる。  

- **何はまだ残るか**  
  **実際の緩和式の選定**、**贅沢税クリップとの順序・二重圧縮**、**手動 FA / CPU 経路との整合**、**決裁メモ**、**観測スクリプトでの再検証**。
