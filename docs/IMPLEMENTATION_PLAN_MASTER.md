## 0. この文書の使い方

- **理想**は理想設計書、**現実**は現状分析書に書く。本書はその**橋**として、**何をどの順で進めるか**だけを扱う。
- 毎回の作業では、本章から **「今やる1件」**（第12章または第5章の先頭）を切り出し、**1タスク1目的**で実装する。
- 実装が終わったら、現状分析書・本書の該当箇所・必要なら `PRODUCT_ROADMAP_AND_VISION.md` を**事実に合わせて**更新する（手順は `AI_WORKFLOW_RULES.md`）。
- 本書に**願望だけの並び**や**順序のない要望一覧**を増やさない。項目は必ずフェーズまたは優先キューにぶら下げる。
- **更新ルール**: 実装順・直近タスク・実行順が変わったら本書を更新する。理想像の変更は `IDEAL_GAME_DESIGN_MASTER.md` を先に更新する。現状の事実は `CURRENT_STATE_ANALYSIS_MASTER.md` を正とする。推測で締めず、「未確認」は doc に明示する。

## 1. いまの計画思想

- **安定性最優先**: クラッシュ・セーブ破損・再現不能を最も恐れる（`.cursorrules`・`PRODUCT` Phase 0 と同旨）。
- **1タスク1目的**: 完了条件を観察可能にし、触る範囲を狭くする（`AI_WORKFLOW_RULES.md`）。
- **コア成立を先、見せ方強化を後**: Python 正本で「1年が最後まで進む」「主要 GM 判断がつながる」「長期で大崩れしない」土台を優先し、ハイライト・2D・音の本格は後段にまとめる。
- **設計の曖昧なまま大型実装しない**: 経営本実装・代表イベントの理想設計との接続などは、**先に設計メモまたは正本 doc の整理**を挟む。
- **Python を正本コアとする**: 表示用データの**正本化**や **JSON 出口の整理**は、将来のクライアント（例: Godot）を見据えた**中間整理**として計画に入れてよい。**Godot 全面移行は本計画の主戦場にしない**。
- **CLI は捨てない**: 開発・CI・スモーク・検証用として残し、**製品の主操作は GUI に寄せる**（`PRODUCT` の「CLI は製品プレイの前提ではない」と整合）。

## 3. 実装の基本戦略

### 先に固めるもの

- **認識の正本**: Steam 関連を含む、矛盾しうる docs の**事実の単一化**（パートナー画面の確認は人間作業が要る場合あり）。
- **長期進行の基準**: `--smoke`・pytest・手動シナリオの**再現可能な定義**（既存資産を正とする）。
- **セーブ・進行・Steam の境界**: 仕様変更時の移行方針（PRODUCT・`STEAMWORKS_DESIGN.md`）。

### 先に実装してよいもの（コアに属し、設計が既に doc 化されているもの）

- GUI 上の**欠けの埋め**（例: シーズン中トレード/FA のガードを CLI と同じにする）は、**仕様が PRODUCT に明記**されているため、設計ブロックが少ない段階で取り組みやすい（ただし1タスクに分割する）。

### 後回しにするもの

- **Godot 本格移行**、**給与テーブル全面見直し**、**クラウドセーブ・Rich Presence（v1 対象外）**、**BGM/MOD/ベータ運用の本格**（現状分析書・PRODUCT と整合）。
- **理想設計書の未確定論点**を、実装順に無理に固定しない。第9章「保留中の設計タスク」に逃がす。

## 6. タスク切り出しルール

- **1タスク1目的**。複数のバグ修正を1コミットに詰め込まない（`.cursorrules`）。
- **完了条件は観察可能**（表示・テスト・ログ・再現手順）。
- **触る範囲は狭い**。共通基盤を触るときは影響範囲を先に列挙する（`AI_WORKFLOW_RULES.md`）。
- **副作用**（セーブ・進行・Steam）を事前に1行で書く。
